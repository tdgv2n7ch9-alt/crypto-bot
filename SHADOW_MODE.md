# SHADOW_MODE.md — теневой контур 5 патчей (ночная сессия #2, Блок 1)

## Что это

Параллельный, не влияющий на боевые сигналы расчёт: на каждом реальном алерте
`signal_loop._send_alert()` (после того как боевой сигнал уже отправлен владельцу и
записан в `signal_journal`) `shadow_engine.py` **читает уже посчитанные данные** этого
же сигнала (`fa_engine.build_full_analysis()` результат: `candles_4h`,
`block11_trade_plan`) и применяет к ним 5 патчей из `patches/`, которые сами НЕ
вмержены в живую логику. Результат — отдельная запись в `journal/shadow_signals.json`.

**Ноль новых сетевых вызовов для самого расчёта** (переиспользует уже полученные
свечи) и **ноль влияния на боевой сигнал**: `shadow_engine.log_shadow_async()`
вызывается ПОСЛЕ отправки алерта и обёрнута в `try/except` и в `signal_loop.py`, и
внутри себя — падение теневого расчёта не может сломать или задержать боевой сигнал.

## Какие 5 патчей подключены и как именно

| Патч | Live-поведение | Что считает shadow |
|---|---|---|
| 01 killzone-hours | `bot.get_killzone_status()` — часы НЕ изменены | `bot.get_killzone_status_shadow()` — часы по `METHODOLOGY_CORE.md` §8 (Asia до 08:00, London 09-12, NY 14-16) |
| 02 rr-gate | `ta_extra.SR_MIN_RR_TP1 = 1.5` — НЕ изменена | `shadow_engine.SR_MIN_RR_TP1_SHADOW = 2.0` — сравнение с уже посчитанным `rr_tp1` |
| 03 breaker/MB | не вычисляется вообще | `ta_extra.classify_breaker_or_mitigation()` — новая аддитивная функция |
| 04 RSI-дивергенция | не вычисляется вообще | `ta_extra.detect_price_indicator_divergence()` — новая аддитивная функция |
| 05 BPR | не вычисляется вообще | `ta_extra.detect_bpr_zones()` + пересечение с зоной входа | 

Патчи 03/04/05 — это ГОТОВЫЕ ФУНКЦИИ, физически добавленные в `ta_extra.py` этой ночью
(в отличие от прошлой ночи, когда они существовали только как неприменённые `.diff` в
`patches/`). Они **не вызываются ни одним live-путём** — единственный вызывающий код —
`shadow_engine.py`. Патчи 01/02 остаются НЕ вмерженными в свои live-константы/функции —
вместо этого добавлены их shadow-версии (`get_killzone_status_shadow()`,
`SR_MIN_RR_TP1_SHADOW`), живые `get_killzone_status()`/`SR_MIN_RR_TP1` не тронуты.

## Формат записи в `journal/shadow_signals.json`

```json
{
  "ts": 1783707652.42,
  "symbol": "ETH",
  "direction": "short",
  "entry_lo": 1806.74, "entry_hi": 1842.52,
  "sl": 1878.29, "tp1": 1735.18, "tp2": 1681.52, "tp3": 1609.97,
  "rr_tp1_live": 2.4,
  "shadow_rr_gate_pass": true,
  "breaker_mitigation": "mitigation",
  "divergence": {"bearish_classical": false, "bearish_hidden": false,
                 "bullish_classical": false, "bullish_hidden": false},
  "bpr_zone_count": 2,
  "bpr_confluence": false,
  "patches_affected": ["03-breaker-mitigation"],
  "discrepancy": [],
  "live_journal_id": 42
}
```

- `patches_affected` — список патчей, которые дали не-нейтральный результат для этого
  конкретного сигнала (например, гейт 2.0 НЕ пройден, дивергенция ПРОТИВ направления,
  BPR-confluence найден). Пустой список — все 5 патчей на этом сигнале ничего не
  добавили/не изменили бы вывод.
- `discrepancy` — человекочитаемые строки для тех же случаев, для быстрого чтения без
  расшифровки полей.
- `live_journal_id` — id записи в боевом `signal_journal`, чтобы позже (когда сделка
  закроется) можно было сопоставить теневой прогноз с реальным исходом (`actual_r`).

## Как читать расхождения (сейчас, вручную)

1. Открыть `journal/shadow_signals.json`.
2. Отфильтровать записи с непустым `patches_affected` — это сигналы, где хотя бы один
   патч дал бы другой результат/предупреждение, чем боевая логика.
3. Для гейта 02 (`shadow_rr_gate_pass: false`) — эти сигналы боевая логика ПРОПУСТИЛА
   бы live-гейтом (1.5), но НЕ пропустил бы более строгий (2.0). Сопоставить с
   `live_journal_id` → `signal_journal` → закрылся ли сигнал в плюс/минус.
4. Для 04 (дивергенция против направления) и 05 (BPR-confluence) — пока чисто
   информационные метки, накопление истории для будущего измеримого сравнения
   win-rate "с меткой" vs "без метки" (нужна БОЛЬШАЯ выборка, сейчас исторических
   shadow-записей нет вообще — контур запущен только этой ночью).

## Что это НЕ делает

- Не отправляет ничего владельцу/подписчикам — только запись в JSON.
- Не меняет ни одной живой константы (`SR_MIN_RR_TP1`, часы `get_killzone_status()`).
- Не является A/B-тестом с автоматическими выводами — просто накопление данных для
  будущего решения владельца по каждому патчу (см. `patches/README.md`).

## Персистентность (важно для Railway)

Как и `signal_journal.json`/`journal/signals.json`, локальный файл
`journal/shadow_signals.json` живёт на ephemeral-диске Railway — обнуляется при
редеплое. Поэтому `shadow_engine.py` также пушит каждую новую запись в GitHub
(`journal/shadow_signals.json` в этом же приватном репо, тот же `GITHUB_TOKEN`, что и
signal_journal, отдельный путь/файл) через `run_in_executor` (не блокирует основной
event loop). Записи иммутабельны (создаются один раз, не обновляются), поэтому
конфликт при параллельной записи решается простым повтором (re-GET + append + PUT), без
сложного merge, который нужен `signal_journal.py` для изменяемых по ходу сделки
записей.

**Не проверено живым прогоном на Railway** (локально `GITHUB_TOKEN` не задан — GitHub-
часть кода структурно копирует уже работающий в проде паттерн `signal_journal.py`, но
собственный round-trip GET/PUT для `journal/shadow_signals.json` подтверждён только
на уровне "не падает при отсутствии токена", не на реальном пуше в репозиторий).
Это следует проверить после деплоя (первый реальный алерт создаст файл в GitHub).

## Что дальше (решение владельца)

Каждый патч остаётся независимым кандидатом на "да/нет" — накопление shadow-истории не
подключает и не должно подключать патчи к боевой логике само по себе. Решение "внедрить
патч N в бой" принимается владельцем отдельно, после того как накопится достаточно
shadow-записей для содержательного сравнения (сейчас — 0 записей, контур запущен этой
ночью).
