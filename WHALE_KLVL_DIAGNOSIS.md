# `whale_klvl_confluence` — диагноз (владелец, 2026-07-22, READ-ONLY)

Задача: подтвердить мёртв ли фильтр на ВСЕЙ истории (не только 22/7-сделочной
выборке) → убрать из чек-листа ИЛИ починить, если сломан расчёт, а не
реально всегда `False`. Диагноз перед действием, ничего не применено.

## Подтверждено: мёртв на ВСЕЙ истории, не только на 22 сделках

Полный локальный экспорт shadow-истории (`git archive origin/main`, активный
файл + все 802 архивных): **26439 записей всего, 10605 с полем
`whale_klvl_confluence` присутствующим (15834 — либо старее патча 06, либо
Фаза-B auxiliary-типы без этого поля вообще) → True: 0, False: 10605.**
`whale_klvl_matches` (список конкретных пересечений) непустой -- **0 раз за
всю историю**. Не только на узкой 22-сделочной выборке -- на всех 10605.

Даже для BTC/ETH/SOL (гарантированно всегда отслеживаемые Whale Radar, см.
ниже) -- 5 записей с полем в истории, **0 из 5 True**. Выборка малая, не
самостоятельное доказательство, но не противоречит общей картине.

## Корневая причина: НЕ баг расчёта -- структурный разрыв покрытия

Прочитан код `compute_whale_confluence()` (`shadow_engine.py:634-658`) --
логика пересечения K-LVL POI-зон с whale-зонами стакана написана корректно
(пересечение отрезков `wz.lo <= kz.hi and wz.hi >= kz.lo`, bid/ask по
сторонам без путаницы) -- это НЕ баг в математике сравнения.

Проблема -- ГДЕ БЕРУТСЯ `whale_zones`: `bot.get_whale_zones(symbol)` читает
`_whale_radar_state.get_zones(symbol)` -- состояние живого WebSocket-скана
Bybit-стакана (`whale_radar.py`). Символы для скана определяются ОДИН РАЗ
при старте процесса: `whale_radar.fetch_top_symbols(n=TOP_N_SYMBOLS=50)`
(топ-50 USDT-перпетуалов Bybit по 24ч обороту) + явно добавленные
`btcusdt`/`ethusdt`/`solusdt` (`bot.py:913-916`, докстринг честно отмечает:
"список топ-N в этом процессе больше НЕ пересобирается после старта").

**412 РАЗНЫХ символов** встречаются в shadow-истории с полем
`whale_klvl_confluence` присутствующим -- это символы, по которым реально
шли AUTO/signal_loop/full_analysis-сигналы. Whale Radar физически отслеживает
максимум **~50** из них (топ по обороту + 3 мажора). Для подавляющего
большинства (**оценочно 360+ из 412, ~88%**) `get_whale_zones()` структурно
ВСЕГДА возвращает `{"bid": [], "ask": []}` -- не потому что "нет
пересечения", а потому что **Whale Radar никогда не видел стакан этого
символа вообще**. `whale_klvl_confluence=False` для них -- гарантированный
артефакт отсутствия данных, не наблюдение о рынке.

Для оставшихся ~12% (реально отслеживаемые топ-50/мажоры, включая
BTC/ETH/SOL) -- выборка проверенных записей мала (n=5 для мажоров), не
позволяет отдельно заключить, реально ли пересечения редки по существу,
или есть ещё один слой проблемы (напр. слишком строгий критерий `klvl=True`
для POI-зон, который сам редко срабатывает) -- **не расследовано глубже**,
честно «н/д», не выдумываю вывод сверх того, что показывают данные.

## Побочная методологическая находка (честно, не скрываю)

Одна проверка внутри этого диагноза была сделана НЕПРАВИЛЬНО и переделана:
изначально пытался прочитать `bot._whale_radar_state.books` живьём через
`railway ssh python3 -c "..."` -- это спавнит ОТДЕЛЬНЫЙ процесс с СОБСТВЕННЫМ
свежим `_whale_radar_state = whale_radar.WhaleRadarState()` (пустым при
инициализации модуля), а НЕ подключается к памяти реально работающего бота
-- тот же класс методической ошибки, что уже был найден сегодня раньше с
`_job_heartbeats`/`_last_send_scheduled_write_ts`. Результат того вызова
("0 символов отслеживается") -- ложный, отброшен, не используется в выводе
выше. Реальный вывод построен ТОЛЬКО на полном локальном экспорте истории
shadow-записей (что реально произошло за всё время), не на попытке
заглянуть в живую память процесса.

## Вердикт: не "сломан расчёт" -- разрыв покрытия (scope mismatch)

Не то ни другое из предложенных владельцем вариантов буквально:

- **НЕ "починить сломанный расчёт"** -- расчёт корректен, ломать нечего.
- **"Убрать из чек-листа"** -- разумно, ЕСЛИ цель фильтра -- реально
  информировать про типичные сигнальные символы (мелко/среднекапы, которые
  и составляют основную массу AUTO/signal_loop сигналов) -- для них фильтр
  структурно НЕ МОЖЕТ сработать никогда, это мёртвый вес в чек-листе.
- **Альтернатива (не то же самое, что "починить баг")**: расширить покрытие
  Whale Radar за пределы топ-50 по обороту (напр. динамически включать
  символы, которые прямо сейчас сканирует AUTO/signal_loop) -- это НЕ багфикс,
  это отдельная фича-инвестиция (живой код, WS-подписки, стоимость по
  соединениям) -- требует отдельного "да" владельца, не тривиальный тог-фикс.

**Ничего не применено -- ни удаление из чек-листа, ни расширение покрытия.**
Решение (убрать сейчас / инвестировать в расширение покрытия / оставить как
есть до дальнейшего разбора) -- за владельцем.
