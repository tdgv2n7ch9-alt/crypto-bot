# AUTO-контур: контролируемый сбор данных — заготовка под решение владельца

Владелец, 2026-07-22 (фоновая очередь, приоритет 2): "средний путь" сбора
AUTO-данных, ЕСЛИ владелец решит частично снять паузу ради накопления
чистых outcome-данных AUTO-контура (`bot.real_full_analysis()`,
`TOP_LONG_AUTO`/`TOP_SHORT_AUTO`), НЕ рискуя подписчиками. **Это ЗАГОТОВКА
-- design-документ, НЕ код, НЕ включение. `PAUSE_LIVE_SIGNAL_EMISSION`
остаётся `true` до отдельного явного "да" владельца. Ничего не внедрено.**

## Зачем это вообще может понадобиться

Сегодняшний `MFE_MAE_VERDICT.md` показал: контрольная AUTO-выборка (n=8)
статистически хрупкая (±1 сделка переворачивает вывод A vs B). Если
владелец захочет НАКОПИТЬ больше AUTO-специфичных исходов (не открывая
полную рассылку подписчикам заново) -- нужен способ продолжать
ГЕНЕРИРОВАТЬ и ОТСЛЕЖИВАТЬ AUTO-кандидатов, при этом НЕ отправляя новые
сигналы в реальные подписческие чаты, пока `patch05_bpr`/`tz13` не
показывают PF>1.

## Механизм (описание, НЕ применено)

Новый флаг `AUTO_SILENT_COLLECTION_MODE` (env-var, default `false`, тот
же паттерн, что `PAUSE_LIVE_SIGNAL_EMISSION`/`MENU_V2_ENABLED` -- мгновенно
обратим без редеплоя). Меняет ДВЕ точки в `send_scheduled()`
(`bot.py:7205+`), обе уже точно определены построчно:

1. **Гейт паузы** (`bot.py`, самое начало функции): сейчас
   `if PAUSE_LIVE_SIGNAL_EMISSION: return`. Становится
   `if PAUSE_LIVE_SIGNAL_EMISSION and not AUTO_SILENT_COLLECTION_MODE:
   return` -- при включённом сборе данных обработка кандидатов
   продолжается, ДАЖЕ когда общая пауза активна.
2. **Точка реальной рассылки подписчикам** (`bot.py:7468`, цикл
   `for cid in chat_ids: ... await send_coin(bot, cid, sym, slug, a,
   text)`): при `AUTO_SILENT_COLLECTION_MODE=true` -- этот цикл
   ПРОПУСКАЕТСЯ целиком (или, для аудита владельцем, редиректится ТОЛЬКО
   на `owner_id` с явной пометкой в тексте карточки "[РЕЖИМ СБОРА
   ДАННЫХ -- НЕ живой сигнал]", то же решение, что уже принято для
   инбокс-маршрутизации строкой выше, `bot.py:7467-7478`).

**Всё, что ВЫШЕ точки 2, продолжает работать БЕЗ ИЗМЕНЕНИЙ** -- журнал
(`signal_journal.log_signal(...)`, нужен для честного отслеживания исхода
через `run_exit_tracker`), shadow-контур (`log_send_scheduled_shadow_async`
+ все Фаза-B инкременты) -- реальные, невыдуманные исходы продолжают
копиться, просто без публичной отправки. Реальный live-риск для
подписчиков = **ровно ноль** в этом режиме, независимо от любого другого
параметра (просто НИКТО не видит сигнал, кроме опционально владельца).

## "1% риск, лимит" -- честная переформулировка

Бот НЕ исполняет сделки на бирже сам (это сигнальный бот, не торговый
исполнитель) -- буквального "1% от депозита" здесь нет, чем управлять.
Переформулировано в ДВА реальных технических рычага ограничения:

1. **Тег данных** -- каждая запись, созданная в этом режиме, помечается
   явно (напр. `source="TOP_LONG_AUTO_COLLECTION"` вместо
   `"TOP_LONG_AUTO"`, либо отдельное поле `collection_mode: true` через
   уже существующий расширяемый вызов `signal_journal.log_signal()`) --
   чтобы БУДУЩИЙ анализ (WR/PF/MFE-MAE) мог честно ОТДЕЛИТЬ
   "накопленное в тихом режиме" от "реально показанное подписчикам",
   не смешивая две методологически разные истории в один вердикт (тот же
   принцип честности, что уже применялся сегодня для fa_engine vs AUTO).
2. **Операционный лимит** (аналог "лимита" из просьбы владельца, не
   денежный, а операционный) -- явный потолок на число ОДНОВРЕМЕННО
   отслеживаемых collection-режим сделок (напр. не больше 3-5 открытых
   разом) и/или дневной лимит новых кандидатов в этот режим (напр. не
   больше 2/день) -- ограничивает объём ручного review/мониторинга для
   владельца, не денежный риск (которого физически нет у самого бота).

## Откат

Тривиален: один флаг обратно в `false` -- поведение мгновенно возвращается
к текущему (полная пауза, если `PAUSE_LIVE_SIGNAL_EMISSION` всё ещё
`true`) без редеплоя (env-var на Railway).

## Оценка трудозатрат (не реализовано, оценка)

Малая-средняя: изменение ДВУХ условий (гейт + точка рассылки) в уже
существующей функции + опциональный тег источника/поля в уже
существующем вызове журнала. Переиспользует 100% остального кода
(scoring/shadow/journal/exit_tracker) без изменений. py_compile+pytest+
деплой+live-verify -- тот же процесс, что и любой другой фикс сегодня,
просто отдельный флаг вместо изменения существующего.

## Что это НЕ отменяет

- `PAUSE_LIVE_SIGNAL_EMISSION` остаётся семантически "публичная рассылка
  выключена" -- этот план НЕ снимает её, а предлагает ДОПОЛНИТЕЛЬНЫЙ,
  более узкий режим ПОВЕРХ неё.
- Промоушены `patch05_bpr`/`tz13` остаются заблокированы (PF<1) --
  никакого отношения к включению/выключению сбора данных, это отдельный
  гейт (PF>1), не гейт объёма выборки (тот уже пройден, `closed=23`).
- Включение `AUTO_SILENT_COLLECTION_MODE` -- **live-изменение эмиссии**,
  требует ТОЧНО ТАКОГО ЖЕ отдельного явного "да" владельца, что и снятие
  самой паузы -- этот документ НЕ является таким "да", это только
  заготовка на случай, если оно будет дано.

**Ничего не внедрено. Код не менялся. `PAUSE_LIVE_SIGNAL_EMISSION`
остаётся `true`. Промоушены заблокированы.**
