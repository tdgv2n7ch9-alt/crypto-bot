## Побочная находка (не в объёме этой задачи, не исправлено)

Формулировка алерта "2.1ч" — вводящая в заблуждение в СТОРОНУ ЗАНИЖЕНИЯ, не
завышения: `run_watchdog()` (`bot.py:1043-1045`) считает
`shadow_age = now - last_shadow_ts`, но `last_shadow_ts =
shadow_engine.get_last_send_scheduled_write_ts()` — **процесс-локальная
переменная в памяти, обнуляется на КАЖДОМ рестарте контейнера** (не читает
исторические данные из файла). Когда `last_shadow_ts is None` (как сейчас,
после сегодняшних рестартов), код честно откатывается на
`since_start` (аптайм ЭТОГО процесса), а не на реальное время с последней
записи. Поэтому "2.1ч" в алерте -- это фактически "2.1ч с момента старта
ТЕКУЩЕГО контейнера" (совпало с порогом 2ч по времени рестарта), а НЕ
реальные ~5.6+ часов молчания с 13:38:43 UTC. Кандидат на отдельную задачу:
персистентный (не in-memory) якорь для `last_send_scheduled_write_ts`, тот
же класс проблемы, что уже решался для других "не переживает рестарт"
метрик в проекте — не исправлено сейчас, вне объёма этого расследования.
