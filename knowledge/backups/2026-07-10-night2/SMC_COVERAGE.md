# SMC_COVERAGE.md — сверка методики (KNOWLEDGE_INDEX.md) с реальным кодом, по паттернам

Снято 2026-07-10. **Важное отличие от `KNOWLEDGE_GAPS.md`**: тот документ строился по файлам
методички и грепом только по `ta_extra.py`/`fa_engine.py`. При составлении ЭТОГО документа
обнаружилось, что в `bot.py` есть **ещё два самостоятельных SMC/ICT-движка**, не
рассмотренных в исходном проходе — `pro_analysis()` (строки 6804-7400, используется только
в `/long`) и повторяющийся блок "smc_*" эвристик внутри `full_analysis()`/`real_full_analysis()`
(строки ~1230-1260 и ~7650-7680). Из-за этого минимум 2 пункта `KNOWLEDGE_GAPS.md` были
**фактически неверны** («отсутствует полностью») — см. раздел «Исправления» ниже. Протокол
правды: не переписываю прошлый документ молча — оставляю его текст, добавляю датированную
поправку туда же и здесь.

Метод этого прохода: для каждого паттерна — `grep`/чтение реального кода (не по памяти),
классификация не «есть/нет», а **какой именно движок** его считает и **где реально
используется** (гейт сделки / только очки скора / только текст карточки / задекларировано,
но мертво).

## 0. Четыре параллельных "движка", это важно понимать перед таблицей

| Движок | Файл/функция | Где используется живьём | Природа детекции |
|---|---|---|---|
| **A. Структурный** | `ta_extra.py` + `fa_engine.py` (`build_full_analysis`) | `/full`, `/coin`, `signal_loop` ступень 2 (гейт `has_setup`) | Реальные фрактальные swing-точки, реальный BOS/CHoCH относительно bias, реальные зоны из touch-точек. |
| **B. ICT (pro_analysis)** | `bot.py:pro_analysis()` | ТОЛЬКО `/long` (`_scan_top_long_sync` → `signal_quality_filter`), очки к `score`, НЕ гейт | Реальные свечные паттерны (3-свечной FVG-гэп, engulfing-подобный OB, 2-swing BOS/CHoCH) — отдельная, более узкая реализация, не связанная с движком A. |
| **C. "smc_*"-эвристики** | `bot.py:full_analysis()` (используется `/precision`) и `bot.py:real_full_analysis()` (используется AUTO-scan, `/long`, `/short`, `/spot`, `x100`, `signal_loop` ступень отбора) | Rocket Score (очки) + текстовые ярлыки "BOS↑"/"OB накопл."/"FVG↑"/"Smart Accum" и т.п., показываемые НА КАРТОЧКЕ СИГНАЛА | **НЕ свечные паттерны** — чистые пороги % изменения цены/объёма (например `smc_fvg_bull = ch1h>=2 and ch24h>3`). Названия совпадают с SMC-терминами, механика — нет. |
| **D. Памп-радар** | `pump_detector.py` | Памп/дамп детект | Z-score объёма + движение цены, TP/SL от `entry ± minRR*risk` без проверки реального BOS после разворота (см. баг №5 расследование). |

**Реальные entry/SL/TP для всех торговых сигналов** (кроме памп-радара) строятся движком A
(`ta_extra.find_sr_zones`/`build_trade_from_structure`) независимо от того, что показывают
движки B/C — движки B/C влияют только на Rocket Score/факторы, не на сами уровни сделки.

## 2. Исправление найденных ранее ошибочных выводов (Протокол правды, п.5)

При составлении этой таблицы обнаружено, что 2 пункта `KNOWLEDGE_GAPS.md` содержат
**фактически неверные** формулировки — не додумка, а проверяемая ошибка предыдущего
прохода (грепом были покрыты только `ta_extra.py`/`fa_engine.py`, `bot.py` не проверялся
на предмет `pro_analysis()`):

- **`KNOWLEDGE_GAPS.md` п.9 и «Предложения из библиотеки» п.1** — "FVG/Imbalance —
  заявлено, не реализовано" — **неверно**. FVG реализован дважды: `ta_extra.find_fvg_zones`
  (используется в `fa_engine`, но только для отображения POI на карточке, не как фактор
  чек-листа) и `pro_analysis.ict_fvg_bull/bear` (используется в `/long` score). Точный,
  исправленный статус — см. таблицу выше: не "отсутствует", а "есть, но не используется как
  гейт/фактор ни в одном из путей, где методичка это рекомендует" — статус 🟡, не 🔴.
- **`KNOWLEDGE_GAPS.md` п.22 и «Предложения из библиотеки» п.4** — "Premium/Discount —
  ПОДТВЕРЖДЕНО ОТСУТСТВУЕТ" (со ссылкой на grep, который в тот момент реально ничего не
  нашёл в двух проверенных файлах) — **неверно как общее утверждение**. `pro_analysis.
  ict_pd_array` считает Premium/Discount/Equilibrium. Уточнённый статус: реализовано, но
  изолированно в `/long`-only движке B, не используется как гейт ни там, ни где-либо ещё,
  OTE-зона (61.8-79%) действительно нигде не реализована — **эта часть исходного вывода
  верна**, только не как "отсутствует полностью".

**Что это меняет для «Предложений из библиотеки»**: пункты 1 и 4 в `KNOWLEDGE_GAPS.md`
нужно читать не как "добавить FVG/Premium-Discount с нуля", а как "решить, стоит ли
поднимать уже существующую, но изолированную детекцию (движок B, только `/long`) до
статуса гейта/фактора в основном движке A, используемом во всех путях" — другая по
масштабу задача (меньше нового кода, больше вопрос архитектурного решения). Оценка
эффекта/измерение — та же логика (аддитивный `levels_source`/`degraded_data`-подобный флаг
в journal, сравнение win-rate до/после), актуальность предложений не отменяется, только
формулировка задачи. Решение — по-прежнему за владельцем, ничего не внедрено.

Сами пункты `KNOWLEDGE_GAPS.md` НЕ переписаны молча — оставлены как есть с датированной
пометкой ниже (в файле) со ссылкой на этот раздел.

## 3. Новая, ранее не описанная находка: вводящие в заблуждение ярлыки на карточке (движок C)

Не gap методики, а находка про честность самой карточки сигнала: факторы "BOS↑", "OB
накопл.", "FVG↑", "Smart Accum", "Liq Sweep" и т.п., которые видит владелец/подписчик на
карточках AUTO-скана/`/long`/`/short`/`/spot`/`x100`/`/precision` (движок C,
`real_full_analysis`/`full_analysis`), **вычисляются НЕ по свечным паттернам**, а по
порогам % изменения цены за 1ч/24ч/7д/30д (например `smc_fvg_bull = ch1h>=2 and ch24h>3`)
— несмотря на SMC-терминологию в названии переменной и на карточке. Реальная,
структурная детекция (движок A: `find_sr_zones`/`build_trade_from_structure`) при этом
ДЕЙСТВИТЕЛЬНО используется для самих entry/SL/TP — расхождение касается только
опознавательных ярлыков-факторов, не самой сделки. Не предлагаю исправление (это меняет
текст, который видят подписчики на живых сигналах — граница "не менять сигнальную логику
без одобрения" применяется и к формулировкам факторов, не только к числам) — фиксирую
находку для решения владельца.

## Как читать этот документ

🟢 = реализовано и по механике соответствует канону. 🟡 = есть механизм, но другой/грубее/
изолированный (используется не там, где методичка рекомендует). 🔴 = нет вообще, ИЛИ
задекларировано в коде, но никогда не срабатывает (мёртвое поле — уточнено отдельно, где
применимо). Ничего из этого не должно внедряться/меняться без отдельного явного
одобрения владельца — документ только для решения по П4/будущим предложениям.
