# RESTORE_GUIDE.md — восстановление данных crypto-bot

Процедуры проверены реально (не теоретические), результаты — ниже с датой прогона.
Боевые файлы при тестировании НЕ трогались — восстановление делалось в `/tmp`, отдельно
от продакшена.

## 1. Восстановление журнала сигналов (`journal/signals.json`) из дневного бэкапа

**Источник истины**: `origin/main:backups/<YYYY-MM-DD>/signals.json` — версионированный
снапшот, создаётся раз в сутки в 03:00 (TZ бота) задачей `run_daily_backup` (`bot.py`,
ROADMAP П1.4). В отличие от рабочего `journal/signals.json` (last-write-wins,
перезаписывается), у файлов в `backups/` есть история — каждая дата свой путь, старые не
трутся.

**Процедура:**
```bash
cd ~/crypto-bot
git fetch origin
# Список доступных дат бэкапа:
git log --oneline --all -- 'backups/*/signals.json'
# Скачать конкретную дату (пример):
git show origin/main:backups/2026-07-10/signals.json > /tmp/signals_backup.json
```
Дальше — либо вручную положить в `journal/signals.json` и перезапустить бота (он подтянет
при следующем `startup_sync()`), либо восстановить в тестовый файл для проверки:
```python
import signal_journal as sj
sj.JOURNAL_FILE = "/tmp/restored_journal.json"  # НЕ боевой путь
import json
backup = json.load(open("/tmp/signals_backup.json"))
sj._journal = {int(k): v for k, v in backup["records"].items()}
sj._next_id = max(sj._journal.keys()) + 1 if sj._journal else 1
sj._save()
```

**Реально прогнано 2026-07-10, результат:**
- Бэкап от 03:00 (`backups/2026-07-10/signals.json`, коммит `6b55ef7`): 79 записей.
- Текущий боевой журнал на момент проверки (`journal/signals.json`, `origin/main`):
  82 записи (3 новых сигнала появились после 03:00 — ожидаемо).
- Все 79 id из бэкапа — подмножество текущих 82 id (`backup_ids.issubset(current_ids)`
  → `True`, ни одна запись не потеряна и не разошлась).
- Восстановление в тестовый файл: количество записей и их id совпали 1:1 с бэкапом
  (`RESTORE INTEGRITY CHECK PASSED`).
- **Вывод**: механизм дневного бэкапа реально работает и данные из него реально
  восстанавливаются без потерь. Проверено один раз (бэкап на момент проверки был
  единственный, от сегодняшнего 03:00) — стоит повторить проверку через несколько дней,
  когда накопится больше снапшотов, честно не считаю это железобетонно доказанным на
  все случаи по одному прогону.

## 3. Telethon-сессия `best_trade_reader.session` — backup ЕСТЬ, но статус НЕ подтверждён

**Резервная копия существует**: `~/crypto-bot-backups/20260710-003800/
crypto-bot-workdir-full.tar.gz` содержит копию `best_trade_reader.session` (входит в
полный снапшот рабочей папки, снятый в начале сессии 2026-07-10 перед git-реконсиляцией).

**Честно: эта копия НЕ проверена как рабочая.** На момент снятия backup (00:38) сессия,
возможно, УЖЕ была неавторизована — живой инцидент (см. PROGRESS.md, запись про Reader v5)
показал, что `reader.py` не может подключиться с признаками поломки, не датированными точно
(лог показывает последний успешный `✅ Telethon авторизован` 2026-07-02, крах-цикл начался
где-то между 2 и 10 июля). **Не выдаю недоказанное за факт**: копия сессии в бэкапе может
быть как рабочей, так и уже сломанной — не проверял и не буду проверять через ещё одну
попытку живого коннекта (уже дважды упирался в `is_user_authorized()=False` на копиях этой
сессии в этой сессии работы, см. PROGRESS.md).

**План восстановления доступа к каналам (реальный, не зависящий от валидности старой копии
сессии) — только через владельца:**
1. `cd ~/crypto-bot && python3 reader.py` в обычном терминале (интерактивный режим).
2. Ввести номер телефона + код подтверждения из Telegram/SMS один раз.
3. Дождаться в логе `✅ Telethon авторизован` — это создаст/обновит
   `best_trade_reader.session` заново, рабочим.
4. `Ctrl+C`, затем вернуть под управление launchd:
   `launchctl kickstart -k gui/$(id -u)/com.bestrade.reader`.

**Если понадобится откатиться на старую копию сессии** (например, если новая
переавторизация тоже почему-то не сработает) — команда есть, но начинать стоит с
переавторизации (шаги 1-4 выше), не с отката на непроверенную копию:
```bash
cp ~/crypto-bot-backups/20260710-003800/crypto-bot-workdir-full.tar.gz /tmp/
cd /tmp && tar xzf crypto-bot-workdir-full.tar.gz crypto-bot/best_trade_reader.session
# сравнить с текущим, скопировать вручную если решите использовать эту копию
```

**Рекомендация на будущее (ROADMAP-кандидат, не внедрено)**: сейчас `best_trade_reader.
session` версионируется только случайно (полными снапшотами рабочей папки на диске, если
кто-то их делает) — нет регулярного автоматического бэкапа именно этого файла, в отличие
от журнала/подписчиков (те бэкапятся ежедневно через `run_daily_backup`). Если потеря
сессии будет повторяться — можно рассмотреть регулярное копирование файла в защищённое
место (не в git — там нет места секретам/сессионным токенам) как отдельную задачу.
