# Order Block строится по телу свечи, не по хай/лоу — расхождение с частью боевого кода

**Полностью дублирует METHODOLOGY_CORE.md §18.1** — уже прочитано и
извлечено в прошлой сессии, обнаружено прямое расхождение с кодом.

**Правило источника**: "OB ВСЕГДА по телу, даже некрасивый OB с большим
фитилём — по телу, не по хвостам" — повторено многократно и явно в
бэктест-разборе.

**Сверка с ботом (Пакет 3 М3)**: `bot.detect_order_blocks()`
(`bot.py:1773-1791`) строит зону по `c["low"]`/`c["high"]` — то есть по
ПОЛНОМУ диапазону свечи ВКЛЮЧАЯ тени, а не по `open`/`close`. Не
терминологический нюанс — разница в фактических границах зоны входа/стопа.

**ПОПРАВКА (Пакет 5 М2)**: перепроверка показала, что у
`bot.detect_order_blocks()` НОЛЬ вызывающих — мёртвый код, ни один боевой
путь его не исполняет. Реальный живой OB-детектор — отдельный инлайн-код
внутри `pro_analysis()` (`bot.py:7401-7435`, третья геометрия зоны: микс
`open`+`high`/`low`+`open`), питающий `pro_score` только пампового
промоушена (`pump_detector.py`), не основной поток long/short.

**ПОПРАВКА 2 (Пакет 7 М1)**: вариант B реализован shadow-only —
`ta_extra.detect_order_block()` считает ОБЕ геометрии (live-зеркало
инлайн-кода `pro_analysis()` и methodology-чистое тело) на одних и тех же
4H-свечах, подключено в `shadow_engine.compute_shadow()` как patch 07.
Живой `pro_analysis()` НЕ тронут. Накопление shadow-данных для оценки
частоты расхождения — в процессе, решение о переносе в бой за владельцем.

**Честно**: статус фикса — подтверждённое расхождение реализации, не
решение о фиксе (граница CLAUDE.md — боевой скоринг требует явного "да"
владельца).
