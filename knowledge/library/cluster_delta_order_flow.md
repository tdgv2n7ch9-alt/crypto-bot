# 15. Кластерный анализ (поток 7.0) — честно НЕ полная построчная экстракция

**Прочитано выборочно (интро + несколько маркеров по ключевым словам "дельта"/
"поглощение" из 92-минутного/2568-строчного транскрипта), не построчно целиком
— причина ниже, не небрежность.**

**Ключевой, подтверждённый структурный вывод: это НЕ пробел в логике бота,
это пробел в ДОСТУПНЫХ ДАННЫХ.** Кластерный анализ = footprint/order-flow
чарты на специализированной платформе (в лекции — "RTT", Resonance Holding) с
данными по КАЖДОЙ сделке (объём покупок vs объём продаж внутри одного бара —
"дельта"). Пример из лекции: часовой бар с 13300 BTC на покупку против 12900
на продажу = дельта +389 (положительная) — при совпадении со свипом
ликвидности это трактуется как сильное подтверждение разворота ("очень
надёжно"). Цвет свечи явно назван НЕ информативным ("даже в бычьей свече
присутствует продажа, важно соотношение, не цвет").

**Это принципиально другой класс данных, чем то, что доступно текущим
источникам бота** (CoinGecko free OHLC, Bybit public klines) — оба отдают
только open/high/low/close/volume ПО БАРУ ЦЕЛИКОМ, без разбивки на
buy-vs-sell объём внутри бара. Получить такую дельту можно либо с платного
tick-level фида (Binance/Bybit aggTrades — доступно по API, но требует
отдельного, значительно более тяжёлого стрима данных и хранения), либо со
специализированной платформы вроде RTT/ATAS. **Не патч, не быстрый фикс —
вопрос инфраструктуры данных**, честно фиксирую как класс, отдельный от
обычных code-level GAPS в `KNOWLEDGE_GAPS.md` (не добавляю туда как обычный
пункт — там пункты подразумевают "код можно дописать", здесь это не так без
новой линии данных).

**Уже частично покрыто прокси-метрикой**: `ta_extra._volume_confirmed()` —
аномальный ОБЩИЙ объём (не направленная дельта) на баре свипа, доступный из
уже имеющихся OHLC-данных. Это грубее, чем "дельта покупки vs дельта
продажи", но той же природы (объём как подтверждение манипуляции) — уже
implемент, cross-referenced в `indicators_volume_rsi_atr_sessions.md`
(батч 12).

**Прочее содержание (платформенный тур RTT, не методология)**: интерфейс
кластерного графика, типы данных (дельта/объём/количество сделок), общий
принцип "не искать паттерн ради паттерна, важен контекст" (то же самое, что
уже звучало про order block в других уроках — красиво выглядящий паттерн без
контекста ничего не значит). Кластерный анализ явно НЕ назван основным
инструментом скальпера (скальпинг — это стакан/уровни/трендовые линии, другой
инструментарий).

**Модуль бота**: нет прямого — упомянутая аналогия `_volume_confirmed()`
(общий объём на свипе, уже есть) ближе всего. Настоящая дельта покупки/
продажи не реализована и не может быть реализована на текущих источниках
данных без отдельного tick-level фида.
