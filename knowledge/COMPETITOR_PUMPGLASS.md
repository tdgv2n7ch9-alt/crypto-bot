# Конкурентный разбор: pumpglass.com vs pump-radar v1/спека v2 (#274-276)

## Протокол правды

pumpglass.com отдаёт HTTP 403 прямому `WebFetch` (бот-детект) --
содержимое получено через reader-прокси (`r.jina.ai`, конвертирует
страницу в текст), тот же контент, что видит обычный посетитель сайта,
дата проверки 2026-07-18. Общее описание продукта (500+ пар,
Telegram-алерты) дополнительно подтверждено `WebSearch` (независимый
источник, совпадает). Конкретные названия колонок ниже -- **[проверено
через reader-прокси, 2026-07-18]**, не первоисточник напрямую (сайт
заблокировал прямой доступ) -- честно отмечаю разницу в уровне
уверенности.

## Что показывает pumpglass.com (у них)

**Главная таблица** [проверено]: Token · **Gain %** · **Peak %** ·
**Alert price** · Start price · Price now · **Duration** · Volume 24h ·
**OI (USD)** · **Funding**.

**Секция "At Peak NEW"** [проверено]: отдельный view "Tokens trading
near the top of their move" -- сортировка по Gain%/Peak%, фильтры по
времени (All/24h/7d/30d). Это **персистентная, browsable таблица**
токенов, а не только push-алерты -- можно зайти и посмотреть историю
пампов за период, отсортировать.

**Telegram-бот** [проверено]: `@pumpglassbot`, "Get Telegram alerts".

**Общая механика** [WebSearch, независимо подтверждает]: сканирует
500+ фьючерсных пар 24/7, алгоритмическая детекция (не координированная
группа), скоринг сигнала, мгновенные Telegram-алерты.

## У нас (pump_detector.py, живое, сверено чтением кода)

**Push-алерт** (`pump_detector.py:1164-1261`, `_compose_alert()`):
%-движение от детекта, объём (x-множитель от нормы + Z-Score), Funding,
**OI ($ + % изменение за 5 мин + текстовая интерпретация матрицы**
цена/OI/funding, `_oi_matrix_label()`), сессия/killzone (с пометкой
качества), rug-риск строка (`rug_radar.format_rug_alert_line()`),
мемкоин-предупреждение по капитализации. **Богаче pumpglass по составу
ОДНОГО алерта** -- OI-матрица с готовой интерпретацией и rug-риск
pumpglass не показывает вообще (по крайней мере не проверено в его
табличном виде).

**Меню "⚡ Памп-радар"** (`pump_detector.get_pump_radar_state()`,
строки 263-298): активные наблюдения (symbol/stage/elapsed_min/
pct_from_level) + **агрегатные** 24ч-счётчики (detected/reversed/
promoted/expired). **НЕ персистентная browsable таблица по каждому
токену** -- нет Gain%/Peak%/Alert-price/Duration/OI/Funding КАК КОЛОНОК
для каждого активного/прошедшего пампа разом. Это структурное отличие
от "At Peak"-таблицы pumpglass -- у нас данные ЕСТЬ (считаются для
алерта), но не персистятся в браузable-вид.

## У нас в спеке (не реализовано, `NEXT_PACKAGE.md` #274-276)

Ни pumpglass [в проверенном объёме], ни наша спека не показывают

целиком token-centric (метрики ПРО токен: gain/OI/funding), у нас

## 3-5 идей (протокол правды: идеи, не факты о конкуренте)

1. **Alert-price-трекинг как персистентная таблица** -- у нас данные
   ДЛЯ этого уже есть (`detect_price`, `peak_price`/`bottom_price`,
   `pump_time`), не хватает ТОЛЬКО персистентного browsable-представления
   (аналог "At Peak"). Дешёвая фича относительно спеки волн 1-3 --
   переиспользует уже посчитанные поля `pump_watch`/`pump_history`, не
   новый источник данных. Кандидат для меню "⚡ Памп-радар" -- расширить
   `get_pump_radar_state()` полями Gain%/Peak%/Duration в явном виде.
2. **Duration как метрика фазы** -- у pumpglass Duration = время с
   алерта. У нас есть похожая идея (`elapsed_min`), но НЕ используется
   как фильтр/сортировка нигде видимого пользователю. Можно завести
   как явный признак "фазы" пампа (ранняя/поздняя стадия по времени +
   % от пика), а не просто счётчик минут.
3. **Голая OI/Funding-таблица менее ценна, чем наша интерпретация** --
   НЕ рекомендую копировать "голые числа" pumpglass буквально, у нас
   УЖЕ есть текстовая интерпретация матрицы (`_oi_matrix_label()`) --
   это преимущество, не пробел. Сохранить как отличие, не размывать
   под "голые цифры как у них".
4. **"At Peak"-фильтр (24h/7d/30d) как паттерн UI, не как копия
   функции** -- у нас нет ничего подобного для истории. Полезный
   UI-паттерн для будущего Mini App (#292, `best_trade_radar_v2.html`
   прототип уже содержит похожую идею временных фильтров по макро-ленте)
   -- переиспользовать паттерн "фильтр по периоду" в дизайне Radar v2,
   не обязательно 1:1 копировать структуру pumpglass.
5. **НЕ копировать слепо "чистый скринер"-философию** -- pumpglass
   целиком about "что движется прямо сейчас" (token-centric,
   алгоритмическая детекция без attribution). Наша философия "торгуем
   оператора, не токен" (whale/order-flow контекст, rug-риск,
   OI-матрица-интерпретация) остаётся ядром -- если добавлять
   персистентную таблицу истории пампов (идея 1), она должна нести
   ТЕ ЖЕ доп. слои (rug/OI-интерпретация), не превращаться в голый
   клон "Gain%/Peak%/Duration без контекста".

## Честные ограничения этого разбора

- Цены/тарифы pumpglass НЕ проверены (не в объёме задания владельца --
  запрошены только метрики таблицы).
- "Публичный трек-рекорд" pumpglass -- не нашёл отдельного явного
  раздела "доказанная история сигналов, не удалённая задним числом"
  (важное отличие от голой таблицы, которую МОЖНО редактировать) --
  честно не проверено, не домысливаю.
- Разбор основан на ОДНОМ снятии через reader-прокси (не живое
  многодневное наблюдение) -- сайт мог показать неполную/A-B-тестовую
  версию страницы.
