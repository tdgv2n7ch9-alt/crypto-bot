## 0. Постоянное правило (владелец, бессрочно, 2026-07-18)

**В ЛЮБОМ подписчицком тексте** (канал, карточки, бот, mini app, посты)
**запрещено** упоминание внутренних методологических источников: имена
"Kira", "ICT", имена аналитиков (Королев и др.), названия
каналов-источников, названия курсов. Замены: «чеклист N/6», «фильтры»,
«авторская разметка», «методология BEST TRADE». Внутренние документы
(`METHODOLOGY_CORE.md`, `PROGRESS.md`, owner-чат) — без ограничений.

**Обязательная проверка перед деплоем ЛЮБОГО публичного формата**:
grep-аудит всех исходящих шаблонов на запрещённые токены (регистронезависимо):
`kira`, `ict`, `королев`, `korolev`, `тт`. Кандидат-реализация — тест
`tests/test_public_text_no_source_tokens.py`, который проходит по ВСЕМ
`format_*`/`_render_*` функциям, формирующим подписчицкий (не owner/[SYS])
текст, с синтетическими входными данными, и матчит результат против
списка запрещённых токенов через `re.search(r'\b(kira|ict|королев|korolev)\b',
text, re.IGNORECASE)`. Токен `тт` намеренно НЕ включён в regex-паттерн
буквально (слишком короткий и частотный — совпадёт с "тест", "путь" и
т.п. ложно-положительно) — для него нужен более узкий шаблон (например,
только как отдельно стоящее слово/аббревиатура в характерном контексте
"ТТ" в верхнем регистре с границами слова) — уточнить у владельца точный
паттерн перед реализацией теста, честно не выдумываю его сейчас.

## 1. Три поверхности

### КАНАЛ

Только продуктовые посты: карточки сигналов (канонический формат, см.
раздел 2), обзор фаз/рынка, ежедневный/еженедельный дайджест, **еженедельный
трек-рекорд из shadow-журнала** (агрегированная статистика — WR/PF по
методологии, без упоминания источников по правилу 0).

**НЕ в канале**: весь `[SYS]`-трафик — уже структурно выполнено сегодня
(`bot.py:159` `SYSTEM_MESSAGE_PREFIX`, `send_system()` шлёт ТОЛЬКО в
`OWNER_CHAT_ID`, не в канал/подписчикам — подтверждено чтением кода, это
НЕ требует изменений для выполнения части 1).

**Инфраструктура частично уже есть**: `CHANNEL_ID` env var уже
используется (`bot.py:6842`, whale-алерты) — канал-рассылка не с нуля,
переиспользовать существующий механизм для остальных типов сообщений
(зона-тач/лимитки/памп/event/rug), не изобретать новый.

### БОТ

Меню с «← Назад» на каждом уровне (сейчас — частично: `_mv2_render_*`
функции существуют для разных разделов меню v2, `bot.py:4636-5114`, но
честно НЕ проверено при написании этого дизайна, есть ли единообразная
кнопка "Назад" на КАЖДОМ уровне — это должен показать аудит части 4,
конкретные пробелы не описываю здесь без проверки).

Глоссарий inline — новая фича, термины методологии (OB/FVG/BOS/CHoCH/
свип и т.п.) по запросу прямо в карточке/меню, без выхода в отдельный
документ. Настройки фаз-подписки (какие типы алертов подписчик хочет
получать — сейчас единый поток, нужна фильтрация). Калькулятор позиции
(частично уже есть — `card_v2.compute_capital_table()`, вопрос в
доступности как отдельного инструмента вне карточки).

### MINI APP

Дашборд по прототипу `knowledge/best_trade_radar_v2.html` (владелец,
2026-07-18, утверждённый визуальный прототип) через Telegram WebApp API
— кнопка «Радар» в главном меню бота. Прототип уже содержит: макро-лента,
шапка с честным таймштампом (не голый "LIVE"), фазовая структура
(судя по CSS-переменным `--ph1`-`--ph4`). Полный разбор прототипа —
отдельный шаг перед реализацией (не в объёме этого дизайн-документа,
только фиксирую источник истины по внешнему виду).

**Технически**: Telegram Mini App = веб-страница, открываемая внутри
Telegram через `WebApp.init()`/`Telegram.WebApp` JS API, кнопка в боте —
`InlineKeyboardButton(web_app=WebAppInfo(url=...))`. Требует HTTPS-хостинг
статики (Railway уже хостит `miniapp_api.py` — проверить, можно ли туда
же добавить статическую раздачу прототипа, или нужен отдельный сервис).

## 2. Канонический формат карточки сигнала

**Шаблон** (владелец, дословно):

```
[направление · тикер · фаза · PD]
———
риск-план: DCA / SL / TP / R:R
———
риск оператора + L/S
———
чеклист N/6
———
«Почему» — одной строкой
———
[кнопка 1] [кнопка 2] [кнопка 3]
```

**Единый шаблон для ВСЕХ контуров** — сейчас (найдено в разведке для
этого документа, честно не полный список): `card_v2.py` уже реализует
БЛИЗКИЙ формат (светофор/вердикт/6-пунктовый чеклист `format_strength_
block()`/что делать/за-против/контекст/капитал/тайминг) — но НЕ все
типы алертов через него проходят. Отдельные, СВОИМИ форматами живут:
zone-touch (`bot.py:7947` "Watchlist ALERT"), ЛИМИТКИ
(`bot.py:4417` `_limitki_execution_card_text`), EVENT-RADAR
(`event_radar.py:386` `format_event_alert`), RUG (`rug_radar.py:251,263`),
BANK/AKE setup-монитор алерты (`bank_setup_monitor.py:198-215`). Полная
инвентаризация — часть 4 ниже.

**Служебные блоки убрать из подписчицких карточек** (пример "#284-стиль"
— честно, не смог найти в коде явную запись, что конкретно означает
"#284-стиль" блок без доступа к самой карточке #284 живьём — нужно
уточнение у владельца ЧТО именно убирать, не гадаю). Owner-версия
карточки (полная, с диагностикой/техническими деталями) остаётся
отдельно, подписчицкая — только канон выше.

## 3. Пост-исход: авто-reply при закрытии

**Дизайн** (переиспользует уже готовую инфраструктуру, ничего с нуля):
`signal_journal.py`'s `_notify_outcome()` (`signal_journal.py:503-516`)
УЖЕ отправляет уведомление владельцу при закрытии сделки — расширить
на подписчицкий канал (если карточка была опубликована туда) через
`reply_to_message_id` (Telegram API поддерживает нативно), используя
`origin_msg_id`/`origin_chat_id` (уже есть в схеме записи, `signal_
journal.py:441`, добавлено для #272 "кликабельные алерты").

**Данные для reply**: результат (TPn/SL_HIT) — уже есть
(`rec["outcome"]`). **MFE/MAE** — теперь доступны после сегодняшнего
`mfe_mae.py` (очередь п.6): `rec["mfe_price"]/rec["mae_price"]` (живой
трекер) или `mfe_mae.running_mfe_mae_r()` для перевода в R-мультиплы.
Инфраструктура для этой части ГОТОВА уже сегодня — реализация самого
reply-сообщения (текст + вызов `reply_to_message_id`) не начата.

## 4. Аудит текущих форматов — маппинг на канон, что ломается

| Тип сообщения | Файл:функция | Формат сейчас | Соответствие канону | Что ломается при переходе |
|---|---|---|---|---|
| ТОП ЛОНГ/ШОРТ (AUTO) | `bot.py:send_scheduled()` + `card_v2.py` | Частично card_v2 (светофор/вердикт/чеклист/капитал) | 🟡 Близко к канону, но не 1:1 (иной порядок блоков, нет явного "риск оператора+L/S" как отдельного блока в текущем шаблоне) | Порядок блоков, добавить операторский риск/LS явным блоком |
| Zone-touch (watchlist) | `bot.py:7947`, `check_watchlist()` | Ad-hoc текст, "Watchlist ALERT: SYM @ price" | 🔴 НЕ соответствует канону (нет чеклиста/риск-плана в этом виде — сама природа alert'а другая, это триггер, не полная карточка) | Нужно решить: zone-touch остаётся отдельным ЛЁГКИМ форматом (уведомление "коснулось зоны, см. карточку") или тоже разворачивается в полный канон -- design-вопрос владельцу |
| ЛИМИТКИ (author-зоны) | `bot.py:4417` `_limitki_execution_card_text` | Свой формат: зона/лестница DCA/SL/размер позиции/ликвидации/rug-строка | 🟡 Близко (DCA/SL/риск уже есть), но нет чеклиста N/6 (author-зоны разметки 1:1, чеклист туда концептуально не подходит -- это НЕ авто-сигнал с фильтрами) | Author-зоны, вероятно, ДОЛЖНЫ остаться отдельным форматом -- канон описывает АВТО-сигналы с чеклистом фильтров, ручная разметка автора этой логике не подчиняется по определению ("применяется 1в1") |
| EVENT-RADAR | `event_radar.py:386,502` | Заголовок + событие (листинг/делистинг), без DCA/SL/чеклиста | 🔴 Не применимо -- это НЕ торговый сигнал, а информационное событие | Не переводить в канон сигнала -- разный класс сообщений, отдельная категория "события", не "точки" |
| RUG-ALERT | `rug_radar.py:251,263` | Строка риска, встраивается в другие карточки | N/A -- уже встроенный компонент, не отдельное сообщение | Без изменений |
| BANK/AKE setup-монитор | `bank_setup_monitor.py:198-215` | ChoCH/retest/invalidation алерты, свой формат | 🔴 Не применимо -- узкоспециализированный мониторинг конкретного разлока, не общий сигнальный поток | Отдельная категория, не смешивать с канон-карточками |
| WHALE-алерты | `bot.py:6863+` (`_WHALE_WATCH` цикл) | Funding/OI/LS сводка | 🔴 Не применимо -- это дериватив-дайджест, не точка входа | Отдельная категория |
| ПАМП-РАДАР | `pump_detector.py` (алерты не найдены как отдельный `format_*`, встроены в основной цикл) | Не проверено полностью в рамках этого дизайна | ⚪ Требует отдельного прохода по `pump_detector.py` -- честно не покрыто в этой итерации аудита | TODO следующая итерация |

**Честный вывод аудита**: канонический формат из части 2 применим
буквально к ОДНОМУ классу сообщений — авто-генерируемым точкам входа
(ТОП ЛОНГ/ШОРТ и подобные), где УЖЕ есть чеклист/скор/DCA-план. Остальные
6+ типов сообщений (zone-touch, ЛИМИТКИ, EVENT-RADAR, RUG-строка,
BANK/AKE-монитор, WHALE-дайджест) структурно НЕ являются "точками входа
с чеклистом" — попытка силой впихнуть их в канон исказила бы их смысл
(author-зона не имеет "чеклиста фильтров", у неё разметка владельца 1:1;
событие листинга не имеет DCA/SL). **Рекомендация (не решение)**: канон
части 2 — стандарт для КЛАССА "авто-точка входа", не универсальный
шаблон для ВСЕХ исходящих сообщений; zone-touch/ЛИМИТКИ/EVENT-RADAR и
т.п. получают СВОИ, более лёгкие стандарты (не описанные владельцем в
этом запросе -- нужно отдельное решение по каждому классу).

## 5. Оценка этапов

### (а) Канон карточек + чистка канала

**Объём**: унификация ТОЛЬКО класса "авто-точка входа" (см. вывод части
4) под канон части 2 -- переиспользует существующий `card_v2.py`
практически целиком, добавляет явный блок "риск оператора + L/S"
(частично уже считается для WHALE-контекста, `bot.py:6863+`, нужно
подключить как источник в саму карточку) + инвентаризация/удаление
служебных блоков + forbidden-token grep-тест (правило 0).

**Оценка**: средняя -- существующий `card_v2.py` даёт основу, не с нуля;
основная работа -- маппинг источников риска-оператора в карточку +
тест forbidden-tokens + ревизия текстов на предмет утечки источников.

### (б) Меню / Назад / глоссарий

**Объём**: аудит существующих `_mv2_render_*` (`bot.py:4636-5114`) на
единообразие кнопки "Назад" (честно не проверено в этом дизайне --
первый шаг реализации, не факт, что там дыра); новый inline-глоссарий
(термины -> короткие объяснения, вызов из карточки/меню); настройки
фаз-подписки (новая структура данных в `subscribers.py`? -- не
проверено, требует отдельного разбора схемы подписчиков).

**Оценка**: средняя-высокая -- глоссарий и настройки подписки — новые
фичи с нуля, не косметика поверх существующего.

### (в) Mini App MVP

**Объём**: HTTPS-хостинг статики прототипа (Railway, возможно через
`miniapp_api.py`, не проверено, поддерживает ли он статическую раздачу
или только API), `WebAppInfo`-кнопка в боте, JS `Telegram.WebApp` init,
подключение прототипа к РЕАЛЬНЫМ данным (сейчас прототип — статический
визуальный макет, не факт, что там уже есть fetch-логика к API бота --
не проверено).

**Оценка**: высокая -- новая поверхность с нуля (даже с готовым визуальным
прототипом дизайн ≠ работающее приложение с живыми данными), наибольший
объём среди трёх этапов.

**Рекомендация по порядку (не решение)**: (а) первым -- наименьший
объём, наибольшая переиспользуемость существующего кода, самый быстрый
видимый результат владельцу. (б) и (в) можно вести параллельно после
(а), если владелец решит распараллелить -- независимы друг от друга.
