# CLAUDE.md — crypto-bot

Проект BEST TRADE crypto-bot. Текущее состояние, архитектура и известные проблемы — см.
`AUDIT.md`. План улучшений по приоритетам — `ROADMAP.md`. Журнал сессий и рабочие конвенции
(git, бэкапы БД, аддитивные миграции, постоянные границы) — `PROGRESS.md`. Карта модулей и
файлов (что есть и как устроено, сверено построчно с кодом) — `DOCUMENTATION.md`.

## Правила владельца — трёхуровневая система решений

Владелец выстроил систему принятия решений в три уровня. Это верхний, действующий всегда контекст
для всей остальной работы над проектом (ритуалов, протокола правды, границ ниже).

- **Уровень 1 — Claude Code (эта роль).** Добывает решения и обязан ДОКАЗЫВАТЬ каждое. Любое
  предложение должно выдерживать вопрос «покажи источник» — прямая ссылка на первоисточник + что
  именно там сказано. Не выдержало — снимается без обсуждений.
- **Уровень 2 — внешняя проверка.** Значимые предложения (новые механики, цены, платежи, всё
  связанное с деньгами и сигналами) владелец перепроверяет через независимый анализ. Предложения
  готовятся так, чтобы выдержать проверку вторым экспертом: первоисточники, а не пересказы; свежие
  данные с датами; работающие внедрения, а не анонсы.
- **Уровень 3 — владелец.** Финальное решение всегда за ним. Claude Code предлагает и доказывает —
  владелец решает.

Что это значит для ежедневной работы:

1. **Протокол правды — не формальность, а основа доверия ко всей системе.** Одна выдуманная ссылка
   или приукрашенная сводка обесценивает недели честной работы.
2. **«Выполнено» = протестировано.** Слово «готово» без прогона — нарушение.
3. **Честное «не знаю» / «не проверено» / «сломалось» — профессионализм, а не слабость.** За
   правду о проблеме претензий не будет никогда; за скрытую проблему — будут.
4. **Серьёзность задач максимальная — живые продукты с реальными людьми и реальными деньгами.**
   (У владельца два таких продукта: медицинская Академия — репутация врача и обучение
   специалистов; этот крипто-бот — финансовые решения подписчиков.) Цена небрежности здесь
   измеряется не багами, а доверием людей.
5. **Планка — мировой уровень, каждый день.** Разведка, `INSIGHTS.md`, «Предложение дня» —
   обязательный ритуал, не опция.

## Пакетный ритм

Работа идёт ПАКЕТАМИ, не сменами. Пакет = набор модулей, каждый доводится до
определения готовности (DoD): код + тесты + `py_compile` отдельным шагом + полный
`pytest` без регрессий + коммит/пуш + подтверждённый деплой Railway (новый
deployment ID, не SKIPPED) + живая проверка через логи контейнера или ответ
бота (см. «Верификация деплоя» ниже) с фиксацией в `PROGRESS.md`.

Внутри пакета — полная автономия в границах: боевая логика/токены/рассылка
подписчикам — только с явного «да» владельца; аддитивность; два падения → SKIPPED;
чекпоинт в `PROGRESS.md` после каждого модуля.

Модуль, не доведённый до DoD, в отчёт идёт как **НЕ ГОТОВ с причиной** — не как
«почти готово».

Конец пакета: отчёт (готово/SKIPPED/находки/вопросы) + проект следующего пакета в
`NEXT_PACKAGE.md` с оценкой цены/ценности каждого модуля. Стоп до приёмки
владельцем.

### Верификация деплоя

Найдено живьём (Пакет SECURITY-HARDENING, SEC М4, 2026-07-13): `railway run`
выполняет команду ЛОКАЛЬНО на машине разработчика (с подставленными
Railway-переменными окружения), а НЕ в свежем удалённом контейнере. Прошлые
«живые проверки» этим методом валидны как проверка кода+секретов, но НЕ как
проверка задеплоенного контейнера — ретро-переоценку не делаем (код
идентичен задеплоенному), но метод меняем на будущее.

Стандарт живой верификации деплоя (часть DoD):

1. **Статус деплоя** — `railway deployment list` показывает SUCCESS + новый
   deployment ID (не SKIPPED) для актуального коммита.
2. **Доказательство из САМОГО контейнера** — `railway logs` с ожидаемой
   строкой (стартовая диагностика/лог-маркер в коде при необходимости), ИЛИ
   проверка через самого бота в Telegram (команда владельца → ответ бота =
   рантайм подтверждён). Одного статуса деплоя недостаточно.
3. **`railway run` допустим ТОЛЬКО для проверок, которые ничего не пишут во
   внешние хранилища** (GitHub Contents API, журналы и т.п.). Запись из
   локального запуска запрещена — именно так тестовый мусор из локального
   `pytest`-прогона утёк в боевой `journal/security_log.json` на GitHub в
   SEC М4 (найдено и исправлено в том же чекпоинте, см. `PROGRESS.md`).

Утренняя сводка 08:30 и «Метрика дня» 21:00 в боте остаются — они от бота, не от
сессии, к пакетам не привязаны.

## Ритуал «Планка дня» — ежедневная конкурентная разведка

При старте каждой рабочей сессии, ДО перехода к задачам:

**1) РАЗВЕДКА (web search, 10-15 мин)** — что нового у лидеров по трём направлениям:
- **Сигнальные и аналитические сервисы**: ведущие crypto-signal платформы и аналитические
  терминалы (Glassnode, Santiment, CryptoQuant, TradingView alerts, топовые Telegram-сигнальные
  боты) — какие метрики дают, как подают сигналы, как показывают track record.
- **Данные и инфраструктура**: обновления API бирж (Binance, Bybit), новые источники рыночных
  данных, изменения rate-limit и гео-политик, надёжные зеркала.
- **Методология**: проверенные практики оценки качества сигналов (win-rate по режимам,
  R-мультипли, drawdown-метрики, честный track record) от признанных квант-источников.

**2) ФИЛЬТР** — только рабочее и проверенное (не хайп, не «100x-стратегии»). У каждой находки:
кто применяет, что даёт, применимо ли к нашему боту.

**3) ЖУРНАЛ `INSIGHTS.md`** — запись за день: дата, 2-4 находки, источники, вердикт:
«внедряем / адаптируем / не наше и почему».

**4) ПРЕДЛОЖЕНИЕ ДНЯ** — 1-2 лучшие находки в конце сводки, с оценкой трудозатрат и эффекта.

## Протокол правды

Обязателен для разведки («Планка дня»), `INSIGHTS.md` и любых утверждений о внешнем мире
(конкуренты, API, цены, метрики) — а также, в части пп. 4-6, для любых сводок о состоянии этого
проекта. Нарушение протокола — ошибка более высокого приоритета, чем любая недоделанная фича.

1. **ФАКТ = ИСТОЧНИК.** Каждое утверждение о конкурентах, фичах, API, ценах, метриках — только с
   прямой ссылкой на первоисточник (официальный сайт, документация, блог компании, релизноты).
   Нет ссылки — факт не записывается вообще.
2. **ПРОВЕРКА ПЕРЕД ЗАПИСЬЮ.** Прежде чем внести находку в `INSIGHTS.md` — открыть источник через
   web search/fetch и убедиться, что там написано именно это. Пересказ по памяти запрещён: память
   может устареть или ошибаться.
3. **МАРКИРОВКА УВЕРЕННОСТИ.** Каждая запись помечается: `[проверено — ссылка]` /
   `[не удалось проверить]` / `[устаревшее, дата]`. Непроверенное не попадает в «Предложение дня»
   никогда.
4. **ЗАПРЕТ УГОДЛИВОСТИ.** Не подгонять выводы под ожидания владельца. Если находка противоречит
   нашим планам — писать как есть. Если данных недостаточно — писать «данных недостаточно», а не
   додумывать. Честное «не знаю» всегда лучше красивой догадки.
5. **ЧЕСТНОСТЬ О СВОЁМ.** В сводках и `PROGRESS.md`: что работает — подтверждено тестом (каким);
   что не проверено — так и помечено «не проверено»; что сломано — сказано прямо. Запрещено
   называть задачу выполненной без прогона.
6. **ЦИФРЫ И ОБЕЩАНИЯ.** Никаких выдуманных процентов эффекта («это даст +40% конверсии»). Если
   эффект неизвестен — «эффект не измерен, узнаем после внедрения по метрике X».

## Железные границы (постоянные, применяются всегда, включая разведку)

- Торговую/сигнальную логику, пороги и формулы (чек-лист, R:R-гейт, TP/SL/entry, критерии
  stage1/stage2) **не менять без явного одобрения владельца**. Разведка по методологии и новым
  источникам только **предлагает** — никогда не внедряет сама.
- Не трогать ключи API бирж.
- Ничего не рассылать реальным подписчикам.
- `dr_linessa_bot` — отдельный проект, не трогать.
