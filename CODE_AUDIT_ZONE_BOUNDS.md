# CODE_AUDIT_ZONE_BOUNDS.md — развёрнутые границы зоны (lo>hi), read-only

Вечный бэклог `CLAUDE.md` п.7 ("дублирование зонной логики") — узкий класс
внутри более широкого паттерна "zone lo/hi containment check", встречающегося
~13+ раз по проекту (`ta_extra.py`/`signal_loop.py`/`level_watch.py`/
`card_v2.py`/`bot.py`/`shadow_engine.py`/`backtest/journal_backfill.py`).
Большинство мест ЗАЩИЩЕНЫ от случая `lo > hi` (`min(lo,hi), max(lo,hi)` перед
сравнением, например `signal_journal._touches_entry()`, `card_v2.py:68`,
`bot.py:4493/4546/4659/4765`, `shadow_engine.py:815`) — это НЕ находка, это
норма.

## Находка: `level_watch.py` — 2 функции БЕЗ этой защиты

`level_watch.zone_state()` (`level_watch.py:245-257`) и `level_watch.
distance_pct()` (`level_watch.py:262-269`) делают `lo, hi = zone["lo"],
zone["hi"]` напрямую, без `min()/max()`-guard'а, в отличие от всех
остальных ~11 мест того же паттерна в проекте. Если `zone["lo"] > zone
["hi"]` — `lo <= price <= hi` всегда False (пустой диапазон), функция
молча вернёт "не в зоне"/"далеко" даже когда цена реально между двумя
границами.

**Живая проверка данных** (`journal/watch_zones.json`, 34 зоны на момент
проверки): **0 из 34 реально развёрнуты** — риск сейчас чисто
теоретический, не воспроизведённый на боевых данных. Валидации на запись
зон (гарантии `lo<=hi` при создании), защищающей от будущего развёрнутого
случая, НЕ найдено (`grep` на "validate.*zone"/явную проверку порядка при
добавлении зоны — пусто).

**Оценка риска**: низкий (0/34 живых случаев, простой однострочный фикс
по образцу уже существующего в 11+ других местах паттерна), но
структурная асимметрия реальна — `level_watch.py` единственный модуль
с этим классом проверки без защиты. Кандидат для фикса ИЗВЕСТНЫМ
паттерном (Класс B по автономии v2 CLAUDE.md — тот же фикс, что уже
применён в 11 других местах, не новая архитектура) — **не применяю
