# SIGNAL_CARD_2.0_SPEC.md — спецификация «Карточка сигнала 2.0» (ночная сессия #2/#3, Блок 11)

Документ, НЕ код. По `METHODOLOGY_CORE.md` (весь свод правил, накопленный за
проект) + находкам конкурентной разведки (`INSIGHTS.md`, 2026-07-10 ночь: ни один из
3 проверенных конкурентов — Cornix/3Commas/Santiment — не публикует честный R-based
track record, это уже наше отличие). Цель: не "добавить больше полей", а сделать
карточку **проверяемой** — каждое число со своим источником, каждое решение (грейд,
вход, гейт) с объяснением ПОЧЕМУ, без переизобретения уже работающих частей.

## 1. Что уже есть сейчас (честная опись, не критика)

Текущая карточка (`bot._build_signal_post`, ~250 строк) уже содержит: направление
(лонг/шорт/спот), rocket score + грейд (A+/A/B/C) с рекомендацией по размеру
позиции, DCA-вход (3 транша), SL, TP1/2/3 с % от входа, факторы "за"/"против"
(RSI/MACD/тренд/Supertrend/EMA), R:R. `fa_engine.render_full_analysis_card` (для
`/full`) даёт более полную версию — 13 блоков с расшифровкой каждого. Это уже
СИЛЬНАЯ база — карточка не примитивная, задача 2.0 не "переписать с нуля", а
закрыть конкретные, накопленные за ночи пробелы.

## 2. Чего честно не хватает (по METHODOLOGY_CORE + этой ночи)

1. **Источник R:R неясен для получателя карточки** — сейчас показывается число
   (например "R:R 1:1.8"), но не откуда оно взялось: структурная зона? ATR-фоллбэк?
   Методика (§9) явно требует, чтобы R:R был обоснован структурой, не угадан —
   карточка должна честно писать "R:R от структуры (2 касания)" vs "R:R от ATR-
   фоллбэка (нет чёткой структуры)" — сейчас это поле `levels_source` УЖЕ считается
   (`fa_engine.py`), но не всегда доходит до итоговой карточки читаемо.
2. **Нет explicit invalidation-условия** — методика (§1, §11) явно разделяет "сделка
   работает, пока НЕ произошло X" (например, слом structure против direction, потеря
   K-LVL зоны) — сейчас карточка даёт SL (цену), но не текстовое условие "почему
   именно здесь", помимо голого числа.
3. **Killzone-качество не подсвечена явно на карточке AUTO/TOP-сигналов** —
   `fa_engine`-версия учитывает killzone в чеклисте (Блок 5), но короткая карточка
   `/top`-style не показывает, был ли сигнал во время хорошей сессии — методика (§8)
   явно считает время сигнала фактором качества.
4. **Confluence-факторы (breaker/MB, BPR, дивергенция) нигде не отображаются** —
   даже как информация — уже посчитаны в shadow-контуре (ночь #2, Блок 1), но
   живому получателю сигнала недоступны вообще, ни на одной карточке.
5. **Нет честного "почему НЕ вошли бы 2 месяца назад"-track record рядом с сигналом**
   — конкуренты (Cornix/3Commas) тоже этого не делают (см. `INSIGHTS.md`), но у нас
   уже ЕСТЬ данные (`signal_journal`/`BACKTEST_REPORT.md`) для честной строки
   "исторически по этому источнику: win rate X%, средний R Y" — не отображается на
   самой карточке сигнала, только в отдельной команде `/journal`.
6. **DXY/S&P500-конфлюенс посчитан, но недоступен на карточке** (`MISMATCH_REPORT.md`
   п.12) — данные есть, отображения нет.
7. **Мемкоин-риск и degraded_data — есть как внутренние флаги, не всегда явно и
   единообразно показаны на ВСЕХ типах карточек** (AUTO/x100/Памп-радар/ручной
   `/full` — разные функции рендера, разная полнота).

## 3. Спецификация карточки 2.0 — структура

Не revolutionary — эволюционная, добавляет 4 новых блока к уже существующей
структуре, ничего не убирает:

```
[Заголовок] {ЛОНГ/ШОРТ} {SYMBOL} — {грейд A+/A/B/C} ({rocket score}/100)
[Источник]  {AUTO-скан / x100 / Памп-радар / ручной /full} · {источник R:R: структура/ATR-фоллбэк}

[Вход]      DCA: 50%@{entry1} / 30%@{entry2} / 20%@{entry3}
[SL]        {sl} ({sl_pct}%) — {invalidation-условие текстом, НЕ только число}
[TP1/2/3]   {tp1}({R:R} / {%}) {tp2}(...) {tp3}(...)

--- НОВОЕ: Контекст (2.0) ---
[Killzone]  {текущая сессия} — {качество A+/A/B/C/D, см. §8} {если Dead Zone — явное предупреждение}
[Confluence]{если есть} Breaker/MB: {тип} · Дивергенция: {классическая против направления, если есть} · BPR: {да/нет}
[Макро]     DXY {знак+направление} · S&P500 {знак+направление} (см. MISMATCH_REPORT п.12 — сейчас НЕ считается в гейте, только информационно)

--- НОВОЕ: Честность (2.0) ---
[Track record источника] "{источник}: N закрытых сигналов, win rate X%, средний R Y" (из signal_journal, если N>=5 — иначе "недостаточно данных")
[Данные]    {все источники живые} ИЛИ {degraded_data: список недоступных полей, честно "н/д" вместо нулей}

[Факторы]   {существующий список ОК/против, без изменений}
[Риск]      {существующая таблица размера позиции на 1/2/3% депозита}
```


## 5. Пример ПОСЛЕ (карточка 2.0, тот же сигнал, добавленные блоки отмечены →)

```
🟢 ЛОНГ BTC — грейд A (78/100)
→ AUTO-скан · R:R от структуры (2 касания зоны)

Вход: $61500 / $61200 / $60900
SL: $59800 (-2.8%) → инвалидация: слом структуры ниже $59800 (потеря бычьего HL)
TP1: $63200 (+2.8%, R:R 1:1.8) TP2: $64500 TP3: $66800

→ --- Контекст ---
→ Killzone: 🇬🇧 Лондон Open (A+) — хорошее время для входа
→ Confluence: Breaker (не Mitigation) на входной зоне · дивергенции нет · BPR: нет
→ Макро: DXY ↓ (в пользу лонга) · S&P500 ↑ (в пользу лонга) — информационно, не гейт

→ --- Честность ---
→ Track record AUTO-скана: 30 закрытых сигналов, win rate 78%, средний R +0.15
→   (честно: R:R-поле этого источника расследуется, см. RR_INVESTIGATION.md)
→ Данные: все источники живые

Факторы:
✅ RSI в зоне покупок
✅ MACD бычий
❌ Тренд не подтверждён
```

## 6. Что НЕ делать (границы спеки)

- Не показывать confluence-факторы как ГЕЙТ (блокировку сигнала) — только
  информационно, пока shadow-данных недостаточно для решения о гейте (см.
  `SHADOW_ANALYSIS.md`).
- Не выдумывать track record там, где сделок <5 — честное "недостаточно данных для
  трек-рекорда", не пустое поле и не подогнанное число.
- Не менять сам расчёт R:R/SL/TP этой спекой — только отображение уже посчитанного
  и явно недостающих полей (`levels_source`, invalidation, killzone-качество).
