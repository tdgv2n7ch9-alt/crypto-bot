# CODE_AUDIT_ASYNC_BLOCKING.md — синхронные блокирующие вызовы внутри async-функций

Read-only аудит (вечный бэклог `CLAUDE.md` п.7: "синхронные вызовы в async-хвостах") —
список с файл:строка, БЕЗ правок. Метод: AST-скан (не regex) всего репозитория (исключая
`.claude/worktrees/*` — там найдены артефакты СТАРОГО кода прошлых forked-агентов, не
текущее состояние main), ищущий вызовы `requests.get/post/put/delete/request`,
`time.sleep`, `urllib.urlopen` внутри тела `async def` — такие вызовы блокируют ВЕСЬ
event loop (включая APScheduler-джобы AUTO/bsc_wallet_monitor/whale_radar в том же
процессе) на время сетевого таймаута, а не только текущий handler.

## Находка (1, во всём репозитории)

**`bot.py:13025`, внутри `async def _search_coin_by_symbol(symbol)`** (`bot.py:13019`):

```python
r = requests.get(url, headers=headers, params=params, timeout=15)
```

Прямой синхронный `requests.get` с `timeout=15` внутри `async def` — блокирует event loop
до 15с в худшем случае (таймаут CMC API), пока ждёт сеть.

**Путь вызова**: единственный вызывающий — `_do_full_analysis()` (`bot.py:13138`,
`coin = await _search_coin_by_symbol(symbol)`) — fallback, срабатывает ТОЛЬКО когда символ
не найден в локальном кэше `get_all_coins()` при команде `/full SYMBOL`. Не часть
периодического джоба (AUTO/signal_loop/bsc_wallet_monitor) — редкий, по требованию
конкретного пользователя, не системный тик.

**Оценка риска**: низкий-средний. Не системный (не влияет на все тики постоянно, только
на тот момент, когда КОНКРЕТНЫЙ пользователь вызывает `/full` с незнакомым символом), но
при срабатывании блокирует ВЕСЬ бот (включая AUTO/bsc_wallet_monitor тики других
подписчиков) на длительность реального сетевого таймаута CMC — тот же класс проблемы, что
уже чинили в других местах (`run_in_executor`, см. ниже), просто с более редким триггером.

**Установленный паттерн фикса — уже используется в этом же файле** (не нужно изобретать
новый подход): `run_in_executor(None, ...)` уже применяется для аналогичных блокирующих
вызовов в `bot.py` (строки 4222, 6839, 6843 — например
`await loop.run_in_executor(None, get_binance_ohlc, sym, "2h", 120)`). Фикс для
`_search_coin_by_symbol` — обернуть строку 13025 тем же способом:
`r = await loop.run_in_executor(None, functools.partial(requests.get, url, headers=headers, params=params, timeout=15))`.

## Что НЕ сделано (сознательно, read-only аудит)

- Фикс НЕ применён — известный паттерн, но код не трогаю без отдельного "да" (тот же
  owner-gate, что и для готового `except:pass`-фикса).
- Не проверено практикой, как часто `/full SYMBOL` реально попадает в этот fallback
  (нет метрики "сколько раз символ не нашёлся в `get_all_coins()`") — оценка риска
  "низкий-средний" по логике кода, не по частоте в проде.
- Другие классы блокирующих вызовов (`open()` на большие файлы, CPU-bound синхронные
  вычисления внутри async — например тяжёлый `pandas`/`numpy` расчёт без `run_in_executor`)
  этим сканом НЕ покрыты — скан целенаправленно узкий (сетевые/sleep блокировки), не полный
  профиль производительности event loop.
