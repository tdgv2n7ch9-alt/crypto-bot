# UNLOCK_TRACKER_ESTIMATE.md — оценка собственного unlock-трекера (EVENT-RADAR М1)

Владелец 2026-07-13: разлоки через DropsTab Basic ($19/мес) — НЕТ, не платим сейчас.
Вместо этого — оценка (не реализация) собственного трекера: публичные вестинг-графики
топ-30 + on-chain через Etherscan. Ниже — часы, покрытие, честные ограничения.
**Ничего не внедрено, только оценка на решение владельца.**

## Контекст (уже проверено живьём, см. `KNOWLEDGE_GAPS.md`)

3 источника разлоков (tokenomist.ai, CryptoRank, DropsTab) проверены живым WebFetch
2026-07-13 — у всех платный доступ к программному API с реальным охватом. Бесплатного
пути с широким покрытием не существует. Отсюда и запрос на оценку собственного решения.

## Два независимых компонента

### Компонент A: статический курируемый календарь вестинга (топ-30)

**Что это**: вестинг-графики (даты клиффа, % разлока по датам, объём в токенах)
публикуются проектами ОДИН РАЗ при запуске (tokenomics-документ на сайте/gitbook/
Notion/whitepaper) и меняются редко. Нет генерического бесплатного API, который отдаёт
это программно для многих проектов сразу (то же самое, что уже нашли для DropsTab/
tokenomist/CryptoRank) — но для ОДНОГО конкретного проекта эти данные обычно найти
можно вручную, открыв его собственную документацию.

**Формат**: курируемый JSON/YAML в репозитории (`data/unlock_schedule.json` по аналогии
с `etherscan_whale.KNOWN_EXCHANGE_ADDRESSES` — тот же принцип "статический список,
честно не претендует на полноту", уже есть прецедент в проекте), по проекту:
`{symbol, cliff_date, unlock_events: [{date, pct_of_supply, amount_tokens}], source_url}`.

**Честное покрытие**: топ-30 по market cap на практике включает BTC/ETH/XRP/BNB/SOL/
стейблкоины (USDT/USDC/...) — токены БЕЗ вестинга вообще (либо полностью в обращении,
либо алгоритмические). По опыту курса разведки рынка, реально VC-финансируемых токенов
с активным вестингом в топ-30 обычно **15-20 штук**, не 30. Честно, это не проверено
живьём сейчас (потребовало бы пройтись по актуальному топ-30 CoinGecko и для каждого
проверить тип токеномики) — оценка по опыту, не факт.

**Трудозатраты**:
- Разведка по ~15-20 проектам (найти официальный tokenomics-документ, живая проверка
  по Протоколу правды — открыть источник, а не по памяти) — **~30-40 мин на проект**
  (поиск + верификация + запись) = **~10-13 часов**.
- Схема данных + загрузчик (`unlock_schedule.py`, чтение JSON, вычисление "сколько дней
  до ближайшего разлока", фильтр >1% MCap) — **~2 часа**.
- Тесты (pure-function, тот же паттерн, что остальной проект) — **~1.5 часа**.

### Компонент B: on-chain через Etherscan (кросс-проверка, НЕ замена календаря)

**Важное честное ограничение**: on-chain мониторинг ловит только **УЖЕ СЛУЧИВШИЙСЯ**
перевод с известного вестинг/командного кошелька — это ретроспективное подтверждение,
не предсказание за 7/3/1 день, как просил владелец изначально в спецификации EVENT-RADAR
М1. Advance warning может дать ТОЛЬКО календарь (Компонент A) — Etherscan-часть годится
как проверка "разлок по расписанию действительно произошёл" (или НЕ произошёл — тоже
ценная информация, "перенесли/отменили").

**Ключевое честное ограничение 2**: чтобы мониторить конкретный вестинг-кошелёк, нужно
ЗНАТЬ его адрес — так же, как `etherscan_whale.py` честно признаёт для биржевых адресов
("Etherscan не отдаёт публичные label'ы... курируемый статический список"). Адрес
вестинг-контракта/командного кошелька для конкретного проекта либо публикуется самим
проектом (иногда), либо вычисляется вручную по паттерну переводов на биржи в прошлом
(ненадёжно, требует ручного анализа per-проект).

**Трудозатраты**:
- Найти/подтвердить вестинг-адреса для тех же 15-20 проектов из Компонента A — **сильно
  варьируется по проекту**, честно: не все публикуют адрес вообще. Оптимистичная оценка
  при удаче найти адрес для ~половины (7-10 проектов) — **~20 мин/проект = ~3 часа**.
- Код интеграции: инфраструктура УЖЕ ЕСТЬ (`etherscan_whale.py`, тот же троттлинг/API-ключ,
  Пакет 9 М4) — адаптация под вестинг-адреса вместо биржевых, тонкая обёртка —
  **~2-3 часа**.
- Тесты — **~1 час**.

### Компонент C: алерты 7/3/1 день + >1% MCap фильтр + интеграция в EVENT-DIGEST

- Логика "дни до события" из статического календаря (Компонент A) + фильтр по free-tier
  market cap данным (уже есть в проекте, CoinGecko) — **~2-3 часа**.
- Подключение в EVENT-DIGEST/утреннюю сводку (по образцу уже сделанного паттерна
  digest-секций этой ночью) — **~1.5 часа**.
- Живая проверка + деплой + DoD — **~1.5 часа**.

| A: схема данных + загрузчик + тесты | 3.5 |
| B: адреса вестинг-кошельков (частичный успех) | 3 |

## Честное сравнение с платным путём

| | Собственный трекер | DropsTab Basic $19/мес |
|---|---|---|
| Охват | ~15-20 крупных токенов, ручная курация | Тысячи токенов, автообновление |
| Точность дат | Средняя (публичные доки не всегда точны/актуальны) | Высокая (верифицировано платформой) |
| Advance warning | Да (из статического календаря) | Да |
| On-chain подтверждение | Частичное (не все адреса известны) | Да, встроено |
| Стоимость | ~25-30ч разовых + ~2-3ч/квартал | $19/мес = ~228$/год |
| Точка окупаемости по времени | Если час работы дороже ~$8/ч эквивалента, платный путь дешевле уже в первый месяц | — |

## Рекомендация (предложение, не решение)

Технически осуществимо, но узкое покрытие (15-20 из "топ-30") и умеренная точность
(ручная курация, не верифицированная программно против ончейн-контрактов вестинга)
делают это скорее "лучше, чем ничего" решением, чем полноценной заменой платного
сервиса. Разумный средний путь, если решено НЕ платить: реализовать ТОЛЬКО Компонент A
(календарь, ~14-17 часов) без Компонента B (on-chain кросс-проверка) — B добавляет
время без гарантированного покрытия (многие вестинг-адреса просто не публикуются).
Итоговая узкая версия: **~18-21 час** вместо ~25-30, с честной пометкой в самой фиче
"адреса вестинга не проверены on-chain, только по опубликованному графику".

Решение — за владельцем: (а) не делать вообще, отложить до готовности платить; (б)
узкая версия Компонент A only; (в) полная версия A+B+C.
