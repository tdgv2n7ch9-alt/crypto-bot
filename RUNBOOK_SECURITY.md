# RUNBOOK_SECURITY.md — инцидент-план (Пакет SECURITY-HARDENING М7)

Пошаговые действия для владельца на три сценария: утёк BOT_TOKEN, угнан GitHub-аккаунт,
спам/DDoS-атака на бота. Плюс кнопка `/lockdown` для мгновенной заморозки.

## Кнопка: `/lockdown` / `/unlock`

`/lockdown` (только OWNER, из чата с ботом) — мгновенно замораживает ВСЕ хендлеры кроме
владельца (`access_control.enforce()`, см. `access_control.py`). Любой не-OWNER chat_id
получает молчаливый отказ на любую команду, независимо от роли (VIP/TRIAL тоже
блокируются). Эффект мгновенный — флаг в памяти процесса меняется первым, до любой
попытки сохранить состояние в GitHub.

`/unlock` (только OWNER) — снимает заморозку, бот работает в обычном режиме.

**Состояние lockdown переживает рестарт контейнера** (best-effort синхронизация через
GitHub Contents API, `data/lockdown_state.json`, тот же паттерн, что `subscribers.py`/
`security_log.py`). Честное ограничение: если контейнер перезапустился ВО ВРЕМЯ активного
инцидента и GitHub в этот момент недоступен на старте — состояние восстановится в
"разблокировано" по умолчанию (безопасный дефолт для обычной работы, но НЕ для активного
инцидента). Если инцидент ещё не закрыт после любого рестарта бота — **проверьте статус и
при необходимости переиздайте `/lockdown` вручную.** Явного `/lockdown status` пока нет —
самый быстрый способ проверить: написать боту любой командой с постороннего (не-owner)
аккаунта и посмотреть, отвечает ли бот (не отвечает = lockdown активен ИЛИ этот аккаунт и
так не имеет роли — см. сценарий 3 ниже про то, как отличить).

## Сценарий 1: утёк BOT_TOKEN (Telegram)

Признаки: подозрительная активность от бота, которую вы не запускали (сообщения не по
расписанию, ответы на команды, которые никто не отправлял) — или прямое подтверждение
утечки (токен найден в публичном месте: git-история, скриншот, чат).

1. **Немедленно**: `/lockdown` из чата с ботом (замораживает всех, кроме вас, пока
   разбираетесь — не останавливает сам процесс бота, но глушит реакцию на посторонних).
2. **Ротация токена**: у @BotFather → `/mybots` → выбрать бота → `Bot Settings` →
   `API Token` → `Revoke current token` — старый токен немедленно перестаёт работать
   (Telegram отзывает его на своей стороне, это не наше приложение), генерируется новый.
   Процедура и живая проверка ротации — уже задокументирована в `SECRETS_AUDIT.md` §5
   (Пакет SECURITY-HARDENING М2).
3. Обновить `BOT_TOKEN` в Railway Variables (сервис crypto-bot) — новым значением.
   Railway автоматически перезапустит процесс с новым токеном.
4. Проверить деплой ожил: `railway logs` — строка `getMe "HTTP/1.1 200 OK"` с новым
   токеном в URL (последние 6 символов токена видны в логе HTTP-запроса — сверить, что
   это НЕ старый отозванный токен).
5. `/unlock` из чата с ботом — снять заморозку.
6. Секрет-скан истории (`gitleaks git --log-opts="--all"`) — убедиться, что НОВЫЙ токен
   никогда не коммитился (если утечка была из git-истории, а не из другого источника —
   старого коммита с токеном тоже стоит остерегаться на будущее, но переписывать
   git-историю задним числом рискованно и в этом пакете не делается без отдельного "да"
   владельца).

## Сценарий 2: угнан GitHub-аккаунт (или подозрение на угон)

Признаки: неожиданные коммиты не от вас/не от Claude Code, изменённые Personal Access
Token в настройках аккаунта, письма GitHub о входе с незнакомого устройства/IP.

1. **Немедленно со стороны GitHub** (не бот): Settings → Password and authentication →
   `Sign out of all sessions`. Если 2FA ещё не включена — включить немедленно (см. М6
   ниже, чек-лист).
2. **Personal Access Token, который использует бот** (переменная `GITHUB_TOKEN` в Railway
   Variables) — отозвать в GitHub (Settings → Developer settings → Personal access
   tokens → найти токен → Delete), сгенерировать новый с тем же минимальным набором прав
   (repo — на приватный репозиторий; не давать лишних scope).
3. Обновить `GITHUB_TOKEN` в Railway Variables новым значением. Все модули, использующие
   GitHub Contents API (`subscribers.py`, `signal_journal.py`, `security_log.py`,
   `access_control.py` для lockdown-состояния), подхватят новый токен на следующем
   вызове — рестарт процесса не обязателен, но `railway logs` стоит проверить на ошибки
   401/403 после смены (если старые запросы "в полёте" со старым токеном).
4. Проверить `git log origin/main` на посторонние коммиты за подозрительный период —
   честно оценить, требуется ли `git revert` (не force-push, не переписывание истории)
   на что-то из этого.
5. `/lockdown` не обязателен для этого сценария (угон GitHub не даёт прямого доступа к
   Telegram-боту), но стоит рассмотреть, если неясен полный масштаб (например, если тот
   же угнанный доступ мог задеть Railway Variables — тогда это уже сценарий 1 тоже).

## Сценарий 3: спам/флуд-атака на бота

Признаки: `EVENT_FLOOD_GUARD` в security-логе / алерт владельцу от
`_maybe_alert_owner_flood()` (>200 команд/мин суммарно по не-OWNER чатам, см.
`access_control.py`, Пакет SECURITY-HARDENING М3), либо массовый перебор инвайт-кодов
(`EVENT_AUTO_BAN` — уже автоматически банит атакующий chat_id при 5+ неудачных попытках
подряд, алерт владельцу приходит автоматически).

1. Штатный анти-абьюз (rate-limit 20 команд/мин на chat_id, автобан инвайт-перебора) уже
   работает без вмешательства — большинство спам-паттернов гасятся автоматически.
2. Если атака шире (много разных chat_id, скоординированно) — `/lockdown` глушит всех
   не-owner разом, пока не разберётесь, откуда атака и надо ли что-то менять (например,
   временно повышать флуд-порог не нужно, обычно наоборот).
3. Проверить `journal/security_log.json` (через `get_daily_summary()` или напрямую) —
   сколько `denied`/`rate_limited`/`flood_guard` событий, с каких chat_id, есть ли
   паттерн (один источник vs распределённая атака).
4. Для конкретных злоупотребляющих chat_id — `/revoke <chat_id>` (постоянный бан, роль
   NONE) вместо общего lockdown, если атака точечная, а не массовая.
5. `/unlock`, если был активирован lockdown, после того как ситуация под контролем.

## Честные ограничения этого раннбука

- Нет автоматического `/lockdown status` — проверка состояния сейчас требует либо
  вспомнить, вызывали ли вы `/lockdown` последним, либо протестировать сторонним
  аккаунтом (см. выше). Кандидат на будущий пакет, не сделано в этом.
- `/lockdown` не останавливает сам процесс бота и не отзывает токены — это ТОЛЬКО
  заморозка реакции на не-owner апдейты. Ротация токенов (сценарий 1) — отдельный,
  обязательный шаг для реального утёкшего секрета.
- Ни один шаг здесь не рассылает подписчикам и не трогает боевую сигнальную логику —
  соответствует железным границам `CLAUDE.md`.
