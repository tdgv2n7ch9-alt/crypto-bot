# INSIGHTS.md — журнал ежедневной конкурентной разведки

Формат записи за день: дата, 2-4 находки (что, кто применяет, источник), вердикт для каждой
(«внедряем / адаптируем / не наше и почему»), в конце — предложение дня (1-2 находки, трудозатраты,
эффект). Границы: разведка только предлагает, торговую/сигнальную логику сама не меняет — см.
`CLAUDE.md`.

Записей пока нет — журнал будет заполняться начиная со следующей рабочей сессии по ритуалу
«Планка дня».

## 2026-07-10 — первый прогон ритуала

### Находки

**1. Bybit REST API: лимит по умолчанию — 600 запросов за 5 секунд на IP**
[проверено — [bybit-exchange.github.io/docs/v5/rate-limit](https://bybit-exchange.github.io/docs/v5/rate-limit), прямой запрос страницы, дата проверки 2026-07-10]
Официальная документация Bybit v5 прямо указывает: "600 requests within a 5-second window
per IP" по умолчанию (≈120 req/s), с возможностью повышения лимита по заявке для VIP-тиров.
**Применимо к боту:** у нас уже есть `_bybit_kline_lock` (общий rate-limiter на Bybit-вызовы,
bot.py) — эта цифра даёт ориентир, насколько большой запас у текущего лимитера относительно
официального потолка. Не проверял, укладывается ли текущая частота вызовов бота в этот лимит
с большим запасом или близко к границе — отдельная проверка, не в рамках этой находки.
**Вердикт:** не наше (не требует действия) — источник только для справки, потолок далеко не
достигается при текущем масштабе бота (одна инсталляция, не HFT).

**2. Формула expectancy в R-мультиплях — стандарт индустрии для оценки торговой системы**
[проверено — [pnlledger.com/expectancy-r-multiples-the-plain-english-guide](https://www.pnlledger.com/expectancy-r-multiples-the-plain-english-guide/), прямой запрос страницы, дата проверки 2026-07-10]
Expectancy (в единицах R) = p × AvgWinR + (1 − p) × AvgLossR, где p — win rate, AvgWinR/
AvgLossR — средний R выигрышных/проигрышных сделок. Источник явно предупреждает не делать
выводов при маленькой выборке или "рваном" разбросе результатов, но НЕ называет конкретное
число сделок как порог — я специально попытался найти точную цифру (в другом источнике
всплывало "30+ предварительно, 100+ для надёжности"), но при прямой проверке эта цифра не
подтвердилась на первоисточнике, который я смог открыть — **[не удалось проверить]**,
не записываю как факт.
**Применимо к боту:** `signal_journal.get_journal_summary()` уже считает `avg_r` (средний
факт. R по закрытым сигналам) — по сути кусок expectancy-формулы уже есть, но именно
формулы `p × AvgWinR + (1-p) × AvgLossR` как явной метрики (отдельно win/loss средние, не
только общий avg_r) в `/journal`/`/stats` пока нет.
**Вердикт:** адаптируем — кандидат на маленькое дополнение к П2 (не новая инфраструктура,
чисто новая метрика поверх уже собранных `actual_r` в журнале). Не вносится сама, только
предложение.

**3. Binance API: изменение подписи запросов (percent-encoding) с 2026-01-15 — не применимо к боту**
[проверено — [developers.binance.com/docs/wallet/change-log](https://developers.binance.com/docs/wallet/change-log), прямой запрос страницы, дата проверки 2026-07-10]
С ~2026-01-15 07:00 UTC Binance требует percent-encoding payload перед вычислением подписи
для ПОДПИСАННЫХ (signed) эндпоинтов — иначе `-1022 INVALID_SIGNATURE`.
**Применимо к боту:** проверил по памяти сессии (AUDIT.md/этот же аудит) — прямые вызовы
`api.binance.com` в боте либо мёртвый код (см. `AUDIT.md` §1.1, П3), либо публичные
эндпоинты без подписи (klines/depth/24hr — не требуют API-ключа/подписи). Это изменение
касается ТОЛЬКО подписанных запросов — бот их к Binance не делает вообще.
**Вердикт:** не наше — проверил и явно исключил, не гадаю "на всякий случай".

### Отброшено при проверке (честно, не включаю как факт)
- Заявление о повышении `RAW_REQUESTS` лимита Binance до 300,000/5мин — встретилось в
  поисковой выдаче (сводке), но при прямом запросе официального changelog
  (developers.binance.com/docs/wallet/change-log) эта цифра НЕ подтвердилась ("No specific
  2026 RAW_REQUESTS rate limit increases... documented" на этой странице). Не записываю —
  либо источник поисковой выдачи ошибся, либо это на другой странице документации, которую
  я не проверил — **[не удалось проверить]**, а не факт.
- Конкретные цифры win-rate по сигнальным сервисам (Glassnode/Santiment/CryptoQuant) —
  поисковая выдача явно указала, что источники "don't contain specific win-rate percentages
  or detailed historical track records" — честно не нашлось, не выдумываю числа.

### Предложение дня
**Добавить явный расчёт expectancy (в R) в `/stats`/`/journal`** — поверх уже собранных
`actual_r` по закрытым сигналам, разбивка win-rate/avg-win-R/avg-loss-R отдельно (не только
общий `avg_r`, как сейчас). Трудозатраты: малые (1 функция в `signal_journal.py`, использует
уже существующие данные, не новый источник). Эффект: не измерен — узнаем полезность после
внедрения, по тому, меняет ли этот срез решения владельца о доверии к конкретным источникам
сигналов сильнее, чем уже показываемый `avg_r`.
