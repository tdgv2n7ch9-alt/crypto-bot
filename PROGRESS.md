# PROGRESS.md — crypto-bot

Журнал сессий и рабочие конвенции для этого проекта. Формат — по образцу другого проекта
пользователя: конвенции сверху (не меняются от сессии к сессии, если явно не пересмотрены),
журнал сессий снизу (хронологически, новые записи в конец).

## Конвенции

**Git и удалённый репозиторий**
- `origin` — `https://github.com/tdgv2n7ch9-alt/crypto-bot.git`, без токена в URL. Аутентификация —
  через `credential.helper osxkeychain`, токен вводится вручную при первом push в сессии, не хранится
  в конфиге репозитория.
- Прод обновляется через отдельный механизм — прямые коммиты в `origin/main` через GitHub
  Contents/Git Data API (см. `subscribers.py`, `signal_journal.py`), в обход обычного `git push` из
  рабочего каталога. Из-за этого локальная git-история и `origin/main` разошлись (см. `AUDIT.md`
  §2.1) — это известное, задокументированное состояние, не трогать без отдельного решения о
  реконсиляции (ROADMAP П1.1).
- Перед любой операцией, способной затронуть историю (`merge`, `rebase`, `reset`), — сначала
  `git status`, проверка незакоммиченных изменений, ничего не сбрасывать без явного запроса.

**"База данных" проекта**
- Реальной БД нет — роль БД играют JSON-файлы в приватном GitHub-репозитории через Contents API:
  `data/chat_ids.json` (подписчики), `journal/signals.json` (журнал сигналов и их исходов).
- **Бэкап перед миграцией**: перед любым изменением схемы этих JSON-файлов (переименование поля,
  смена структуры, удаление поля) — обязательно снять копию текущего файла (например, скачать
  текущее содержимое через Contents API и сохранить со штампом даты) до применения изменения.
  Сейчас такого шага в коде нет (ROADMAP П1.4) — при любой будущей миграции его нужно выполнять
  вручную, пока автоматический бэкап не появится.
- **Аддитивные миграции**: новые поля добавляются с дефолтным значением для старых записей, старые
  поля не удаляются и не переименовываются "на месте" — если поле реально не нужно, оно помечается
  как deprecated в комментарии и перестаёт писаться, но старые записи с ним не трогаются задним
  числом. Ломающие изменения формата — только с отдельным явным одобрением и с бэкапом по пункту
  выше.

**Границы работы (постоянные, до отмены)**
- Не трогать ключи API бирж (Bybit/Binance/CMC и т.п.) — ни читать значения, ни менять.
- Не менять торговую/сигнальную логику (пороги чек-листа, R:R-гейт, формулы TP/SL/entry, критерии
  stage1/stage2 в `signal_loop.py`/`fa_engine.py`) без явного одобрения по каждому изменению.
- Не рассылать ничего реальным подписчикам в процессе разработки/тестирования — тестовые прогоны
  только на "сухую" (без реального `send_message`) или в собственный чат владельца по явному
  запросу.
- `dr_linessa_bot` — отдельный проект, не трогать, не упоминать в изменениях этого репозитория.

### 2026-07-10 (продолжение) — П3 часть 1: DXY-источник + CMC retry/alert

**DXY — честный результат исследования, не то, что буквально просили.** Просили "заменить
неофициальный Yahoo-эндпоинт на стабильный источник". Проверил реально (не по памяти,
Протокол правды п.2), результат: **документированной бесплатной real-time замены для
именно ICE DXY не нашлось**:
- Twelve Data — символ `DXY` НЕ поддерживает: `symbol_search?symbol=DXY` вернул `DXYN`/`DXYZ`
  (не связанные акции), не индекс. Источник: [twelvedata.com/docs](https://twelvedata.com/docs),
  прямой запрос к `api.twelvedata.com/symbol_search`.
- Stooq (частая community-замена Yahoo) — сейчас закрыт JS proof-of-work анти-бот челленджем
  (`/__verify` с SHA-256 challenge), НЕ скриптуется голым `requests.get()`. Проверено прямым
  `curl` — HTTP 200, но тело ответа — JS-челлендж, не CSV.
- FRED `DTWEXBGS` — официальный, документированный, НО это **другой индекс** (26 валют
  trade-weighted vs 6 валют с фиксированными весами 1973г у ICE DXY) и суточные данные с
  лагом, не real-time замена. Источник:
  [fred.stlouisfed.org/series/DTWEXBGS](https://fred.stlouisfed.org/series/DTWEXBGS/).
- FMP (financialmodelingprep.com) — поддержка DXY/DX=F не подтверждена документацией,
  не тестировал с реальным ключом (нет ключа) — оставляю как "не проверено", не как "не
  работает".
- Существующий Yahoo-эндпоинт при этом РЕАЛЬНО РАБОТАЕТ прямо сейчас (проверено `curl`,
  вернул `regularMarketPrice: 100.937`, валидные данные). История v56-v70 хотфиксов была
  про баги **парсинга** в коде бота (generator/dict, multiline, indent), не обязательно
  про недоступность самого источника.
- **Решение**: не менял источник данных (нет доказанной лучшей альтернативы), вместо этого
  устранил вероятную РЕАЛЬНУЮ причину повторных багов — 4 почти-дублирующихся инлайн-блока
  парсинга (bot.py, было ~2496-2512, ~3708-3744, ~10057-10070) с bare `except: pass`,
  тихо оседающим в 0 без единого лога. Заменены на один `_fetch_yahoo_chart()` (retry на
  транзиентную ошибку, честный лог, статус в `_DATA_SOURCE_STATUS["yahoo_finance"]`).
  Формулы/пороги macro_score (`sc+=...`) НЕ ТРОНУТЫ — только надёжность фетча.
  **Открытый пункт для владельца**: если нужна именно платная официальная лицензия на DXY
  (Bloomberg/Refinitiv/ICE напрямую) — это отдельное решение с бюджетом, не техническая
  правка кода.

**CMC 401/429 — retry с оговоркой + алерт.** `_fetch_cmc_markets()` уже содержал явный
комментарий "не ретраим 429, чтобы не жечь платный кредит впустую" — уважаю это решение
исходного автора, НЕ добавил слепой retry на HTTP-коды ошибок (429/401), только на
транзиентные сетевые сбои (timeout/connection error, `_cmc_get()`, 1 retry/1с backoff) --

  "интермиттентных 401" из памяти проекта — не подтверждено, что это ИМЕННО причина
  (нет доступа к реальному Railway env var), но это дешёвая проверка на будущее.
- `_record_source_result()` — единая точка учёта с счётчиком подряд идущих отказов,

- `run_watchdog` расширен: >=3 отказов подряд по любому источнику -> алерт владельцу
