# PROGRESS.md — crypto-bot

Журнал сессий и рабочие конвенции для этого проекта. Формат — по образцу другого проекта
пользователя: конвенции сверху (не меняются от сессии к сессии, если явно не пересмотрены),
журнал сессий снизу (хронологически, новые записи в конец).

## Конвенции

**Git и удалённый репозиторий**
- `origin` — `https://github.com/tdgv2n7ch9-alt/crypto-bot.git`, без токена в URL. Аутентификация —
  через `credential.helper osxkeychain`, токен вводится вручную при первом push в сессии, не хранится
  в конфиге репозитория.
- Прод обновляется через отдельный механизм — прямые коммиты в `origin/main` через GitHub
  Contents/Git Data API (см. `subscribers.py`, `signal_journal.py`), в обход обычного `git push` из
  рабочего каталога. Из-за этого локальная git-история и `origin/main` разошлись (см. `AUDIT.md`
  §2.1) — это известное, задокументированное состояние, не трогать без отдельного решения о
  реконсиляции (ROADMAP П1.1).
- Перед любой операцией, способной затронуть историю (`merge`, `rebase`, `reset`), — сначала
  `git status`, проверка незакоммиченных изменений, ничего не сбрасывать без явного запроса.

**"База данных" проекта**
- Реальной БД нет — роль БД играют JSON-файлы в приватном GitHub-репозитории через Contents API:
  `data/chat_ids.json` (подписчики), `journal/signals.json` (журнал сигналов и их исходов).
- **Бэкап перед миграцией**: перед любым изменением схемы этих JSON-файлов (переименование поля,
  смена структуры, удаление поля) — обязательно снять копию текущего файла (например, скачать
  текущее содержимое через Contents API и сохранить со штампом даты) до применения изменения.
  Сейчас такого шага в коде нет (ROADMAP П1.4) — при любой будущей миграции его нужно выполнять
  вручную, пока автоматический бэкап не появится.
- **Аддитивные миграции**: новые поля добавляются с дефолтным значением для старых записей, старые
  поля не удаляются и не переименовываются "на месте" — если поле реально не нужно, оно помечается
  как deprecated в комментарии и перестаёт писаться, но старые записи с ним не трогаются задним
  числом. Ломающие изменения формата — только с отдельным явным одобрением и с бэкапом по пункту
  выше.

**Границы работы (постоянные, до отмены)**
- Не трогать ключи API бирж (Bybit/Binance/CMC и т.п.) — ни читать значения, ни менять.
- Не менять торговую/сигнальную логику (пороги чек-листа, R:R-гейт, формулы TP/SL/entry, критерии
  stage1/stage2 в `signal_loop.py`/`fa_engine.py`) без явного одобрения по каждому изменению.
- Не рассылать ничего реальным подписчикам в процессе разработки/тестирования — тестовые прогоны
  только на "сухую" (без реального `send_message`) или в собственный чат владельца по явному
  запросу.
- `dr_linessa_bot` — отдельный проект, не трогать, не упоминать в изменениях этого репозитория.

## 2026-07-11 (продолжение) -- 3 бага карточки Pump-Reversal (найдены владельцем на живой карточке EVAA)

### Баг 1: OI-рассинхрон (шапка +1.7%/5мин, РАЗБОР +0.0%/5мин в одном сообщении)

**Root cause найден и подтверждён на реальных данных**: `bot._get_oi_change(symbol)` --
функция с ПОБОЧНЫМ ЭФФЕКТОМ, мутирует `_OI_HISTORY[symbol]` при КАЖДОМ вызове.
`_compose_alert()` (шапка) и `_build_analysis_block()` (блок РАЗБОР) раньше независимо
вызывали `ctx.get_oi_change()` каждый сам по себе -- второй вызов через секунды после
первого видел "предыдущее" значение, только что записанное первым, отсюда near-zero %.
Third место (shadow-логирование в `_confirm_pump_reversal`) делало то же самое --
итого 3 независимых снапшота на одну карточку.

**Фикс**: один снапшот funding/OI/killzone на всю карточку. `_compose_alert()`/
`_build_analysis_block()` получили опциональный параметр `market_snapshot` -- если
передан, используют его вместо независимого фетча (обратная совместимость: ранние
стадии алерта, где РАЗБОР не строится, по-прежнему фетчат сами, без параметра).
`_confirm_pump_reversal()`/`_confirm_dump_reversal()` теперь фетчат ОДИН раз в начале
и передают снапшот везде, включая shadow-лог.

**Смоук на реальных данных** (не синтетика) -- воспроизвёл точный баг на живой
`bot._get_oi_change('BTC')`: посеял `_OI_HISTORY` состоянием "5 минут назад OI был на
1.7% ниже" (реалистичная имитация настоящего дрейфа), затем два вызова подряд:
первый дал `+1.73%`, второй (через мгновение) -- `+0.00%` -- **практически идентично
цифрам из отчёта владельца (+1.7% vs +0.0%)**. С фиксом (снапшот один раз) -- оба
места получают одинаковые `+1.73%`.

### Баг 2: EMA-стек "бычий" при цене ниже EMA20 и EMA50 (4h)

**Root cause**: `ta_extra._stack_label()` проверял ТОЛЬКО порядок EMA20>EMA50>
EMA100>EMA200, полностью игнорируя, где сейчас цена -- EMA лагают за ценой, порядок
может не "догнать" свежий разворот, отсюда ложный "бычий" вердикт на самом деле
развернувшейся цене. **Важно**: это НЕ просто текст карточки Памп-радара --
`ta_extra.ema_stack_score_delta()` читает "stack" и даёт +-8 к Rocket Score в
`fa_engine.py`/`bot.py` (real_full_analysis) -- фикс меняет реальный скоринг живых
сигналов по всему боту, не только карточку. Владелец одобрил явно ("почини
честность").

**Фикс**: `_stack_label(last, price)` теперь требует ДВУХ условий для "бычий"/
"медвежий" -- порядок EMA И цена ПОДТВЕРЖДАЕТ порядок (выше/ниже ОБЕИХ ближних
EMA20/EMA50). Расхождение порядка и цены -> "смешанный" (переходное состояние,
честно, не ложный вердикт). `price=None` (вызов без цены) -- откат к чистому
порядку EMA для обратной совместимости мест без цены под рукой.

**Смоук на реальных живых свечах** (BTC/ETH/SOL, только что): BTC/ETH -- цена выше
обеих EMA, порядок и цена согласны -> честно "бычий". **SOL прямо сейчас**: цена
$78.02 НИЖЕ EMA20 ($78.44) И EMA50 ($78.52) -- ровно та ситуация, что нашёл владелец
на EVAA -- новый код честно даёт "смешанный", не ложный "бычий"/"медвежий".
