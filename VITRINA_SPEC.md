## 1. КАНАЛ — 5 канонических типов постов

Аудит из #292 §4 честно показал: буквальный канон-шаблон (§2 того же
документа) применим только к ОДНОМУ классу сообщений (авто-точки входа);
остальные типы структурно другие. Итоговая таксономия — **5 типов**,
каждый со своим форматом, не насильно втиснутый в один шаблон:

### Тип 1 — Канон-карточка авто-сигнала (ТОП ЛОНГ/ШОРТ)

Реализация: `card_v2.assemble_card_v2_canon()` (`card_v2.py:322-357`) —
буквальный порядок из #292 §2: `[тикер·направление·фаза·PD] / риск-план
DCA-SL-TP / риск оператора+L/S (`format_operator_risk_block`,
`card_v2.py:246-256`) / чеклист N/6 / «Почему» одной строкой
(`format_why_line`, `card_v2.py:263-274`)`. Тесты — `tests/test_card_v2.py`
(добавлены этой ночью, см. пункт 3 ниже), grep-guard —
`tests/test_public_text_no_source_tokens.py::test_assemble_card_v2_canon_
clean`. **Готов к подключению** (только сборка, вызывающий код send_
scheduled() пока строит текст иначе — см. DoD раздел 4).

### Тип 2 — Лёгкий алерт zone-touch

Не полная карточка (природа события другая — триггер "цена коснулась
зоны", не новый сигнал). Формат: короткое уведомление + ссылка/reply на
уже опубликованную карточку. Сейчас (`bot.py:7947`) — ad-hoc текст
"Watchlist ALERT: SYM @ price", НЕ канон, требует лёгкой ревизии текста
(не структуры).

### Тип 3 — Лимитки (разметка автора)

Свой формат, НЕ канон (`bot.py:4417` `_limitki_execution_card_text`):
зона / лестница DCA 50/30/20 / SL за зоной / размер позиции / ликвидации
/ rug-строка. Разметка владельца применяется 1:1 — чеклист сюда
концептуально не подходит (не авто-фильтрованный сигнал). Уже в бою для
раздела «⭐ ЛИМИТКИ» — не путать с Типом 1.

### Тип 4 — Событие (Event-Radar)

Информационное, не торговая точка: листинг/делистинг/анлок/конференция.
`event_radar.py:386` `format_event_alert`. Без DCA/SL/чеклиста — другой
класс сообщения по определению.

### Тип 5 — Еженедельный дайджест трек-рекорда

Агрегированная статистика WR/PF по методологии из shadow-журнала — БЕЗ
упоминания источников (правило 0 ниже). Пока не реализован как отдельный
канальный пост — данные уже считаются (`shadow_outcome_analysis.py`),
не хватает только формата поста + расписания публикации. Естественный
кандидат на переиспользование `miniapp_api.py`-эндпоинта `/api/v1/track-
record` (см. раздел 3) как источника чисел для текста поста — одна и та
же логика подсчёта, два разных вывода (Mini App JSON / канал текст).

**Правило 0 (бессрочное, #292)**: ни в одном из 5 типов — упоминание
внутренних методологических источников (имена курсов/аналитиков/каналов).
Замены: «чеклист N/6», «фильтры», «авторская разметка», «методология
BEST TRADE». Grep-guard тест — `tests/test_public_text_no_source_tokens.py`.

## 3. MINI APP — что РЕАЛЬНО уже есть (важное расхождение с #292 2026-07-18)

**Бэкенд `miniapp_api.py` (428 строк) — 6 живых REST-эндпоинтов**,
подтверждено чтением кода (не докстрингом, реальными `app.router.add_get`,
`miniapp_api.py:401-406`):
- `/api/v1/health`
- `/api/v1/track-record` (`_make_track_record_handler`)
- `/api/v1/zones` (`_make_zones_handler`)
- `/api/v1/glossary` (`_make_glossary_handler`)
- `/api/v1/signals` (`_make_signals_handler`, источник — TOP_LONG_SIGNALS/
  TOP_SHORT_SIGNALS, `miniapp_api.py:268`)
- `/api/v1/dashboard` (`_make_dashboard_handler`, источник —
  `bot.get_btc_market_context()`, `miniapp_api.py:344`)

Докстринг модуля (`miniapp_api.py:17-18`) сам честно фиксирует объём:
"только /api/v1/track-record, /api/v1/zones, /api/v1/glossary +
/api/v1/health. dashboard/signals/portfolio" — но по факту `/api/v1/
signals` и `/api/v1/dashboard` УЖЕ реализованы (докстринг устарел
относительно кода, не наоборот — честно фиксирую расхождение, не
исправляю чужой докстринг в объёме этой ночи без явного "да").

**Визуальный прототип** — `knowledge/best_trade_radar_v2.html` (264
строки, утверждён владельцем 2026-07-18):
- **Конвейер PD-Score, 4 фазы**: 1·Зарождение (Vol Surge >5x, |ΔP|<3%,
  OI-импульс) → 2·Манипуляция (Wash-Score, Gini >0.98, возраст <90д) →
  3·Живой поток (CVD, Book Imbalance >70/30, стены) → 4·Дамп/шорт-вахта
  (CEX-inflow, дивергенция, откат −3-5%).
- **Живая таблица**: токен/фаза-чип/PD-score/%изменение/цена вход→сейчас/
  спарклайн/про-режим доп. колонки (объём, funding, Gini, риск-бейдж,
  возраст сигнала).
- **Макро-лента** (тикер BTC/BTC.D/USDT.D/ETH/S&P/DXY/VIX/ликвидации).
- **Честный таймштамп** — не голый "LIVE", реальное время последнего
  обновления + жёлтая точка при staleness >3 интервалов.
- **Протокол правды публично**: "N записей в журнале / M закрытых
  исходов / винрейт — калибруется, публикация при n≥20 на контур / 0
  сигналов задним числом" — прямое визуальное воплощение протокола
  правды этого проекта для подписчика, не только для владельца.
- Прототип — **статический HTML/CSS/JS-макет с фейковыми данными**
  (`tapeData` — захардкоженная строка, таблица — статичные `<tr>`), НЕ
  подключён к `miniapp_api.py` эндпоинтам. Подключение фейковых данных к
  реальным (`fetch()` к `/api/v1/dashboard`/`/api/v1/signals`) — не
  сделано, это САМ ОБЪЁМ реализации Mini App MVP, не мелочь.

**Технически** (#292 §1, без изменений): Telegram Mini App = веб-страница
внутри `WebApp.init()`, кнопка `InlineKeyboardButton(web_app=WebAppInfo(url=...))`.
Railway уже хостит `miniapp_api.py` (aiohttp) — раздача статики ЛИБО через
тот же aiohttp-инстанс (`app.router.add_static`, стандартный паттерн, не
проверено, добавлено ли), ЛИБО отдельный маленький сервис — решение не
принято, не гадаю.
