## Живая проверка полей (фон, 2026-07-22) — датасет и структура подтверждены

Источник из `INSIGHTS.md` (`publicreporting.cftc.gov`) — Socrata REST API.
Конкретный dataset id для "Traders in Financial Futures, Futures-only,
Combined" (TFF, тот же отчёт, где BTC под кодом `133741`):
**`https://publicreporting.cftc.gov/resource/gpe5-46if.json`** — проверено
живым запросом сегодня (без ключа, свободный доступ подтверждён):

```
GET https://publicreporting.cftc.gov/resource/gpe5-46if.json
    ?$where=cftc_contract_market_code='133741'
    &$limit=3&$order=report_date_as_yyyy_mm_dd DESC
```

Реальный ответ (последний отчёт на момент проверки — **2026-07-14**, "2026
Report Week 28" — т.е. лаг ~1 неделя от даты запроса 2026-07-22, совпадает с
задокументированным в `INSIGHTS.md` "еженедельно, по пятницам, данные за
предыдущий вторник"):

| Поле | Значение (пример, 2026-07-14) | Что это |
|---|---|---|
| `open_interest_all` | 19385 | Весь открытый интерес |
| `dealer_positions_long/short_all` | 6754 / 2138 | Дилеры/маркетмейкеры |
| `asset_mgr_positions_long/short` | 4779 / 1964 | Институциональные управляющие активами |
| `lev_money_positions_long/short` | 4015 / 11506 | **Leveraged Money** — хедж-фонды/CTA, ближайший аналог "спекулянтов" |
| `other_rept_positions_long/short` | 389 / 143 | Прочие отчитывающиеся |
| `nonrept_positions_long/short_all` | 871 / 1057 | Неотчитывающиеся (розница, косвенно) |
| `change_in_*` | — | Недельное изменение по каждой категории |
| `pct_of_oi_*` | — | Доля каждой категории от OI |

**Самый релевантный для потенциального сигнала срез — `lev_money_positions_
long/short`**: в этом снимке short (11506) почти втрое больше long (4015) —
leveraged-money заметно чистый шорт по контрактам. Это НЕ утверждение о
предсказательной силе (не проверялось), просто подтверждение, что поле
реально существует и реально меняется (не константа/заглушка).

## Возможная форма интеграции — по аналогии с уже существующим паттерном

Проект уже имеет прецедент для похожего источника: `oi_funding_ls_shadow`
(OI/funding/Long-Short ratio с CoinGecko/бирж, Фаза B Derivatives,
`shadow_engine.py`) — informational shadow-поле, прикрепляемое к сигналу на
момент генерации, НЕ влияющее на чек-лист/гейты, только для последующего
статистического анализа. CFTC COT логично встроить ТЕМ ЖЕ способом:

1. **Еженедельный фоновый job** (не по тику сигнала — данные обновляются раз
   в неделю, дёргать чаще бессмысленно и невежливо к бесплатному источнику):
   раз в сутки (например, вместе с утренним `morning_brief`) — проверка, не
   вышел ли новый отчёт (сравнить `report_date_as_yyyy_mm_dd` с последним
   сохранённым), если да — сохранить сырые поля + вычислить net leveraged-
   money (`long - short`) и `pct_of_oi_lev_money_long - pct_of_oi_lev_money_
   short` в `journal/cftc_cot_btc.json` (новый маленький файл, тот же
   паттерн, что `journal/watch_zones.json` и т.п.).
2. **Прикрепление к сигналу** (аналог `amd_phase_methodology`): при генерации
   каждого сигнала (`compute_shadow()`) добавить informational-поле
   `cftc_cot_snapshot` = последний известный снимок net leveraged-money +
   его "возраст" в днях (важно честно показывать, что данные почти всегда
   на 8-14 дней устарели относительно момента сигнала — недельная
   гранулярность СТРУКТУРНО не может быть свежее).
3. **НЕ per-signal индикатор в классическом смысле** — из-за недельной
   гранулярности этот источник ближе по природе к макро-контексту дня
   (как Fear&Greed/DXY/VIX, уже "живые" в Блоке 4), чем к per-setup фильтру
   вроде AMD-фазы. Более логичное первое применение — **строка в утренней
   сводке** ("Leveraged money BTC: net short/long, изменение за неделю"),
   не gate-условие. Второе, отложенное применение (после накопления
   нескольких недель истории) — shadow-корреляция "экстремальная net-
   позиция за N недель ↔ последующий разворот" (нужна отдельная
   ретро-валидация, тот же уровень строгости, что `AMD_FILTER_SHADOW.md`,
   ПОСЛЕ накопления данных, не на старте).

## Стоимость

- **Денежная — $0.** Публичный Socrata REST, без ключа/токена (см.
  `INSIGHTS.md`).
- **Rate-limit риск — практически нулевой и структурно изолированный.**
  1 запрос/сутки к домену, полностью отличному от всех текущих боевых
  провайдеров (`CoinGecko`/`QuickNode`/`Bybit`) — не пересекается ни с одним
  известным rate-limit из уже задокументированных инцидентов (#278/#285,
  правило "изоляция RPC/API впрок" в CLAUDE.md). Соблюдает и "не более
  ~50 запросов" порог из того же правила с огромным запасом.
- **Инженерная — низкая.** Один HTTP GET раз в сутки + маленький JSON-файл
  состояния + одна дополнительная строка в `morning_metrics.py`/`compute_
  shadow()`. Меньше работы, чем Whale Radar Вариант A.

## Честно: чего этот план НЕ даёт

1. **Никакой доказанной предсказательной силы для BTC-сигналов этого бота.**
   Институциональный "COT extreme = разворот" — известная эвристика в
   традиционных рынках (товары/валюты), но НЕ проверялась на этом проекте и
   НЕ проверялась специально для крипто-CME-фьючерсов — нужна отдельная
   shadow-валидация на накопленной истории (недели/месяцы), прежде чем
   рассматривать как фильтр, тем же протоколом, что и AMD-фильтр сегодня.
2. **Недельная гранулярность — структурный потолок**, не временное
   ограничение источника. Даже при идеальной интеграции сигнал не может
   быть "свежее" пятничного релиза — не подходит для внутридневных
   решений, только макро-фон.
3. Библиотека `cot_reports` (github.com/NDelventhal/cot_reports,
   зафиксирована в `INSIGHTS.md`) не проверялась в работе — решение
   "использовать её или парсить Socrata JSON напрямую" (как в живой
   проверке выше) отложено, JSON-подход выглядит проще (нет нового
   зависимости, уже используем `requests`).

## Рекомендация (оценка, не решение)

Низкая инженерная цена ($0, ~1 доп. HTTP-вызов/сутки, отдельный от боевых
провайдеров) делает добавление СТРОКИ В УТРЕННЮЮ СВОДКУ дешёвым и
безопасным первым шагом (класс, близкий к уже "живым" DXY/VIX/Fear&Greed).
Использование как gate/фильтр сигналов — преждевременно без отдельной
shadow-валидации на накопленной истории, приоритет ниже, чем уже идущие
контуры (AMD-фильтр — Эксперимент B, tz13/patch09 накопление).
