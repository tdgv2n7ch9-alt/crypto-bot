## 1. КРИТИЧНО — три мёртвые кнопки, общий root cause найден живьём

**Владелец, живая приёмка с телефона**: РЫНОК (главное меню), РАДАРЫ → Зоны,
МОИ → «Мои разметки (/zones)» не отвечают при тапе. Памп-радар и Rug-Radar
отвечают нормально (роутинг жив).

**Диагноз (read-only, `railway logs --lines 1500`, живые traceback'и, не
догадка)**:

- **Общий структурный факт**: в `bot.py` НЕ зарегистрирован `app.add_error_
  handler(...)` — ни одного вызова во всём файле (проверено grep'ом). Любое
  необработанное исключение внутри ЛЮБОГО callback-хендлера ловится
  python-telegram-bot'ом по умолчанию, логируется одной строкой `[ERROR] No
  error handlers are registered, logging exception.` + traceback — и просто
  исчезает с точки зрения владельца: `q.answer()` уже отработал (строка 5588,
  вызывается БЕЗУСЛОВНО до диспетчеризации), спиннер гаснет, сообщение не
  меняется. Внешне выглядит как «кнопка не отвечает», хотя код реально упал.
  Это и есть класс бага, который подозревал владелец («except:pass») — только
  на уровне отсутствующего error_handler'а Framework'а, а не буквального
  `except: pass` в нашем коде.

- **РЫНОК** (`callback_data="mv2_rynok"`, bot.py:3324) — `_mv2_callback_router`
  (bot.py:5440) зовёт `_mv2_render_rynok(q)`. Живой traceback:
  ```
  File "/app/bot.py", line 5114, in _mv2_render_rynok
      await q.edit_message_text(text, parse_mode="Markdown", ...)
  telegram.error.BadRequest: Can't parse entities: can't find end of the
  entity starting at byte offset 965
  ```
  Тот же класс бага, что уже чинили this session в `event_radar.py`/
  `morning_metrics.py` (см. `sanitize_markdown_free_text()`/
  `_sanitize_free_text()`, `feedback_deploy_workflow.md`) — непарный
  `_`/`*`/`` ` ``/`[`/`]` в свободном тексте, интерполированном в карточку.
  Источник текста в `_mv2_render_rynok()` (bot.py:5080-5114) —
  `get_btc_market_context()` (label/warning-поля) и
  `onchain_metrics.format_onchain_card_text("BTC")` — оба уже сами обёрнуты в
  try/except с fallback на «н/д», НО это защищает только от исключений при
  ПОЛУЧЕНИИ данных, не от непарного спецсимвола ВНУТРИ уже полученного текста
  при рендере в Markdown.

- **Зоны / Мои разметки** — ОБА идут через один и тот же `cmd_zones()`
  (bot.py:13902), просто с разным `back_to`:
  ```
  File "/app/bot.py", line 5459, in _mv2_callback_router
      await cmd_zones(_mv2_fake_update(q), ctx, back_to="mv2_radary")
  File "/app/bot.py", line 13902, in cmd_zones
      await update.message.reply_text(text, parse_mode="Markdown", ...)

  entity starting at byte offset 965/966
  ```

  Общий root cause для ЭТИХ ДВУХ кнопок подтверждён (одна и та же функция).
  Вероятный источник спецсимвола — `note`-поле зоны в `journal/watch_zones.
  json` (свободный текст владельца/аналитика, ничем не экранируется перед
  Markdown-рендером) — **требует проверки**, не проверено в этом инкременте
  (не подтверждено, что это именно `note`, а не другое поле — честно, не
  гадаю дальше без чтения `cmd_zones()` целиком). Связь с ночной правкой
  `level_watch.py` («развёрнутые границы зон») — НЕ подтверждена и НЕ
  опровергнута: сам traceback падает на этапе Markdown-рендера, не на этапе
  вычисления границ, так что это может быть совпадением по времени, а не
  причинно-следственной связью. Требует отдельной проверки перед фиксом.

**Рекомендованный фикс (не применён, только план)**:
1. Системный: зарегистрировать `app.add_error_handler(...)` в `bot.py` —

   САМ КЛАСС «тихой мёртвой кнопки» на будущее — любая новая аналогичная
   ошибка перестанет быть невидимой.
2. Точечный: применить уже существующий `sanitize_markdown_free_text()`/

   (btc context label/warning, onchain text) и в `cmd_zones()` (после
   подтверждения точного поля-источника).
3. Регресс-тест «каждая кнопка меню отвечает» — по запросу владельца (п. а
   его сообщения) — сквозной тест callback-роутинга: для каждого
   `callback_data`, реально генерируемого меню/подменю, есть живой хендлер и

## 2. Дедупликация кнопок ТОЧКИ/Исполнение

Владелец: BTC LONG ×3, BTC SHORT ×3, AKE SHORT ×2, AVAX LONG ×2 — дубли в
разделе «Монеты в работе» / Исполнение. Новый формат кнопки:
«СИМВОЛ НАПРАВЛЕНИЕ · вход X · id», группировка по символу при >1 записи.

**Диагноз частично проведён**: `_mv2_tochki_collect_items()` (bot.py:4823)
читает из `TOP_LONG_SIGNALS`/`TOP_SHORT_SIGNALS`/`SPOT_PORTFOLIO` — простые
dict'ы, ключ = символ, физически не могут содержать дубль одного символа.
Значит источник «×3» — НЕ этот сборщик. Не дочитано: сама функция построения
кнопок/InlineKeyboardButton-рядов в `_mv2_render_tochki()` (bot.py:4914+, файл
дочитан только до этой строки) — вероятно, дубли приходят из другого места
(возможно, «Исполнение» — ОТДЕЛЬНЫЙ экран от «ТОЧКИ», читающий
`journal/signals.json`, где у одного символа+направления РЕАЛЬНО может быть
несколько разных `id`-записей — само наличие «id» в желаемом новом формате
это подтверждает). Нужно дочитать `_mv2_render_tochki()` целиком + найти
экран «Исполнение», прежде чем проектировать фикс.
