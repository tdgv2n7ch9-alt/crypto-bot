# RUG_WATCHLIST.md — массовый rug-скоринг (Пакет 11 Ф3)

**Дата**: 2026-07-13, ~00:35-00:42 UTC. **Скрипт**: `rug_mass_scan.py` (этот
пакет), запущен через `railway run python3 -u rug_mass_scan.py` — реальные
`ETHERSCAN_API_KEY`/`GITHUB_TOKEN` из прод-окружения.

## Честно: задача не выполнена в полном объёме — CoinGecko free-tier rate limit

Запрошено: топ-300 CoinGecko через `rug_radar.compute_rug_risk()` с живым
Etherscan. Получено по факту:

- **Список монет**: только **240**, не 300 — сам `bot.get_all_coins()` словил
  `429 Too Many Requests` на странице 2 CoinGecko `/coins/markets` и честно
  вернул то, что успело загрузиться (240) вместо падения.
- **Детальные данные (`cg_detail`, нужны для fdv_mcap/age_listing/
  exchange_transfers детекторов)**: получены только для **8 из 240** монет
  (3.3%). Остальные **232/240 — `429 Too Many Requests`** на
  `/coins/{id}` уже с ПЕРВОЙ монеты (BTC), не постепенно — то есть лимит был
  исчерпан ещё до старта этого скрипта (вероятно, предыдущими вызовами
  `get_all_coins()` в этом же процессе + общим IP-лимитом Railway).
- **Etherscan вызовы**: всего **7** за весь прогон (бюджет 30000 не выбран
  даже на 0.03%) — Etherscan НЕ был узким местом, узкое место —
  исключительно CoinGecko `/coins/{id}`.
- **Время**: 422 секунды на 240 монет (~1.76с/монету), упёрлось в throttle
  `bot._cg_get()`'s внутренний rate-limit, а не в сам объём работы.

**Вывод по инфраструктуре (для следующего пакета)**: `bot._cg_get()`'s
текущий throttle рассчитан на единичные/редкие вызовы `_get_cg_detail()`
(x100-сканер, pump_detector — по 1 кандидату за раз), НЕ на последовательный
проход 300 монет подряд. Массовый rug-скан в текущем виде **не масштабируется**
без: (а) намного более мягкого троттлинга (секунды, не доли секунды между
вызовами `/coins/{id}` — резко увеличит время прогона), либо (б) платного
CoinGecko-тира, либо (в) кэширования/переиспользования уже полученных detail-
ответов между запусками. Это НЕ починено в рамках этого пункта — честно
зафиксировано как находка для владельца/следующего пакета.

## Результат: НЕ строю "топ-20 самых красных" — это было бы нечестно

С detail-данными только у 8/240 монет, 238/240 монет показали **score=0** —
но это означает **"нет данных для оценки"**, НЕ "монета безопасна". Строить
рейтинг "топ-20 красных" на этой выборке означало бы либо показать список,
доминированный отсутствием данных (не риском), либо тихо подменить
"неизвестно" на "ноль" — оба варианта вводят в заблуждение. Вместо этого —
честно показываю то немногое, что реально посчиталось.

## Остальные 232/240 монет

Все — `cg_detail` недоступен (`429`), фактически оценены только по
`vertical_growth` (единственный детектор, не требующий `cg_detail` —
использует `percent_change_30d`/`volume_24h`/`market_cap` из базового
CoinGecko markets-ответа) — и ни у одной эта проверка не сработала (score=0
у всех 232). Полный список символов и сырые risk-объекты — в
`rug_scan_dump.json` (в этом worktree, не закоммичен как часть отчёта — это
промежуточный дамп для отладки, не финальный артефакт).

## Честная рекомендация владельцу

1. Данные этого прогона **недостаточны** для содержательного вывода "какие
   токены сейчас самые рискованные по rug-скору" — задача Ф3 требует либо
   больше времени (медленный троттлинг), либо платного CoinGecko-доступа,
   либо переработки на batch-эндпоинты CoinGecko, где это возможно.
2. Etherscan-часть системы работает штатно (7 вызовов без ошибок, троттлинг
   не был проблемой) — бутылочное горлышко чисто на стороне CoinGecko detail.
3. Предлагаю на будущее: либо (а) ограничить массовый скан меньшим числом
   кандидатов (топ-30-50, уже отфильтрованных по другому критерию, как это
   уже происходит в x100/pump_detector), либо (б) намеренно растянуть прогон
   на много минут/часов с большими паузами между `/coins/{id}`-вызовами.
