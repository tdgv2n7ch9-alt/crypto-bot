# Whale Radar — расширение покрытия (владелец, 2026-07-22, ДИЗАЙН, НЕ внедрено)

Продолжение `WHALE_KLVL_DIAGNOSIS.md`: `whale_klvl_confluence` мёртв не
из-за бага в коде, а из-за структурного разрыва покрытия -- Whale Radar
отслеживает ~50 символов (топ по 24ч-обороту Bybit + BTC/ETH/SOL), а наши
сигналы идут по 412 разным символам за всю историю. Здесь -- проработка
"сколько символов реально нужно, откуда данные, какая стоимость" для
РЕАЛЬНОГО фикса (расширение покрытия), НЕ багфикса. **Только план +
оценка, ничего не внедрено, код не менялся.**

## Сколько символов реально нужно (честные числа из журнала)

Локальный подсчёт по `journal/signals.json` (все 477 записей, вся история):

| Срез | Различных символов |
|---|---|
| Любой источник (`full_analysis`/`signal_loop`/AUTO/X100) | **140** |
| Только наш AUTO-контур (`TOP_LONG_AUTO`+`TOP_SHORT_AUTO`) | **93** |
| Сейчас отслеживает Whale Radar (`TOP_N_SYMBOLS`) | **50** (+ BTC/ETH/SOL явно) |

Текущее покрытие (~50) закрывает МЕНЬШЕ ПОЛОВИНЫ (35.7%) даже узкого
AUTO-среза (93), не говоря о полном наборе (140). Символы из AUTO-списка
явно смещены к мелко/среднекапам (`BANANAS31`, `PIEVERSE`, `SNDKB`,
`SPCXB`, `USELESS`, `ZBCN`, `VSN`, `KITE`, `TAC` и т.п.) -- систематически
НЕ входят в топ-50 по обороту, это и есть суть проблемы (бот целенаправленно
ищет волатильные движения на менее ликвидных монетах, а Whale Radar
настроен на самые ликвидные).

## Откуда данные -- ограничения источника (проверено по официальной документации)

Источник -- ТОТ ЖЕ, что уже используется (`wss://stream.bybit.com/v5/
public/linear`, публичный WebSocket, без ключа). Проверены официальные
лимиты Bybit V5 WS (`https://bybit-exchange.github.io/docs/v5/ws/connect`,
прочитано 2026-07-22):

- **Для Futures/Linear (наш случай) -- явного лимита на число `args` в
  одном subscribe-запросе НЕТ** (лимит в 10 аргументов документирован
  ТОЛЬКО для Spot, у нас linear/perpetual).
- **НО есть общий лимит на ОДНО публичное соединение: суммарная длина
  массива `args` не может превышать 21 000 символов**, независимо от
  типа актива.
- IP-ограничение: не более 500 НОВЫХ соединений за 5 минут (не относится
  к расширению числа символов в уже открытом соединении).

**Оценка ёмкости одного соединения**: наш формат топика --
`orderbook.200.{SYMBOL}` (~22 символа) + `publicTrade.{SYMBOL}` (~20
символов) = **2 топика на символ**, в среднем ~21 символ/топик. При лимите
21 000 символов -- **до ~500 символов (1000 топиков) в ОДНОМ соединении**,
без необходимости шардировать по нескольким WS-подключениям. И 140
(полное покрытие), и 93 (только AUTO) укладываются с большим запасом (28%
и 19% от потолка соответственно) -- даже покрытие ВСЕХ ~500 монет, которые
сканирует AUTO-цикл (превышающее максимум-в-истории 140 сигнальных с
хорошим запасом на новые), теоретически влезло бы в один коннект, но уже
впритык к границе -- не рекомендую как первый шаг (см. ниже).

## Стоимость

- **Денежная -- $0.** Публичный WS Bybit, без ключа, без лимитов по
  тарифному плану (в отличие от QuickNode/Etherscan-класса проблем в
  других частях проекта). Расширение покрытия -- чисто вопрос
  архитектуры/кода, не бюджета.
- **CPU/память -- линейный рост, НЕ нагрузочно тестировалось.** Скан-цикл
  (`_scan_loop`, `WHALE_SCAN_INTERVAL_SEC=5`) уже итерируется по списку
  символов каждые 5с -- переход с 50 на 140-150 символов утраивает число
  итераций скана и пропорционально увеличивает объём входящих
  WS-сообщений (orderbook-дельты + printы сделок на новые символы).
  Честно: нет данных о реальном CPU-профиле на Railway при таком росте --
  не измерено, не выдумываю цифру. Разумная гипотеза (не факт): при 3x
  росте числа символов рост нагрузки ЭТОЙ конкретной задачи тоже около
  3x, но абсолютная база сейчас низкая (проверить после внедрения, не до).
- **Инженерная -- зависит от варианта (ниже).**

## Три варианта (без выбора, владельцу)

### Вариант A: статическое увеличение `TOP_N_SYMBOLS`

Поднять `TOP_N_SYMBOLS` с 50 до ~150-200 (с запасом над историческими
140). Простейшее изменение -- одна константа + добавление явного списка
самых частых AUTO-символов, которые НЕ входят даже в топ-200 по обороту
(доля мелких малоликвидных монет, которые Bybit ticker-эндпоинт мог бы не
отдать в топ-N вообще, если совсем низкий оборот -- нужно ОТДЕЛЬНО
проверить, входят ли ВСЕ 93 AUTO-символа в топ-200 Bybit-тикеров вообще,
это НЕ проверялось в рамках этого дизайна, честно "не проверено").
**Минус**: статичный список, который снова устареет по мере смены рыночного
фокуса (та же болезнь, что и сейчас, просто с большим числом символов --
паллиатив, не структурное решение).

### Вариант B: периодическое обновление списка (SYMBOL_REFRESH_SEC уже существует, не используется)

`whale_radar.py` УЖЕ ИМЕЕТ константу `SYMBOL_REFRESH_SEC = 6 * 3600`, но
она физически не подключена -- `bot.py`-докстринг честно фиксирует
"список топ-N в этом процессе больше не пересобирается после старта".
Дизайн: раз в `SYMBOL_REFRESH_SEC` -- пересобрать `symbols` как
ОБЪЕДИНЕНИЕ (a) топ-N по обороту (как сейчас) И (b) множества символов,
по которым shadow-контур писал записи за последние N дней (уже доступно
из `journal/shadow_signals.json`, не нужен новый источник данных) --
самонастраивающийся список, который следует за реальным фокусом сигналов,
не только за объёмом торгов. **Требует**: (1) логику пересборки списка +
(2) логику ПЕРЕПОДПИСКИ на новый набор топиков без потери уже
накопленного `whale_levels`/`lifetimes` состояния для символов, которые
остаются в списке (не пересоздавать книгу с нуля для уже отслеживаемых
символов -- иначе каждый цикл обновления обнулял бы возраст уже
накопленных зон). Умеренная сложность -- решает КОРНЕВУЮ причину
(статичность списка), не только временно увеличивает число символов.

### Вариант C: полное покрытие 500 AUTO-сканируемых монет

Совпадение с полным пулом кандидатов AUTO-цикла (не только уже
сигнализированные 93/140, а ВСЕ ~500, которые сканирует `send_scheduled()`
до фильтрации). Даёт максимальное теоретическое покрытие, но упирается в
лимит одного соединения (see выше, впритык к 21000 символов) -- потребовал
бы либо шардирования на 2+ WS-соединения (усложняет реконнект-логику,
слияние состояния между соединениями), либо смириться с риском
превышения лимита при росте цен символов длиннее среднего (тикеры вроде
`1000BONKUSDT`/`1000000MOGUSDT` длиннее обычных -- не учтено в примерном
расчёте выше). Самый затратный по инженерству вариант, отдача
(дополнительное покрытие сверх 140) под вопросом -- большинство ИЗ 500
кандидатов НИКОГДА не проходят гейты и не становятся реальными сигналами.

## Рекомендация (оценка, не решение)

Вариант B несёт больше инженерной работы, чем A, но решает корневую
причину (статичность), а не просто отодвигает её на бОльшее число
символов. Вариант C -- избыточен относительно реальной пользы (покрывать
монеты, которые никогда не сигналят, ради теоретической полноты).
**Решение -- за владельцем**, ничего не внедрено.

## Что НЕ проверено в рамках этого дизайна (честно)

1. Входят ли ВСЕ 93 AUTO-символа в топ-200 Bybit-тикеров по обороту, или
   часть настолько малоликвидна, что не появится там вообще (это отдельно
   определяло бы, работает ли вариант A/B "из коробки" или нужен
   explicit-список символов вне топ-N).
2. Реальный CPU/bandwidth-профиль Railway-контейнера при 3x росте числа
   отслеживаемых символов -- гипотеза, не измерение.
3. Длина тикеров с числовыми префиксами (`1000BONKUSDT` и т.п.) в расчёте
   ёмкости соединения -- используется среднее, не худший случай.

**Ничего не внедрено. Код не менялся. Live не тронут. `PAUSE_LIVE_SIGNAL_
EMISSION` без изменений (`true`).**
