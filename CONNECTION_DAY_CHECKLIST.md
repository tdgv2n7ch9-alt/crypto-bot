# CONNECTION_DAY_CHECKLIST.md — «День подключения»: пошаговый план активации при PF>1 (владелец, ночная очередь 2026-07-22/23, DOCS-ONLY)

Цель: когда критерий промоушена (n≥20 закрытых уникальных исходов, PF>1 —
см. `NEXT_PACKAGE.md` "Промоушен shadow-контуров... критерий") реально
выполнится для контура, весь путь от "цифры сошлись" до "витрина живёт у
подписчиков" занимает **день, не неделю** — потому что порядок и
верификация каждого шага прописаны заранее, не изобретаются в моменте.
**Ничего в этом документе не выполнено сейчас — это план на будущий день,
не инструкция к немедленному действию.**

## Точка входа: что именно триггерит этот чек-лист

Один из shadow-контуров (`tz13`/`patch05_bpr`/`patch09` — см. текущие
цифры в `PROGRESS.md`/`SHADOW_ANALYSIS.md`) достигает **n≥20 закрытых
уникальных исходов (дедуп по `journal_id`) И PF>1**. Это ФАКТ, который
сессия обязана зафиксировать в `PROGRESS.md`/утреннем брифе, но **решение
о промоушене — исключительно владелец** (см. CLAUDE.md "Точки владельца").
Этот чек-лист начинается ПОСЛЕ явного "да" на промоушен, не раньше.

## Порядок шагов (строго последовательно, каждый — свой DoD)

### Шаг 0 — Подтверждение критерия (до "да", подготовка факта)

- [ ] Пересчёт PF/WR/n на СВЕЖИХ данных прямо перед решением (не
  вчерашняя цифра из брифа) — `shadow_outcome_analysis.py` /
  `morning_metrics.py`, тот же метод, что уже используется для отчётов.
- [ ] Явно указать: КАКОЙ контур (`tz13`/`patch05_bpr`/`patch09`/AMD-
  эксперимент), n закрытых, PF, WR, за какой период.
- [ ] Показать владельцу цифры + источник (сам расчёт, не пересказ).

**Владелец говорит "да" на промоушен ИМЕННО ЭТОГО контура — точка
невозврата плана ниже, начинается Шаг 1.**

### Шаг 1 — Канон-карточка: подключение к send_scheduled()

- [ ] Источник данных для `operator_risk_lines`: уже считается для Rocket

- [ ] **grep-guard тест ОБЯЗАН быть зелёным ДО этого шага, не после**:

### Шаг 2 — Флаги эмиссии (порядок важен — сначала снять паузу узко, не широко)

- [ ] `PAUSE_LIVE_SIGNAL_EMISSION` (`bot.py:3214`, default `true`) —
  **НЕ трогать глобально**, если промоутится только ОДИН контур (тот же

  ENABLED`, `bot.py:3228`, снят узко для AUTO-контура без затрагивания
  остальных). Для промоушена tz13/patch05_bpr/patch09 — завести
  АНАЛОГИЧНЫЙ узкий флаг для конкретного контура, не расширять
  существующий `AUTO_LIVE_EMISSION_ENABLED`, если контур не AUTO.
- [ ] Предохранители по образцу Эксперимента B (`bot.py` секция после
  `AUTO_LIVE_EMISSION_ENABLED`): лимит одновременных позиций,
  kill-switch (≥4 из первых 5 закрытых в минус → авто-стоп + owner-алерт),
  риск-пометка на карточке. Реализовать ТЕ ЖЕ функции для нового
  контура, не изобретать заново — `_auto_concurrent_limit_reached()`/
  `_auto_emission_kill_switch_triggered()` как шаблон.
- [ ] Owner-алерт на каждую live-сделку нового контура (тот же паттерн,
  что уже есть для AUTO-эксперимента).

### Шаг 3 — Канал (CHANNEL_ID)

- [ ] `CHANNEL_ID` (`bot.py:7114`) уже используется для whale-алертов —

- [ ] **Проверка ПЕРЕД первой публикацией**: тестовый пост в канал

  по тексту в логе) — тот же принцип "живая верификация", что и для
  деплоя кода.

### Шаг 4 — Deploy

- [ ] Только `tools/deploy.sh` (см. CLAUDE.md "Верификация деплоя") —

### Шаг 5 — Живая приёмка первых сделок

  остановиться и разобраться, не повод его выключить.

## Что НЕ входит в "день подключения" (честно, отдельные, более долгие задачи)

- **Mini App подключение к реальным данным** (`VITRINA_SPEC.md` §3,
  `best_trade_radar_v2.html` → живой `fetch()` к `miniapp_api.py`) — НЕ
  блокирует день подключения нового сигнального контура, отдельная
  параллельная задача, свой график.
- **Настройки фаз-подписки/меню-аудит** (`VITRINA_SPEC.md` §2, открытые
  пункты) — не блокируют публикацию карточек, независимая доработка.
- **Тип 5 (еженедельный дайджест трек-рекорда)** — не требуется для
  первого дня публикации точек входа, отдельный формат/расписание.
