# TEXT_AUDIT.md — каталог текстов бота + аудит читаемости (Пакет 20, Этап 1)

ПЛАН-24, Блок А (владелец). **Этап 1 — инвентаризация, без правок кода**
(правки — только "быстрые победы", Этап 3, отдельными коммитами, если
найдутся). Ссылки — file:line по состоянию коммита `0706244`.

## Честно про охват этого прохода

Полный каталог "буквально каждого текста" в проекте такого размера (13к+
строк `bot.py` + десяток модулей) за одну ночь без потери качества
проверки — нереалистичная цель. Этот документ покрывает **всё, что
владелец явно перечислил** в поручении, с разной глубиной:
- **Полная 10-пунктовая оценка** — 7 типов сигналов/алертов (уже
  детально прочитаны в П3, здесь формализованы по чек-листу).
- **Структурная оценка** (шапка/действие/термины/"н/д", без построчного
  подсчёта длины) — экраны Меню v2 (главный + 5 разделов) и ключевые
  служебные тексты (`/start`, "нет данных").
- **НЕ покрыто в этом проходе** (честно, не выдаю за готово): `/zones`,
  `/precision`, `/coin`, полный текст `/full` (13-блочная карточка
  `fa_engine.render_full_analysis_card`), все варианты текста ошибок
  построчно. Список — в конце документа, для Этапа 1 продолжения.

## Чек-лист (владелец, дословно, для каждого текста)

1. Шапка: тип+символ+сторона за 1 секунду? 2. Структура: `━━━`, ≤4
строк/блок? 3. Строка ≤36 символов? 4. Эмодзи-маркеры едины (📍💰📊🎯⚠️🛑⏰)?
5. Числа: разряды/знаки корректны? 6. Термины: действие или словарь-в-
строке? 7. Нет пустых значений — только "н/д"? 8. Понятно действие?
9. ≤12 строк в ленте + "Развернуть"? 10. Ритм заголовок→данные→вывод→действие?

## Часть A — 7 типов сигналов/алертов (полная оценка)

Базовые находки по шапке/разделителю/длине — из `docs/SIGNAL_VISUAL_STANDARD.md`
(П3), здесь расширены оставшимися пунктами чек-листа.

| # | Пункт | Оценка |
|---|---|---|
| 1 | Шапка | ЧАСТИЧНО — символ+сторона есть, эмодзи РАЗДЕЛА нет (`bot.py:10437`) |
| 2 | Структура ━━━ | НЕТ — `SEP="➖➖➖➖➖➖➖➖➖➖"` (`bot.py:10418`), не канон |
| 3 | ≤36 симв/строка | НЕ ПРОВЕРЕНО построчно (нет автотеста длины, в отличие от `card_v2.check_signal_line_lengths()`) |
| 4 | Эмодзи-маркеры | ЕСТЬ (📍/📊/📈 присутствуют, `bot.py:10444-10446`) |

### A.2 Памп/дамп — `pump_detector._compose_alert()`, `pump_detector.py:1051+`

Ближе всех к стандарту среди "богатых" карточек (SEP канонический,
маркеры 📍/📊/⏰/⚠️ есть, `pump_detector.py:1138-1152`). Термины (funding/OI/
Z-Score) — БЕЗ словаря-в-строке. "н/д" — честно используется явно
(`"н/д (первый сигнал)"` паттерн уже проверен в этой сессии на других
местах). >12 строк, без "Развернуть". Действие — есть (сценарные строки
`_scenario_lines`).

### A.3 Whale — `bot._format_whale_alert()`, `bot.py:6047+`

SEP канонический (`bot.py:6096`). Числа честно форматированы (`fp()`,
проценты со знаком). Термины (Funding/OI/L/S Ratio) — без словаря-в-
строке, тот же пробел, что везде. "Факторы" блок — переменная длина,
>4 строк на блок нарушает п.2 при большом числе факторов. >12 строк
гарантированно (карточка доходит минимум до "РАСШИФРОВКА СКОРА").

### A.4 Supertrend — `bot.check_supertrend_signals()`, `bot.py:~6482-6512`

Разделителей `━━━` НЕТ СОВСЕМ — единственный тип без блочной структуры
вообще. Шапка без символа/стороны на первой строке (п.1 — НЕТ). "н/д
(первый сигнал)" — честная находка Пакета 18/19 уже применена
(`bot.py:...` "Прошлый сигнал: н/д"). ~8-10 строк — ближе всех к лимиту 12
естественно, без урезания. Действие явное ("рассмотри лонг/шорт").

### A.5 x100 — `bot.py:10661`

Структурно НЕ карточка сигнала (мульти-монетный список), см. П3 — чек-лист
"шапка {SYMBOL} {SIDE}"/"≤12 строк на сигнал" к нему неприменим по
конструкции. Нужен отдельный чек-лист "дайджест-списка" — не в этом
документе.

### A.6 Event-Radar — `event_radar.format_event_alert()`, `event_radar.py:386+`

Лучший результат по чек-листу: SEP = `card_v2.SEP` напрямую (`event_radar.py:402`),
5-6 строк (естественно ≤12), явная строка "В портфеле нет"/"Есть в твоём
портфеле" — НЕТ голых пустых значений. Тикер честно "н/д (тикер не
подтверждён)", если не распознан (`event_radar.py:393`) — образцовое
соблюдение п.7. Единственный минус: нет SIDE в принципе (события не имеют
направления сделки — не дефект, а честная граница типа).

### A.7 Zone-touch — `bot.check_watchlist()`, `bot.py:7079-7101`

Author-ветка ближе к стандарту (символ+сторона в шапке). Обычная ветка —
шапка `"🎯 Цена вошла в зону!"` без символа/стороны (п.1 — НЕТ). Разделителей
нет в обеих ветках. 8-9 строк — в пределах 12 естественно. `📝 {note}` —
если `note` пустая строка в разметке, текст покажет "📝 " с пустым
хвостом — потенциальный пункт 7 (пустое значение вместо "н/д"), НЕ
проверено на живых данных в этом проходе, честно помечаю как находку,
требующую живой проверки.

## Часть C — служебные тексты (структурная оценка)

- **`/start`** (MENU_V2_ENABLED=true путь) — `welcome_text_v2()`, уже
  покрыт в Части B. **Найдена честная находка**: LEGACY-ветка `/start`
  (`bot.py:3205-3224`, срабатывает только при `MENU_V2_ENABLED=false`) —
  мёртвый код в проде сейчас (флаг true), но содержит УСТАРЕВШИЙ и
  неточный текст источников данных ("On-chain: Lookonchain", "CMC
  Топ-500 монет" — реально бот использует CoinGecko первично, тот же факт
  уже честно задокументирован в докстринге `welcome_text_v2()` самим
  проектом, `bot.py:2834-2836`, но LEGACY-текст никогда не обновлялся
  вслед). Не риск сейчас (недостижимо), но кандидат почистить/удалить,
  если владелец подтвердит, что откат на LEGACY меню никогда не
  планируется.
- **"Нет данных" паттерны** — НЕ единообразны по всему проекту: варианты
  "н/д", "— (нет данных)", "нет активных сигналов", "Активных зон автора
  нет" все несут одну и ту же идею честного отсутствия данных, но разными
  словами в разных местах. Не ошибка сама по себе (каждый текст по
  контексту понятен), но не единый словарь — кандидат в "быстрые победы"
  ТОЛЬКО если владелец захочет строгую унификацию терминологии "нет
  данных" по всему боту.
