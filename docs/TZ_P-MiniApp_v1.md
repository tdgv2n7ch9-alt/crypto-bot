# ТЗ П-MiniApp v1.0 — Telegram Mini App «BEST TRADE Terminal»

Статус: готово к запуску по «да» владельца. Не блокирует Фазу B. Принцип: Mini App —
витрина. Мозг остаётся в боте. Ни одна логика не дублируется.

## 0. ЦЕЛИ И ЖЁСТКИЕ ОГРАНИЧЕНИЯ

**Цель:** интерфейс уровня терминала внутри Telegram — вся информация бота, ноль
потерь, быстрее и нагляднее текстовых карточек.

**Железные ограничения (нарушение = провал приёмки):**

1. **Один источник истины.** Mini App читает ТЕ ЖЕ функции, что и текстовые экраны
   (`_limitki_zone_status`, `shadow_outcome_analysis`, `_fetch_auto_liq_line` и т.д.)
   через тонкий API-слой. Запрещено пересчитывать статусы/скоры на фронте.
2. **Ноль потерь информации.** Каждое поле текстовой карточки присутствует в
   приложении. Чек-лист соответствия — раздел 8.
3. **Текстовый бот живёт дальше.** Mini App — дополнение. Алерты, команды,
   карточки в чате не трогаются. Любой экран приложения имеет текстовый
   эквивалент.
4. **Read-only v1.** Приложение ничего не пишет (никаких ордеров/изменений зон).
   Единственная запись — пользовательские настройки отображения (риск %,
   избранное) в отдельном storage, не в боевых файлах.
5. **Бюджеты данных.** CoinGecko не дёргается фронтом напрямую — только кэш бота.
   Ни одного нового потребителя внешних API.

## 1. АРХИТЕКТУРА

```
Telegram App
 └─ Mini App (React SPA, статика)
     └─ HTTPS → API-слой бота (aiohttp/FastAPI внутри worker
                или отдельный лёгкий сервис Railway)
         └─ существующие функции bot.py / shadow_engine /
            watch_zones.json / journal (+archive)
```

**Компоненты:**

- **Фронт:** React + Tailwind, сборка Vite, один bundle. Графики: lightweight-charts
  (TradingView OSS) — свечи+зоны; recharts — бары/спарклайны. Тёмная тема по
  умолчанию, следует `themeParams` Telegram.
- **API:** 8–10 read-only JSON endpoint'ов (раздел 3). Кэш ответов 5–30 сек в памяти —
  фронт никогда не бьёт во внешние API.
- **Хостинг статики:** Railway static / GitHub Pages — точка владельца (деньги = $0 в
  обоих вариантах).
- **Auth:** Telegram WebApp `initData` с проверкой HMAC-подписи (bot token) на
  каждом запросе. v1 — только owner chat_id 7009350191; поле `allowed_ids`
  заложить сразу под будущий клуб.
- **Realtime:** v1 — polling 10 сек на активной вкладке (просто, надёжно); v1.1 —
  WebSocket push (уже есть опыт Bybit WS). Пуш-алерты остаются в чате бота — это
  сила Telegram, не дублируем.

## 2. НАВИГАЦИЯ

Нижний таб-бар, 4 вкладки: **Дашборд · Зоны · Сигналы · Мои**. Хедер каждой
вкладки: BTC live-цена + статус соединения (🟢 live / 🟡 кэш N сек / 🔴 нет данных —
честное «н/д» визуально).

## 3. ЭКРАНЫ И API (полный функционал)

### 3.1 📊 ДАШБОРД — «рынок за 5 секунд»

`GET /api/dashboard`

- **Тикер-лента:** BTC, ETH, SOL + топ движения дня — цена, %24ч, спарклайн 24ч (из
  кэша бота).
- **Светофор рынка** (плашка цветом состояния): направление ликвидности
  (USDT.D+BTC.D), альт-направление (BTC.D+тренд), макро-фон (S&P асимметрия/
  DXY/VIX), funding-перегрев. Каждая строка = готовый вывод словами, тап →
  развёртка «почему» (те же данные, что в РЫНОК-экране бота).
- **Контуры (доверие):** мини-таблица tz13/П05/П09/EMA/L-S — закрыто/WR/PF/до
  гейта 20. Прямо из `closed_outcomes_report()`.
- **Лента событий дня:** входы в зоны, пампы, ликвидационные кластеры, новые
  разметки автора — reverse-хронология, тап → карточка.
- **Виджет ЛИМИТКИ:** счётчики 🟡/🟢/⚪/🔴 + ближайшая зона с дистанцией %.

### 3.2 🗺 ЗОНЫ — карта всех зон автора

`GET /api/zones` (единый источник: `_limitki_zone_status` + `watch_zones.json`)

- **Список карточек:** символ, LONG/SHORT, диапазон, prio, tier, статус цветным
  блоком (🟢 пульсирует), live-цена, дистанция %, заметка автора, ✓-отметки.
- **Фильтры-чипы:** статус / LONG-SHORT / prio / символ. Сортировка: по дистанции
  (default), по prio, по символу.
- **Детально (тап):** график lightweight-charts — свечи (кэш OHLC бота), зона
  прямоугольником, SL-уровень, текущая цена линией; таймфрейм 1D/4H из
  доступного кэша; вся мета зоны + история статуса.
- **Свайпы:** вправо — в избранное (localStorage-аналог Telegram CloudStorage),
  влево — свернуть.

### 3.3 🎯 СИГНАЛЫ — терминал сделки

`GET /api/signals` (AUTO-карточки + promoted), `GET /api/track-record`

- **Карточка сделки:** визуальная вертикальная шкала entry-зона → SL → TP1/TP2/TP3
  с % и R:R на каждой ступени; DCA 50/30/20 засечками на шкале.
- **Ползунок «мой риск»:** 1/2/3% → пересчёт сайзинга локально (формула из бота,
  вынесена в общий модуль расчёта — единый источник).
- **Мета полностью:** тип сетапа, POI H4, агрессив/консерватив, сессия, чек-лист
  Kira|ICT N/6 раскрываемым списком (какие пункты выполнены), Rocket Score с
  разложением слагаемых, ликвидации рядом, МЕМКОИН-бейдж + половинный размер.
- **Трек-рекорд (главная вкладка доверия):** таблица закрытых исходов по
  контурам, график equity-кривой по журналу (архив+актив через
  `get_local_records`), WR/PF/средний R:R, честные пометки «до гейта 20: N».
- **[❓ Словарь]:** каждый термин — тап → bottom-sheet с определением из
  `glossary.py` (тот же контент).

### 3.4 🧳 МОИ — портфель и план

`GET /api/portfolio` (парсинг `spot_portfolio_print.xlsx` через openpyxl — уже умеем)

- **Сводка:** стоимость, P&L $ и %, распределение 50/35/15 донат-диаграммой против
  целевого.
- **Позиции:** карточки монет — количество, средняя, текущая, P&L цветом.
- **DCA-лестницы:** AVAX 4.10/3.45/2.90, SUI 0.55/0.46/0.39 — прогресс-барами: какие
  ступени исполнены, дистанция до следующей, SL-линия.
- **Watch-лист** (LINK, UNI, DYDX, PYTH, ORDI, AAVE, ONDO, STRK…) с ценами и
  зонами, если есть в watch_zones.

## 4. ДИЗАЙН-СИСТЕМА

- Тёмная тема (фон #0e1621 семейства Telegram), акценты: зелёный/красный ТОЛЬКО
  по смыслу (LONG/SHORT, P&L), статусы зон — те же 4 цвета что в боте (🟡🟢⚪🔴) —
  единый визуальный язык бот↔приложение.
- Типографика: моно-шрифт для цен/уровней, системный для текста. Карточки —
  скруглённые блоки, сетка 8px, «воздух» обязателен.
- Скелетоны при загрузке, никогда пустой экран; ошибка сети → жёлтая плашка
  «данные N сек назад» (кэш), не белый экран.
- Хаптика Telegram WebApp на ключевые действия.

## 5. НАДЁЖНОСТЬ («работало идеально, без сбоев»)

1. API отдаёт из кэша бота; внешний сбой (CoinGecko 429) → приложение показывает
   последний кэш с честным таймстампом, не ломается.
2. Фронт — statically hosted: упасть может только API; при недоступности — экран
   «бот думает» + retry с backoff.
3. Версионирование `/api/v1/`; изменение схемы = новый маршрут, старый живёт релиз.
4. Rate-limit на API (token bucket per chat_id) — защита бюджета.
5. Логирование каждого 5xx через `log.error()` (закон №1).
6. Нагрузочный smoke: 100 req/мин на все endpoint'ы без деградации бота (алерты не
   должны задерживаться — проверка APScheduler-тиков во время нагрузки).

## 6. БЕЗОПАСНОСТЬ

- HMAC-валидация `initData` на каждом запросе; просроченный `auth_date` (>1ч) → 401.
- Whitelist chat_id (v1 — owner). Никаких токенов/кредов во фронте — bundle
  публичный по определению.
- API — только чтение; CORS замкнут на домен Mini App.
- Никаких пользовательских данных в git.

## 7. ЭТАПЫ И DoD

**Этап 1 — API-слой (3–5 дней):** endpoint'ы дашборда/зон/сигналов на существующих
функциях + HMAC + кэш + тесты (каждый endpoint: снапшот-тест схемы + сверка с
текстовым экраном на одних данных). DoD: curl с валидной подписью = JSON,
идентичный данным карточек бота.

**Этап 2 — Каркас фронта (3–5 дней):** таб-бар, Дашборд + Зоны на живом API, тёмная
тема, скелетоны. DoD: владелец открывает с iPhone кнопкой в боте, видит live-статусы
зон = экрану ЗОНЫ.

**Этап 3 — Сигналы + трек-рекорд (4–6 дней):** шкала сделки, ползунок риска,
таблица исходов, словарь. DoD: promoted-сигнал отображается со 100% полей
текстовой карточки (чек-лист раздела 8).

**Этап 4 — МОИ + полировка (3–4 дня):** портфель, DCA-лестницы, свайпы, хаптика,
нагрузочный smoke. DoD: полный проход владельцем всех вкладок с телефона, 0
дефектов приёмки.

Суммарно: ~2.5–3.5 недели фоном, ≤1-2 редеплоя на этап, каждый этап — отдельный
пакет с приёмкой по скринам.
