# ROADMAP_2W_DRAFT.md — черновик плана на 2 недели по итогам ночного анализа

Снято 2026-07-10. **Это черновик-предложение, ничего не начато.** Синтез всего, что
накопилось за сессию: `SECRETS_AUDIT.md`, `RESTORE_GUIDE.md`, `BACKTEST_REPORT.md`,
`knowledge/KNOWLEDGE_GAPS.md`/`SMC_COVERAGE.md`, живая проверка `signal_journal` на
2026-07-10. Порядок и решение по каждому пункту — за владельцем; ничего не подразумевает
автоматического старта. Пункты, трогающие сигнальную/торговую логику, помечены явно —
для них нужно отдельное «да» по каждому, как и раньше.

## Как читать оценки эффекта
Нигде ниже нет придуманных процентов. Где эффект неизвестен — так и написано, с
предложенным способом измерить его ПОСЛЕ (не вместо) внедрения, через уже существующие
инструменты (`signal_journal`, `/stats`, `backtest/journal_replay.py`).

## Неделя 1 — закрыть найденное, не трогая сигнальную логику

### 1.1 Секреты (SECRETS_AUDIT.md) — решение владельца, затем быстрая правка
- **Что:** `CMC_API_KEY` захардкожен как дефолт в `bot.py:117` — живой ключ в открытом
  виде в приватном репо прямо сейчас. Правка тривиальна (`os.getenv("CMC_API_KEY")` без
  второго аргумента, по образцу `BOT_TOKEN`), но меняет поведение при отсутствии env var
  на новом деплое — нужно явное «да».
- **Трудозатраты:** ~5 минут кода + подтверждение, что `CMC_API_KEY` реально задан в
  Railway env vars ПЕРЕД мержем (иначе бот тихо потеряет CMC на следующем деплое).
- **Эффект:** не измеряется метрикой — это гигиена секретов, не сигнальная фича.
- **Отдельно, вне моего контроля:** решение о ротации GitHub PAT (найден дважды —
  `Trading/BEST TRADE Handoff.pdf` и `.claude/settings.local.json`) и о валидности старого
  `BOT_TOKEN` из git-истории — обе строго на владельце (доступ к GitHub/BotFather).

### 1.2 Telethon-сессия reader.py — интерактивное действие владельца
- **Что:** `best_trade_reader.session` неавторизована (живой инцидент, не новый).
  Восстановление — только интерактивный вход через терминал, шаги уже в
  `RESTORE_GUIDE.md §3`. Не автоматизируется удалённо.
- **Трудозатраты:** ~5 минут владельца (телефон/SMS-код).
- **Эффект:** восстанавливает 13 архивируемых/сигнальных Telegram-каналов reader.py,
  сейчас полностью офлайн — весь этот источник данных недоступен, пока не сделано.

### 1.3 Проверить, почему `signal_loop`/`pump_radar` пока не оставили ни одной записи в журнале
- **Что:** `get_journal_summary()` на 2026-07-10 показывает 79 записей, все три источника
  — `TOP_LONG_AUTO`/`TOP_SHORT_AUTO`/`TOP_SPOT_AUTO`. Ноль записей от `signal_loop`
  (ROADMAP П2-фича, работает с этой сессии) и `PUMP_RADAR`. Не обязательно баг — может
  быть честно "пока не было подходящих сетапов" (ступень 2 `fa_engine.has_setup` строгая
  по конструкции) — но не проверено, надо явно отличить "работает, просто редко
  срабатывает" от "тихо не работает".
- **Трудозатраты:** ~30 минут — прочитать логи Railway за последние 24-48ч на предмет
  реальных прогонов `signal_loop`/`pump_detector`, без изменения кода.
- **Эффект:** если находка "тихо не работает" — экономит будущую путаницу в `/stats`
  ("почему по signal_loop 0 сигналов за неделю" будет с ответом, не гаданием).
- **Не сигнальная логика** — только диагностика/логи, безопасно сделать без approval.

### 1.4 Долечить Фазу 2 (транскрибация видео) и последние непрочитанные PDF
- **Что:** фоновый процесс транскрибации (faster-whisper) — 66/159 на 2026-07-10,

## Неделя 2 — предложения, требующие явного «да» владельца (сигнальная логика)

Порядок — не приоритет, все требуют отдельного решения. Для каждого — сначала
ретроспективная проверка на уже накопленных данных `signal_journal`, где возможно, а не
изменение вслепую (см. `KNOWLEDGE_GAPS.md` "Предложения из библиотеки" — эти пункты
переносятся сюда с уточнённым статусом после `SMC_COVERAGE.md`).

### 2.1 pump_detector.py: реальная проверка BOS после манипуляции (было "баг №5" находка)
- **Источник:** `4_4_Манипуляция_большого_игрока_STBBTS_.pdf` — канон требует слом
  структуры как обязательное подтверждение; сейчас TP строится как `entry ± minRR*risk`
  без проверки.
- **Трудозатраты:** средние — нужна интеграция `ta_extra.smc_setup_type`/`swing_points` в
  `pump_detector.py`'s `_confirm_pump_reversal`/`_confirm_dump_reversal`.
- **Эффект:** не измерен. Гипотеза — снизит число PUMP_RADAR-сигналов, может повысить
  win-rate. **Как измерить:** сравнить `source="PUMP_RADAR"` в `/stats` до/после (сейчас в
  журнале таких записей 0 — сначала стоит накопить историю "до", см. п.1.3 выше).

### 2.2 Ярлыки SMC-факторов на карточке ("BOS↑"/"OB накопл."/"FVG↑") — переименовать или заменить механику
- **Источник:** новая находка этой сессии, `SMC_COVERAGE.md §3` — эти ярлыки в
  `real_full_analysis`/`full_analysis` считаются по % изменению цены, не по свечным
  паттернам, хотя названы как настоящие SMC-термины.
- **Два разных варианта решения (оба требуют «да», разного объёма):**
  - (а) **Дёшево:** переименовать ярлыки в нейтральные названия ("Импульс ↑", "Штиль
    объёма"), не трогая формулы — устраняет вводящее в заблуждение название за ~30 минут.
  - (б) **Дороже:** подключить уже существующие реальные детекторы (`pro_analysis.
    ict_ob_bull/fvg_bull`, либо `ta_extra.find_fvg_zones`) вместо % эвристик — устраняет
    расхождение по существу, не только по названию.
- **Эффект:** не измерен, это про точность отображаемой информации, не про win-rate
  напрямую (хотя вариант (б) может неявно повлиять на rocket_score, если так решит
  владелец).

### 2.3 FVG/Premium-Discount как фактор чек-листа основного движка (не только `/long`-only)
- **Источник:** `SMC_COVERAGE.md §2`, `KNOWLEDGE_GAPS.md` пп.1/4 (исправленные).
  `pro_analysis()` уже считает и то, и другое — вопрос не "писать с нуля", а "поднимать ли
  до статуса гейта/фактора в `fa_engine.py` block5, используемого во всех путях".
- **Трудозатраты:** средние — перенос уже существующей логики, не новая разработка.
- **Эффект:** не измерен. **Как измерить:** добавить `levels_source="fvg"`/поле
  `premium_discount_zone` в `signal_journal` аддитивно, накопить 4-8 недель, сравнить
  win-rate против текущего `"structure"` ПЕРЕД переключением на гейт.

### 2.4 SMT-дивергенция (альт vs BTC/BTC.D) как информационное поле — самый низкий риск
- **Источник:** `40. SMT.pdf`, `KNOWLEDGE_GAPS.md` п.13/предложение 5.
- **Трудозатраты:** средние (нужен доп. запрос BTC.D + сравнение структуры).
- **Эффект:** не измерен. Единственный пункт, который можно сделать ЧИСТО
  информационным (доп. строка в карточке + лог поля), не трогая гейты вообще — поэтому
  самый безопасный из вошедших в черновик по сигнальной части.

### 2.5 R:R-гейт (1.5) vs учебный минимум (1:3) — только ретроспективный анализ, не смена порога
- **Источник:** `KNOWLEDGE_GAPS.md` предложение 2, личная методичка владельца.
- **Трудозатраты:** низкие — данные уже есть в `signal_journal` (`rr`/`rr_tp1` на каждой
  записи), нужен только скрипт разбивки win-rate по бакетам R:R.
- **Эффект:** не измерен, и при N=29 закрытых сделок статистически рано делать выводы
  (см. `BACKTEST_REPORT.md` — индустриальный ориентир 30+ для предварительных, 100+ для
  надёжных). **Рекомендация: отложить сам анализ на 2-4 недели**, пока не накопится
  больше закрытых сделок — попытка сейчас даст шум, не сигнал.

- **Псевдо-события "план Фазы 2 видео"** — уже одобрено и идёт фоном, не требует нового

  ROADMAP.md П4 до накопления существенно большей истории; ничего в этом черновике не
  предлагает менять пороги, только собирать данные для будущего решения.

## Как читать этот документ

Ничего не начато. Каждый пункт Недели 2 (и частично 1.1) требует отдельного явного «да»
