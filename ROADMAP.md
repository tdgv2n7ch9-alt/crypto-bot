# ROADMAP — BEST TRADE crypto-bot

План улучшений по приоритетам. Это **план, не реализация** — ничего из перечисленного здесь не
внедряется без отдельного явного одобрения по каждому пункту. Торговая/сигнальная логика (пороги
чек-листа, R:R-гейт, критерии stage1/stage2, формулы TP/SL) — вне рамок этого roadmap до отдельного
решения; изменения тут касаются надёжности, измеримости, качества данных и инфраструктуры вокруг
логики, а не самой логики.

Границы для всех пунктов: не трогать ключи бирж, не менять торговую/сигнальную логику без явного
одобрения, ничего не рассылать реальным подписчикам в процессе работы, `dr_linessa_bot` не
затрагивать.

---

## П1 — Надёжность (контур выживания)

Цель: бот не должен тихо умирать и тихо терять данные. Сейчас единственная реальная страховка —
`restartPolicyType: ON_FAILURE` на Railway (спасает только от краша, не от зависания) и watchdog
памп-радара (защищает только один WS-канал). Всё остальное — открытое поле.

1. **Реконсиляция git-истории с origin** (см. AUDIT §2.1) — отдельный, самый рискованный пункт.
   Нужно решение: либо переинициализировать локальный репозиторий от чистого `origin/main` (риск
   потери незакоммиченных локальных правок в bot.py/reader.py — их нужно будет сначала явно
   сохранить/применить поверх), либо принять расхождение как есть и просто документировать, что
   единственный источник правды — GitHub, а не эта рабочая копия. Требует твоего решения до любых
   действий — это не техническая мелочь, а вопрос "что считать истиной".
2. **Health/heartbeat для основного процесса**, не только для памп-радара — например, периодический
   self-check (жив ли event loop, жив ли scheduler, когда был последний успешный тик каждой
   фоновой задачи) и способ узнать об этом дистанционно (расширение `/radar_status` до общего
   `/health`, либо внешний heartbeat типа "бот раз в N минут отмечается, тишина = алерт").
3. **Атомарность записи локальных файлов состояния** (`signal_journal.json`, что бы ни хранилось
   локально) — запись во временный файл + rename вместо прямой перезаписи, чтобы крах посреди
   записи не портил файл.
4. **Версионированные бэкапы GitHub-as-DB файлов** (`data/chat_ids.json`, `journal/signals.json`) —
   сейчас это "последняя запись побеждает" без истории снапшотов. Нужен хотя бы дневной снапшот
   в отдельном пути/бранче, чтобы можно было откатиться, если файл повредится или перезапишется
   некорректными данными.
5. **Retry/backoff и timeout там, где их нет** — CMC, часть Bybit REST-вызовов, Yahoo DXY. Не
   меняя *что* запрашивается, только *как надёжно* — таймауты на все `requests.get`, единая
   retry-обёртка с экспоненциальным backoff по образцу уже существующего CoinGecko-лимитера.
6. **Устранить "тихие" bare `except:`** — минимум добавить логирование (что упало, где, с каким
   значением по умолчанию ушло дальше), не меняя поведение при ошибке.
7. **Уборка мёртвого веса** — `bot_old.py` (319KB, нигде не используется), пустые файлы `=`/`Next`,
   разобраться со статусом `backtest_signal()` (либо подключить, либо явно пометить/удалить как
   мёртвый код) — housekeeping-уровень, но снижает шум и риск случайного использования старого кода.
8. **Ротация PAT в GitHub** — токен был виден в открытом виде в `.git/config`; сама утечка устранена
   в этой сессии, но старый токен стоит считать скомпрометированным и отозвать/перевыпустить на
   стороне GitHub (это твоё действие, не автоматизируется отсюда).

## П2 — Измеримость

Цель: не строить трекинг с нуля (он уже есть и работает, см. AUDIT §3), а закрыть конкретные дыры
в том, что уже собирается, и дать владельцу дашборд, а не только текстовую команду.

1. **Разбивка результативности по рыночному режиму** — `ema_stack`, sweep, фаза Вайкоффа уже
   пишутся в каждую запись журнала, но не агрегируются. План: считать win-rate/avg-R в разрезе
   режима (тренд вверх/вниз/флэт, высокая/низкая волатильность), используя уже собранные поля —
   без новых источников данных, просто аналитика поверх существующих.
2. **Журнал отклонённых кандидатов** — сейчас логируются только отправленные сигналы. План:
   отдельный (лёгкий, без разрастания основного журнала) лог "кандидат дошёл до проверки, не
   прошёл гейт X" — нужен для анализа ложноотрицательных срабатываний и будущей калибровки
   порогов по П4.
3. **Ротация/архивация журнала** — политика хранения (например, "полные записи 90 дней, дальше —
   только агрегаты"), чтобы файл не рос бесконечно.
4. **Дашборд/статистика для владельца** — сейчас `/journal` это текстовый отчёт. План: либо
   расширить его (динамика по неделям, топ худших/лучших источников сигналов), либо отдельный
   визуальный отчёт (например, генерируемая по запросу картинка/страница) — конкретная форма к
   обсуждению отдельно, здесь только факт потребности.
5. **Сверка порогов гейтов с фактической результативностью** — раз в период (не автоматически
   меняя пороги!) сравнивать "что показал `/journal`" с текущими константами чек-листа/R:R/rocket,
   чтобы решения о пересмотре порогов принимались на данных, а не только по ощущению — сам
   пересмотр порогов остаётся отдельным решением с явным одобрением (это уже область сигнальной
   логики).

## П3 — Качество данных

Цель: меньше тихой деградации данных, без добавления новых источников ради количества.

1. **Аудит и, если нужно, чистка оставшихся прямых вызовов `api.binance.com`** — подтвердить, какие
   из них реально работают на Railway (гео-блокировка), какие тихо всегда фейлят и маскируются
   bare `except`. План действий по каждому — заменить на уже используемый CoinGecko-путь, или
   явно задокументировать как "не работает, оставлено намеренно" — решение по каждому месту
   отдельно, это уже касается data-путей сигналов, требует твоего ОК.
2. **Замена/усиление DXY-источника** — Yahoo-эндпоинт неофициальный и уже стоил 5+ хотфиксов.
   Рассмотреть более стабильный источник макро-данных или как минимум явную валидацию + staleness
   check вместо тихого сброса в 0 при ошибке.
3. **CMC-ключ** — если 401 продолжает появляться параллельно с квотой, это, вероятно, реальная
   проблема ключа/аккаунта, а не временная перегрузка — требует твоего решения (ротация ключа,
   смена тарифа, либо снижение зависимости от CMC в пользу CoinGecko там, где дублируется).
4. **Явные staleness-проверки** там, где их ещё нет (по образцу уже существующей в `live_prices.py`
   — `<10с = live, >60с = fallback с честной пометкой`) — распространить тот же паттерн на другие
   источники, где сейчас нет проверки возраста данных.
5. **Nitter-зависимость в reader.py** — низкий приоритет (изолирован от ядра бота и подписчиков),
   но стоит держать в уме как хрупкое звено, если релей владельцу станет критичным.

## П4 — Аналитика рынка

Цель: то, что реально повышает качество сигналов и проверено индустрией — без обещаний волшебной
точности. Всё в этом разделе имеет смысл только после того, как П1-П2 дадут достаточно чистых и
измеримых данных, чтобы вообще было что валидировать.

1. **Настоящий бэктест, подключённый к реальному конвейеру** — не через оторванный
   `backtest_signal()` с собственной реконструкцией TP/SL, а walk-forward прогон через тот же
   K-LVL/чек-лист/R:R-гейт, что использует прод, на исторических данных. Это стандартная
   индустриальная практика валидации правил, а не "магия предсказания" — цель: подтвердить или
   опровергнуть, что текущие пороги вообще имеют edge, а не просто выглядят разумно в коде.
2. **Калибровка порогов на исторических + живых данных журнала** (после накопления достаточного
   объёма закрытых сигналов из П2) — это касается сигнальной логики, поэтому только предложение
   с данными на руках, решение и внедрение — отдельно и с explicit approval.
3. **Расширение market-regime awareness** — фаза Вайкоффа уже используется как модификатор Rocket
   Score; возможное развитие — адаптивные (а не статичные) пороги R:R/чек-листа в зависимости от
   волатильности (например, ATR-нормализация), проверенная практика в systematic trading. Тоже
   изменение сигнальной логики — только план, не реализация.
4. **Честная граница возможного**: без ончейн-аналитики уровня платных провайдеров, без
   гарантированной альфы — реалистичная цель этого раздела — постепенно повышать долю сигналов,
   которые подтверждаются собственной историей бота, а не гнаться за новыми "чудо-индикаторами"
   без проверки. Любой новый индикатор/источник добавляется только если для него можно
   сформулировать проверяемую гипотезу и она проходит через журнал/бэктест раньше, чем уходит в
   прод.

---

## Как читать этот план

П1 и П2 не требуют изменения сигнальной логики и в основном безопасны для последовательного
внедрения после твоего одобрения по каждому пункту. П3 местами затрагивает пути данных, питающие
сигналы, — согласование по месту. П4 целиком про сигнальную логику — только после того, как П1-П2
дадут данные, на основе которых решения принимаются осознанно, а не по наитию, и только с твоим
явным одобрением на каждый шаг.
