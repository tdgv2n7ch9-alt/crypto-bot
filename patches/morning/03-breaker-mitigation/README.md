# Патч 3: Breaker vs Mitigation Block классификатор — 🟡 средний приоритет

## Что меняет

Новая аддитивная функция `ta_extra.classify_breaker_or_mitigation(candles, direction)`.
**Не подключена ни к одному вызывающему коду** — применение этого патча само по себе
НИЧЕГО не меняет в поведении бота, только добавляет новую функцию.

Реализует дерево решений из `knowledge/METHODOLOGY_CORE.md §3` (подтверждено 2
независимыми источниками курса):
- **Breaker**: на экстремуме свинга, от которого сформирован новый структурный HH/LL,
  ликвидность СНИМАЕТСЯ (импульс пробивает уровень навылет).
- **Mitigation Block**: ликвидность НЕ снимается — вместо этого формируется более
  высокий лоу/более низкий хай, и уже после этого ломается структура.

## Реализация — важный нюанс

Кандидат на breaker/MB — ВТОРОЙ с конца фрактальный свинг (`lows[-2]`/`highs[-2]`), не
последний. Причина в самой природе фрактального обнаружения: если бы что-то позже
опустилось ниже последнего фрактального лоя И само стало новым фракталом, оно бы уже
считалось последним свингом — поэтому "свип последнего перед структурным движением
свинга" физически виден только на втором с конца. Подробности в докстринге функции.

Протестировано 4 сценария (both direction long/short, breaker/mitigation, edge case
пустых данных) — синтетические свечи, не реальные исторические данные.

## Ожидаемый эффект

Не измерен.

## Как измерим на журнале (если/когда будет решение подключить к реальному пути)

Аддитивное поле `zone_type: "breaker"|"mitigation"|None` в `signal_journal.log_signal()`
(по образцу уже существующего `levels_source`), накопить историю, сравнить win-rate
breaker vs mitigation vs текущий unclassified путь.

## Риск применения самого патча (добавление функции)

Минимальный — новый код, ничего не вызывает, не может сломать существующее поведение.
Риск возникает только на следующем, отдельном шаге — если/когда кто-то решит ПОДКЛЮЧИТЬ
эту функцию к реальному построению SL/входа, что требует отдельного явного решения и не
входит в этот патч.
