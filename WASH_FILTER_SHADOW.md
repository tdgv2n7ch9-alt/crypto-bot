  (`rug_radar.detect_vertical_growth_thin_volume()` проверяет ОБРАТНОЕ —
  ТОНКИЙ объём при вертикальном росте, порог `<15%`; здесь объём наоборот
  избыточен относительно MCap — другой класс паттерна, не покрыт

**Что должно было удержать вход**: ни один из существующих боевых
гейтов/чек-листа не проверяет ни FDV/MCap, ни концентрацию объёма по venue —
`rug_radar.py` считает похожие метрики, но **отдельно от сигнального
чек-листа** (rug-risk — информационная строка в карточке/алерте, не gate).
Реальный удерживающий фактор, если бы существовал: **высокий Vol/MCap
(>100%) + высокая концентрация объёма на одном venue (>50%) одновременно**
— комбинация, которую сейчас не считает ни `rug_radar.py`, ни чек-лист
сигнала.

## Три shadow-признака (кандидаты, без выбора)

**1. Vol/MCap высокий-порог (обратный `detect_vertical_growth_thin_volume`)**

```
wash_volume_ratio_pct = volume_24h / market_cap * 100
```
Порог-кандидат: `> 100%` (AKE ≈101%, MIRA 594% — оба явно выше; обычные
топ-100 монеты обычно в диапазоне 2-20%, не проверено на большой выборке
в рамках этого спека — честно, нужна калибровка на исторических данных
прежде чем фиксировать порог, `>100%` — рабочая гипотеза по двум точкам,
не статистически выведенное значение).

**2. Концентрация объёма на одном venue**

```
top_venue_share_pct = max(volume_by_exchange.values()) / sum(volume_by_exchange.values()) * 100
```
Источник данных — уже используемый `tickers` массив CoinGecko-ответа
(`/coins/{id}?tickers=true`, тот же эндпоинт, что `rug_radar.fetch_
coingecko_detail()` уже вызывает — **0 дополнительных сетевых вызовов**,
поле просто не извлекается сейчас). Порог-кандидат: `> 50%`.

**3. Глубина стакана относительно MCap (Liq/MCap) — единственный НОВЫЙ
сетевой вызов из трёх**

```
book_depth_2pct_usd = сумма bid+ask объёма в USD в полосе ±2% от mid price
liq_mcap_ratio_pct = book_depth_2pct_usd / market_cap * 100
```
Требует прямого REST-снимка стакана (Bybit `/v5/market/orderbook` или
Binance-эквивалент, ОДИН запрос по конкретному символу в момент проверки —
НЕ через Whale Radar WS-инфраструктуру, та рассчитана на持ные ~50
предустановленных символов потоковым способом, для разового rug-check на
произвольном тикере это архитектурно неверный инструмент, нужен именно
разовый REST-снимок). Порог-кандидат: MIRA имела ~$27-37K глубины при
MCap $14.9M → Liq/MCap ≈ 0.2%; предлагаемый порог `< 1%` — тоже
гипотеза по одной точке, требует калибровки.

## Как встроить (по аналогии с уже существующим паттерном, не новая архитектура)

Тот же принцип, что `amd_phase_methodology`/`cftc_cot_snapshot` (план) —
**shadow-only informational-поле** `wash_class_shadow`, прикрепляемое к
записи при генерации сигнала (`compute_shadow()`/`log_send_scheduled_
shadow_async()`), СЧИТАЕТСЯ, но НЕ участвует в `gate_reasons`/`promoted`
на этом этапе. Копится статистика (промоутнутые сигналы с высоким
`wash_volume_ratio_pct`/`top_venue_share_pct` — какой у них реальный
исход), тем же протоколом строгости, что `AMD_FILTER_SHADOW.md` (n≥20
закрытых, PF как критерий) — ПЕРЕД любым рассмотрением как gate.

**Ограничение источника данных**: этот shadow-признак НЕ может покрывать
`full_analysis`-сигналы автоматически без изменения вызывающего кода в
`bot.py` (сейчас shadow-логирование подключено только к AUTO-контуру,
см. докстринг выше про честный пробел на AKE) — расширение на
`full_analysis`/`signal_loop` пути является ОТДЕЛЬНЫМ, более крупным
изменением (не в объёме этого SPEC), честно фиксирую как открытый вопрос,
не решаю здесь.

## Стоимость (оценка)

- Признаки 1 и 2 — **$0, без новых сетевых вызовов** (используют уже
  получаемые CoinGecko-поля).
- Признак 3 — один дополнительный REST-вызов к Bybit/Binance orderbook
  per сигнал (не per тик сканирования) — дёшево по объёму (сигналы
  генерируются на порядки реже, чем AUTO-сканы кандидатов), но требует
  нового небольшого HTTP-клиента (или переиспользования уже
  существующего `_cg_get`-подобного паттерна retry/circuit-breaker для
  Bybit-эндпоинта, если такой уже есть в проекте — не проверено в рамках
  этого документа, отдельная инженерная деталь для реализации).

## Что НЕ сделано (честно, SPEC-ONLY)

1. Пороговые значения (100%/50%/1%) — рабочие гипотезы по ДВУМ точкам
   (AKE, MIRA), не калиброваны на исторической выборке. Прежде чем даже
   думать про gate — нужен прогон по всей истории journal (тот же
   стандарт, что AMD-фильтр).
2. Код не написан, ничего не подключено к `compute_shadow()`/`bot.py`.
3. Не проверено, есть ли уже переиспользуемый Bybit-orderbook REST-клиент
   в проекте (кандидат для признака 3) — если есть, стоимость реализации
   ниже, чем оценено выше.
4. Расширение на `full_analysis`-путь (не только AUTO) — открытый вопрос,
