### 1.2 Как формируются сигналы

Два независимых пути генерации:

**А. `signal_loop.py` — проактивный луп (каждые `STAGE1_INTERVAL_MIN`=15 мин)**
1. Stage 1 (`_stage1_screen`) — дешёвый отбор по экстремумам funding, всплеску OI или свежему liquidity sweep на 4h, фильтр по объёму (`MIN_VOL_USD=5M`), кулдаун 4ч на символ.
2. Stage 2 (`_stage2_check`) — прошедшие отбор уходят в `fa_engine.build_full_analysis()`; сигнал уходит подписчикам только если `has_setup=True` в Block 11 (единый источник правды, без дублирования критериев).

**Б. Ручные сканы / команды (`/long`, `/short`, `/spot`, `/full`, `/coin`, `/x100`, AUTO-рассылка `send_scheduled` раз в 30 мин)** — используют `real_full_analysis()` либо `fa_engine` напрямую, в зависимости от команды (см. память проекта: `/coin`/`/full` — fa_engine, AUTO-луп и топ-сканы — `real_full_analysis()`, выбрано по стоимости на сделку, не по качеству).

**Индикаторы/структуры (`ta_extra.py`, 899 строк):** EMA-стек и контекст, фрактальные swing-точки, wick-ratio фильтры, подтверждение объёмом, liquidity sweep, поиск S/R-зон (кластеризация касаний по нескольким TF), RSI, мульти-TF bias, эвристика волн Эллиотта, SMC (BOS/CHoCH), FVG-зоны, equal highs/lows, эвристика фазы Вайкоффа, классификация зон K-LVL, детекция polarity-flip.

**Реальные гейты качества (`fa_engine.py`):**
- Чек-лист ≥4 из 6 (`CHECKLIST_MIN_FOR_TRADE=4`): согласие с 1D-трендом, свежий sweep в сторону сделки, цена у POI (в пределах 1.5%), активная/близкая killzone, funding не против сделки (±0.05%), структурный R:R ≥1.5.
- Отдельный жёсткий гейт `rr_gate_pass` (R:R ≥1.5) — независимо от суммы чек-листа.
- Точка входа обязана быть из зоны, классифицированной как K-LVL (≥2 критерия подтверждения) — просто зона с 2 касаниями не считается.
- Rocket Score (Block 12, 0-100) влияет на отображение и на гейты в топ-сканах (по памяти проекта — `rocket>=60` + grade A+/A/B), но **не** является гейтом внутри `signal_loop.py` — там решает только `has_setup`.
- Risk-флаг по мем-коинам (rank>200 / mcap<$100M / vol24h<$10M) снижает размер позиции, но не блокирует сигнал.

Пороги — константы, подобранные вручную (по комментариям в коде), а не откалиброванные по истории (см. §5.3 — бэктестов нет).

## 2. Архитектура и зависимости

- **Два процесса** (Procfile): `worker` = `bot.py` (в его же событийном цикле как `asyncio.create_task` крутится и `pump_detector.py` — это не отдельный процесс) и `reader` = `reader.py` (отдельный Telethon-процесс, управляется launchd вне репозитория — `com.bestrade.reader.plist` подтверждён локально). Между процессами **нет общего файла состояния** — единственная связь — общий `BOT_TOKEN` для отправки сообщений владельцу. Если `reader.py` падает, страдает только ретрансляция внешних каналов владельцу; основной бот не затронут. Если падает `bot.py` — падает всё остальное (сканы, сигналы, доставка подписчикам), `reader.py` продолжает жить отдельно.
- **Хостинг**: Railway, Hobby plan, сборка через `Dockerfile` (python:3.11-slim) — миграция с Nixpacks/Railpack произошла 2026-07-07/08 после инцидента с потерей `.py`-исходников в образе. `railway.json`: `restartPolicyType: ON_FAILURE` — это единственный механизм восстановления после краша процесса, специального health-check эндпоинта нет (бот — чистый Telegram long-polling воркер, HTTP-сервера нет вообще).
- **Зависимости** (`requirements.txt`): `python-telegram-bot 20.7`, `requests`, `apscheduler`, `pytz`, `matplotlib`, `numpy`, `websockets`. Нет `pytest` (тесты — самописные смоук-скрипты, см. ниже), нет линтера, нет CI (папки `.github/workflows` нет вообще — тесты не запускаются автоматически ни при коммите, ни при деплое).
- **Хранилище**: реальной БД нет. Два JSON-файла в приватном GitHub-репозитории через Contents API (`data/chat_ids.json`, `journal/signals.json`) играют роль БД — придумано специально из-за эфемерности файловой системы Railway. Остальное состояние (`price_cache`, `alerted_zones`, `pump_alerted`, `active_game` и т.д., bot.py:186-193) — чисто в памяти процесса, теряется при каждом рестарте без каких-либо последствий для корректности (пересобирается на лету).
- **Тесты**: `test_chart_v4.py`, `test_narrative.py` — синтетические смоук-тесты без pytest, запускаются только вручную (`python3 test_*.py`), не подключены ни к какому CI/pre-commit.
- **Мёртвый вес в репозитории**: `bot_old.py` (319KB, 8396 строк) — нигде не импортируется и не упоминается ни в Procfile, ни в Dockerfile, ни в самом bot.py; похоже на оставленную резервную копию до переписывания. Пустые файлы `=` и `Next` в корне репозитория — судя по всему, случайно созданные (опечатка в shell-команде), назначения не несут.

### 2.1 Критическая находка: git-история и origin разошлись

- `git status`: локальная ветка `main` разошлась с `origin/main` — **14 коммитов только локально, 134 (после fetch) только в origin**.
- Причина: по памяти проекта и по коду (`subscribers.py`, `signal_journal.py`) пуши в прод идут **через GitHub Contents/Git Data API напрямую**, в обход обычного `git push` из этого рабочего каталога. Локальный `git`-репозиторий в `~/crypto-bot` физически не участвовал в доброй половине истории, которая реально задеплоена.
- Следствие похуже: `fa_engine.py`, `signal_journal.py`, `pump_detector.py`, `signal_loop.py`, `ta_extra.py`, `chart_v3.py`, `chart_v4.py`, `narrative.py`, `live_prices.py`, `subscribers.py`, `test_*.py`, `Dockerfile`, `railway.json` — **это файлы, физически лежащие на диске и совпадающие по содержимому с origin/main, но НЕ отслеживаемые локальным git** (висят как untracked). Формально это означает, что почти весь бэкенд бота не имеет здесь локальной git-истории вообще — если рабочий каталог будет потерян до следующего коммита, единственная копия правды — GitHub (что само по себе нормально), но **страховка "git как safety net" на этой машине сейчас не работает** для большей части кодовой базы.
- При этом содержимое `bot.py` в рабочей копии и `origin/main` идентично (`BOT_VERSION = v130` в обоих) — рабочий каталог не отстаёт по факту, отстаёт именно git-учёт.
- Это не тронуто в рамках аудита (слияние разошедшихся историй — операция с риском конфликтов и потери незакоммиченных правок в `bot.py`/`reader.py`) — вынесено отдельным пунктом в ROADMAP на явное решение.

## 3. Учёт результативности сигналов (проверка гипотезы "пробел №1")

**Гипотеза не подтвердилась — трекинг исходов реализован и реально работает, а не мёртвый код.**

- При выдаче сигнала (`log_signal()`, signal_journal.py:297-340, вызывается из bot.py:4419, 8275, 8727, 8929, 9207, 9455, 9474 и signal_loop.py:320) фиксируются: id, таймстемпы, источник (TOP_LONG/TOP_SHORT/TOP_SPOT/X100/PUMP_RADAR/signal_loop), символ, направление, entry_lo/hi, SL, TP1/2/3, R:R, rocket_score, снэпшот EMA-стека, снэпшот sweep, источник уровней (structure/fallback_atr), grade (A+/A/B/C), цена на момент сигнала, статус.
- Трекер исходов (`run_tracker()`, signal_journal.py:360-463) реально запущен фоновой задачей при старте (`bot.py:9670`, `asyncio.create_task`), опрашивает `live_prices` раз в 30с, переводит сигнал PENDING → ENTERED (цена коснулась зоны входа) → TP1/2/3_HIT либо SL_HIT (SL проверяется первым — консервативно), либо PENDING → EXPIRED через 72ч без входа.
- Агрегированная статистика (`get_journal_summary()`, signal_journal.py:473-518): доля вошедших в рынок, win-rate, средний фактический R, разбивка по источнику и по grade, окна 24ч/7д/весь период. Доступна владельцу через `/journal`. Есть и обратная связь в реальном времени — `get_stats_for_source()` подмешивает "закрыто N, win-rate X%" прямо в карточки новых сигналов.

**Реальные пробелы внутри уже работающего трекинга (не отсутствие, а неполнота):**
1. **Нет разбивки по рыночному режиму.** `ema_stack`/`sweep`/фаза Вайкоффа сохраняются в каждой записи, но нигде не агрегируются — в докстринге прямо написано "для будущего статистического анализа". То есть данные собираются, но не используются.
2. **Запись нативная, не атомарная.** `_save()` (signal_journal.py:288-294) — обычный `open(...,"w")` + `json.dump`, без временного файла + rename, без файловой блокировки. Крах процесса посреди записи может обрезать/повредить JSON. GitHub-синхронизация (после каждого закрытия сигнала + принудительно раз в час + вручную `/journal_sync`) частично страхует, но не защищает от локальной порчи между синками, и сама по себе не является версионированным бэкапом (перезаписывается тем же файлом при каждом коммите).
3. **Журнал растёт бесконечно** — нет функции прунинга/ротации/архивации нигде в `signal_journal.py`. Сейчас это несколько сотен записей / 49КБ, но потолка нет.
4. **Подписчики без сегментации.** `subscribers.py` хранит только `{chat_id: {subscribed, updated_ts}}` — ни персональных настроек, ни rate-limit на пользователя, только глобальный подписка/отписка через `/start`/`/stop`.
5. **Отклонённые сигналы не логируются** — журнал содержит только то, что реально отправлено. Нет данных о том, что было отфильтровано гейтами и почему (нужно для анализа ложноотрицательных срабатываний и калибровки порогов).

## 4. Метрики: что есть и чего нет

**Есть:**
- Win-rate, доля вошедших в рынок, средний фактический R — по источнику сигнала и по grade, в окнах 24ч/7д/all-time (`/journal`).
- `/radar_status` — диагностика памп-радара (реконнекты, последний тик, состояние WS).
- Версия бота в `/start`.

**Нет:**
- Win-rate/точность в разрезе рыночного режима (тренд/флэт/высокая-низкая волатильность) — данные для этого уже собираются (`ema_stack`, wyckoff phase), но не считаются.
- Общий health/heartbeat самого `bot.py` — процесс либо жив, либо (по факту) о его зависании узнают только по тишине в чате. `/radar_status` покрывает только памп-радар, не сканер, не scheduler, не Telegram polling loop.
- Метрика по отклонённым/отфильтрованным гейтами сигналам (сколько кандидатов дошло до чек-листа и не прошло, по какой причине).
- Данные о задержках/сбоях источников данных как метрика во времени (сейчас — только текстовые логи ошибок, не агрегированные счётчики).
- Любая сверка предсказанных порогов (checklist≥4, R:R≥1.5, rocket≥60) с фактической результативностью — потенциально уже считается вручную через `/journal`, но не автоматизировано и не влияет на пороги.

## 5. Слабые места

### 5.1 Надёжность
- 18 голых `except:` в bot.py (в основном в низкоуровневых fetch-хелперах вроде `get_binance_24h`, OI/funding, ATL) — глотают ЛЮБУЮ ошибку молча, без единой строки лога, возвращая нулевые/пустые дефолты. Сбой здесь невидим и выглядит downstream как "подозрительно нулевые данные", а не как ошибка.
- 81 `except Exception as e:` — почти все логируют `str(e)`, но только одно место (`cmd_market`, bot.py:2581-2582) логирует полный traceback. В остальных случаях стек теряется, что усложняет диагностику постфактум.
- Retry/backoff реализован практически только для CoinGecko (общий rate-limiter + кэш). CMC, часть Bybit-вызовов, Yahoo DXY — одна попытка, без retry, местами без `timeout=` вовсе (bot.py:568, 4087, 5175, 5452) — то есть при зависшем сокете вызов может блокировать выполнение неопределённо долго.
- Health/watchdog есть только у WS памп-радара (60с таймаут, авто-reconnect, уведомление владельца — pump_detector.py:95-144, один из лучше всего инструментированных узлов проекта). Ничего не следит за самим `bot.py`, scheduler'ом или Telegram polling loop — единственная страховка от зависшего (не упавшего) процесса это её отсутствие; `restartPolicyType: ON_FAILURE` спасает только от краша, не от deadlock.
- Локальные файлы (`chat_ids.txt`, `signal_journal.json`) гитигнорятся и эфемерны на Railway — именно это стало причиной инцидента 07-07/08. GitHub-as-DB это чинит частично, но без версионированных бэкапов и атомарной записи (см. §3).

### 5.2 Устаревшие/хрупкие источники данных
- Прямые вызовы `api.binance.com` (spot depth/aggTrades/24hr) оставлены в коде, хотя собственные комментарии проекта подтверждают гео-блокировку Binance на инфраструктуре Railway — не подтверждено, мигрированы ли эти конкретные вызовы; вероятная тихая деградация в проде.
- DXY через недокументированный Yahoo-эндпоинт с bare `except` — история v56-v70 (5+ хотфиксов подряд именно на парсинг DXY) — явный сигнал, что источник структурно нестабилен, а не единичный баг.
- CMC-ключ по памяти проекта интермиттентно отдаёт 401 вперемешку с квотой 429/1010 — состояние ключа хуже, чем просто "квота кончилась".
- `reader.py` зависит от Nitter (4 фолбэк-инстанса) для Lookonchain — сторонний скрейпинг-сервис, исторически нестабильный, но риск изолирован (не затрагивает подписчиков, только релей владельцу).

### 5.3 Отсутствие бэктестов
- `backtest_signal()` (bot.py:5533) существует, но **нигде не вызывается** — мёртвый код, не подключён ни к одной команде. К тому же он реконструирует свои TP/SL по swing high/low, а не через реальный конвейер K-LVL/чек-лист/R:R-гейт — даже если бы вызывался, не валидировал бы фактическую логику `fa_engine`.
- Нет walk-forward или мультисимвольного бэктеста, нет расчёта expectancy на исторических данных.
- Пороги гейтов (чек-лист≥4, R:R≥1.5, funding ±0.05%, мем-флаги) — константы, подобранные по смыслу/комментариям в коде, не откалиброванные и не проверенные против истории.
- Единственная фактическая обратная связь с реальностью — живой трекинг исходов через `/journal` (см. §3), который уже накапливает нужные для калибровки данные, но пока не используется для пересмотра порогов.
