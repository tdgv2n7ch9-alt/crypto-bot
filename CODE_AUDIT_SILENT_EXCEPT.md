# Аудит: `except: pass` / `except Exception: pass` (READ-ONLY, владелец, 2026-07-22)

Фоновая очередь, приоритет "аудит кода read-only" (CLAUDE.md, вечный
бэклог п.7): поиск известного класса дефекта (`except` без `log.error`) --
список файл:строка, **без правок**. Метод: построчный скан всех `.py`
файлов репозитория (кроме `tests/`, воркдеревьев, бэкапов) на паттерн
`except[ Тип][ as e]:` с телом ровно `pass` на следующей непустой строке
(самый явный, безусловно тихий случай -- не считает случаи, где после
`except` идёт `return`/`continue` без логирования, это отдельный, более
мягкий класс, здесь не собирался).

## Выделенная находка: алерт О СБОЕ сам глотается молча (2 места)

Тот же класс бага, что уже был найден и исправлен для `run_watchdog()`
(владелец, 2026-07-19: "раньше сбой отправки самого алерта глотался молча,
а флаг дедупа уже был выставлен -- лог + сброс флага, следующий тик
честно повторит"). Проверено -- ЭТОТ фикс НЕ был распространён на
остальные вызовы `send_system()`/`send_message()` в критичных путях:

**`bot.py:1250-1256`** (`run_daily_backup()`, отправка алерта о СБОЕ
ежедневного бэкапа -- сам бэкап critical, "риск потери данных" по
комментарию в коде):
```python
try:
    await send_system(bot, f"Дневной бэкап {date_str}: ...", critical=True)
except Exception:
    pass
```
Если САМА отправка этого алерта не удастся (сетевой сбой Telegram именно
в момент проблемы) -- владелец не узнает НИ о сбое бэкапа, НИ о том, что
уведомление не дошло. Нет лога, нет retry-логики.

**`bot.py:7118-7125`** (отправка в канал по `channel_id_str`):
```python
try:
    cid = int(channel_id_str)
    await bot.send_message(cid, text, parse_mode="Markdown", disable_web_page_preview=True)
except:
    pass
```
Бонус: это ГОЛЫЙ `except:` (не `except Exception:`) -- ловит даже
`SystemExit`/`KeyboardInterrupt`, более широкий класс, чем обычно
допустимо. Сбой отправки в канал -- тихий, без следа в логах.

**Кандидат-фикс** (НЕ применён): тот же паттерн, что уже используется в
`run_watchdog()` -- `log.error(f"...: {e}")` вместо `pass`, плюс (для
случая 1255) не сбрасывать/не выставлять anti-spam флаг раньше времени,
если сама отправка алерта упала.

выше, но всё равно: (а) голый `except:` шире, чем `except Exception:`

следа в логах, если канал специально настроен, но недоступен.

### Почему это безопасно применить (оценка риска, не решение)

Оба патча -- ТОЛЬКО добавляют логирование в уже существующий `except`,
НЕ меняют условия/поток исполнения/боевую логику сигналов, НЕ трогают
чек-лист/гейты. Класс изменения -- ровно тот же, что уже был применён и
проверен для `run_watchdog()` 2026-07-19 (та же формула:
`except Exception as e: log.error(...)`). По классификации CLAUDE.md
("Автономия v2") это выглядит как класс B (известный паттерн, тот же
класс бага, что уже решался и проверялся раньше) -- но затрагивает
`bot.py` (боевой код) и обозначен владельцем явно как "не применять,
это gate владельца", поэтому остаётся ТОЛЬКО предложением до отдельного
"да".

**Ничего не применено. Live-логика/гейты/чек-лист не менялись. `PAUSE_
LIVE_SIGNAL_EMISSION` без изменений (`true`), промоушены заблокированы.**
